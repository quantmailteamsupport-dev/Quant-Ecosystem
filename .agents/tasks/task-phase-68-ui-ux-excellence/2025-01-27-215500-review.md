# Framer-motion animation primitives and integration across Quant apps

Seven reusable animation components (FadeIn, StaggerList, PageTransition, SpringButton, AnimatedSkeleton, SlidePanel, ScaleOnHover) plus a MotionConfig context provider were added to `@quant/shared-ui`. Existing shell, navigation, and state components gained optional entrance animations gated by an `animated` prop. The four newer frontend apps (quantcalendar, quantdocs, quantdrive, quantmeet) wrap key sections in PageTransition/FadeIn/StaggerList. The motion primitives consume brand-level spring and duration tokens from `@quant/brand`, keeping animation values centralized.

**Watch for:** LoadingState's reduced-motion branch still uses CSS `animate-spin` and `animate-bounce`, defeating the accessibility intent (confirmed). StaggerList injects wrapper `<div>` elements around each child, which breaks semantic HTML when the child is a `<ul>` and corrupts `divide-y` styling (confirmed). The `MotionProvider` context is exported but never consumed by any component in this diff — every component independently calls `useReducedMotion` instead (confirmed).

## High-level view

The `animated` prop was added to shell/layout/navigation components (AppShell, Sidebar, TopBar, BottomNav) but not to the state components (EmptyState, ErrorState, LoadingState, SuccessState). State components animate unconditionally unless `prefers-reduced-motion` fires. This creates two different opt-out mechanisms for the same library: programmatic (`animated={false}`) for structural components and OS-level only for feedback components.

The reduced-motion handling has a correctness gap in LoadingState where the fallback branch still uses CSS-based infinite animations (`animate-spin`, `animate-bounce`, `animate-pulse`). The skeleton variant is never branched at all.

StaggerList's design of wrapping each child in a `motion.div` introduces a DOM structure issue: when the child is a semantic list element like `<ul>`, intermediate divs produce invalid HTML and break CSS utilities like `divide-y` that depend on direct-child selectors. In quantmeet's usage, the `<ul>` is the single child so no per-item staggering occurs at all.

framer-motion is added as a direct dependency of `@quant/shared-ui` (~60KB gzipped). For a monorepo consumed via source this is acceptable, but consumers who import any non-animated component may still pull in the runtime if the bundler can't tree-shake the motion imports.

## LoadingState reduced-motion inversion

In the spinner branch, the reduced-motion path renders:

```tsx
<svg className={`${sizeMap[size]} animate-spin text-blue-600`} ...>
```

`animate-spin` is a Tailwind utility that applies an infinite CSS rotation. The user who set `prefers-reduced-motion: reduce` still sees a spinning element because CSS `animate-spin` does not respect the media query by default (Tailwind only disables it if the project enables the `motion-reduce` variant, which is not configured here).

The dots variant has the same problem with `animate-bounce`:

```tsx
<span className="w-2 h-2 bg-blue-600 rounded-full animate-bounce" .../>
```

The skeleton variant (`animate-pulse`) is never branched at all — it renders identically regardless of `prefersReducedMotion`.

Fix: in the reduced-motion branch, remove the CSS animation classes entirely. For spinner, show a static partial-circle indicator. For dots, show three static dots. For skeleton, show solid gray bars without pulsing.

## StaggerList DOM structure and semantic breakage

StaggerList wraps each child with `motion.div`:

```tsx
{
  React.Children.map(children, (child) => <motion.div variants={itemVariants}>{child}</motion.div>);
}
```

In quantmeet, this wraps a `<ul>` element:

```tsx
<StaggerList>
  <ul className="divide-y divide-[var(--quant-border)] ...">
    <li>...</li>
  </ul>
</StaggerList>
```

StaggerList sees the `<ul>` as a single child, wraps it in one `motion.div` — no per-item staggering occurs. The intended effect (items appearing one by one) is completely absent.

For the stagger to work, each `<li>` would need to be a direct child of StaggerList. But then you get `<motion.div><li>...</li></motion.div>` inside a `<ul>`, which is invalid HTML.

A cleaner API would accept an `as` prop (defaulting to `div`) for the container, and use Framer's variant propagation (children automatically inherit parent variants when they have matching variant keys) rather than wrapping each child in an extra div. Alternatively, a render-prop or `cloneElement` approach can apply variants directly.

## MotionProvider disconnected from consumers

`MotionProvider` and `useMotionConfig` are exported and tested, but zero production components consume the context. Every Motion primitive independently calls `useReducedMotion()`. This means:

- If a consumer wraps their tree in `<MotionProvider>`, it has no effect on the primitives' behavior.
- Each mounted motion component registers its own `matchMedia` listener for `prefers-reduced-motion`. In a page with 20 animated elements, that's 20 listeners doing the same thing.

Either complete the integration (each primitive reads `shouldAnimate` from context, with standalone `useReducedMotion` as fallback when no provider is present), or remove the provider to avoid API confusion.

## Inconsistent animated prop across component categories

The `animated` prop (defaulting to `true`) was added to AppShell, Sidebar, TopBar, BottomNav but **not** to EmptyState, ErrorState, LoadingState, SuccessState. A developer who disables animations for performance profiling can turn off shell/nav animations but not state component animations. The `animated` prop should be consistent across the entire animated surface.

## Test coverage gaps

The tests mock framer-motion entirely, replacing `motion.div` with a plain `<div>`. The mock hardcodes `useReducedMotion: () => false`, so the reduced-motion fallback branches are never exercised.

Not tested: the reduced-motion fallback branch for any component, the `animated={false}` code path in shell/nav components, `AnimatePresence` exit behavior in SlidePanel, MotionProvider actually influencing child behavior.
