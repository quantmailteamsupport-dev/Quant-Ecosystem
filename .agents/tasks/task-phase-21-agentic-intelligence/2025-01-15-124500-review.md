# LLM-driven agent planning loop replaces keyword-matching pilots

The change introduces `IntelligentAgent`, a base class with a six-phase planning loop (observe, plan, propose, request-permission, execute, reflect), and refactors all 12 domain pilots to extend it instead of `WorkerAgent` directly. The AI engine is injected via a port interface (`AIEnginePort`), each pilot registers typed tools in a `TypedToolRegistry`, and the system layering adds per-task spending caps, token-rate limiting, trust-score auto-pause, 24h undo with cross-app metadata, and marketplace signature verification with a sandbox-promotion gate.

Watch for:

- **Plan parsing fallback silently runs all registered tools** when the LLM returns non-JSON, meaning a garbled response triggers maximum side effects. (confirmed)
- **Spending check races with execution**: `canSpend` is checked once before execution starts, but `recordSpend` inside the loop can still exceed the cap if multiple steps run sequentially after passing the initial gate. (confirmed)
- **`blockEnvAccess` is a demonstration stub**, not a runtime sandbox â€” it never wraps actual agent code. (confirmed)

## High-level view

`IntelligentAgent` sits between `WorkerAgent` (lifecycle, state machine, audit trail) and the 12 domain pilots. It owns AI inference, plan parsing, cost estimation, approval gating, and a reasoning trace. The planning loop transitions the state machine through PLANNING -> EXECUTING -> (WAITING_APPROVAL) -> DONE/FAILED, and the observe phase builds context from task params and available tools before asking the LLM for a JSON plan.

Each pilot registers domain-specific tools in its constructor and provides an `execute` override that pre-processes domain data with AI, then delegates to `super.execute()` for the planning loop. This creates a double-execution pattern where the pilot's domain AI work and the base class's planning loop make independent LLM calls â€” the planning loop is unaware of work already done. Finance and Health pilots add an explicit opt-in guard that transitions to FAILED if the user hasn't consented.

`SpendingCapBreachedError` carries cap type, current, and limit so callers can distinguish which boundary was hit. Trust score's `onAutoPause` callback is defined but never wired into `IntelligentAgent`'s lifecycle since `WorkerAgent` creates `TrustScore` without options. The undo engine's `registerMutation` with agent/app metadata enables per-agent batch revert. The marketplace installer gates sandbox promotion behind a passing `SecurityAudit` record, though the signature verification uses hash comparison rather than asymmetric cryptography.

1. **Plan parsing fallback invokes all tools** â€” When `parsePlanSteps` fails to find valid JSON, it constructs a step for every registered tool with empty args. Return an empty plan and fail gracefully instead.

5. **Subclass `execute` + `super.execute` causes double AI planning** â€” Pilots classify/analyze via AI, then `super.execute()` resets trace and re-plans from the raw task description, disconnected from the domain work already performed.
6. **State machine transition in `plan()` assumes PLANNING state** â€” Fragile coupling: `plan()` calls `transition(EXECUTING)` which only works if the machine is in PLANNING (set by `start()`). If any subclass transitions state before calling super, this throws.
7. **Trust score `onAutoPause` is dead code in agent lifecycle** â€” `WorkerAgent` creates `TrustScore` without `TrustScoreOptions`, so the callback is never connected. Auto-pause exists as testable infrastructure but doesn't halt agents in practice.

## Plan parsing fallback invokes every tool

`parsePlanSteps` in `intelligent-agent.ts` attempts to extract a JSON array from the LLM response. If parsing fails (malformed JSON, no array match), it silently enters the fallback branch:

```typescript
if (steps.length === 0) {
  for (const tool of tools) {
    steps.push({
      id: generateId('step'),
      toolName: tool.name,
      args: {},
      tier: tool.requiredTier,
      // ...
    });
  }
}
```

A garbled LLM response results in an execution plan that invokes _all_ tools registered in the agent's registry â€” including the `agent.handoff` meta-tool â€” with empty args. The approval queue gates Tier2+ tools, but Tier0 and Tier1 tools execute without approval, so tools like `email.draft_reply` or `content.draft` would fire with empty arguments. This is fail-open behavior.

## Double LLM execution in subclass override pattern

Every pilot follows this pattern:

```typescript
override async execute(task: AgentTask): Promise<void> {
  // 1. Domain-specific AI work (classify emails, analyze expenses, etc.)
  this.lastResult = await this.analyzeWithAI(data);
  // 2. Delegate to base class planning loop
  await super.execute(task);
}
```

`super.execute()` resets `reasoningTrace` to `[]` and `currentPlan` to `null`, then runs the full observe/plan/execute/reflect cycle. This second AI call plans actions based on the raw task description (not the results of step 1), creating a disconnect: the pilot's domain logic already classified and processed the data, then the planning loop independently decides what tools to invoke. The trace returned by `getReasoningTrace()` only reflects the second (super) execution, not the domain-specific AI calls from step 1.

## Cost estimation and enforcement gap

The `canSpend` check happens once with the aggregate estimate:

```typescript
const costEstimate = this.getCostPreview();
if (!this.spendingLimit.canSpend(costEstimate.totalEstimatedCost)) { ... }
```

Inside `executeActions`, each successful step calls `this.spendingLimit.recordSpend(stepCost)`. If a plan has five Tier1 steps ($0.05 total) and the daily budget has $0.04 remaining, the pre-check uses estimated total cost which passes, but during execution the fifth `recordSpend(0.01)` throws `SpendingCapBreachedError`. The plan is left partially executed with no rollback of prior steps.

## Marketplace signature verification is not cryptographic

```typescript
verifySignature(spec: unknown, signature: string, publicKey: string): boolean {
  const hash = createHash('sha256')
    .update(JSON.stringify(spec) + publicKey)
    .digest('hex');
  return hash === signature;
}
```

Anyone who knows the "publicKey" string can compute a valid "signature" for any spec â€” no private key is involved. This provides tamper detection (spec integrity given a shared secret) but not authenticity (proof that a specific publisher signed it).

## State machine double-transition risk

`WorkerAgent.start()` transitions to PLANNING. Subclass `execute` overrides make AI calls but don't transition the state machine. Then `super.execute()` calls `plan()`, which does `this.stateMachine.transition(AgentState.EXECUTING)`. The valid transitions from PLANNING include EXECUTING, so this works in the normal path. But the coupling is implicit: if any subclass transitions state before calling super, or if `start()` was not called, `plan()` throws an invalid transition error. No guard or assertion makes this contract explicit.

## Trust score auto-pause is disconnected

`TrustScore.recordFailure()` calls `this.onAutoPause?.(this.agentId)` when the score drops below 15. But `WorkerAgent` creates its own `TrustScore` in its constructor without passing `TrustScoreOptions`:

```typescript
// worker-agent.ts
this.trustScore = new TrustScore(permissionLevelToScore(config.defaultPermission));
```

`IntelligentAgent` inherits this `trustScore` and never replaces it. The `onAutoPause` callback and `agentId` field exist on the interface but are never connected to the kill-switch in the actual agent lifecycle.
