# Universal search orchestrator with RAG synthesis and cross-collection similarity (v2)

All five issues from the v1 review have been addressed. The RAG synthesizer now catches failures and returns search results without the AI answer. Scope search is concurrent via `Promise.all`. Pagination offset is computed correctly. The snippet highlighter HTML-escapes source text before injecting `<mark>` tags. The router schema accepts `isAdmin` from the request body and passes it to permissions. Each fix has a corresponding test.

**Watch for:** The router accepts `isAdmin` as a client-supplied body field with no server-side auth gate — any caller can claim admin access (confirmed). The PDF fallback extractor indexes structural noise from PDFs that lack BT/ET markers (likely).

## High-level view

The router's `isAdmin` field is now wired correctly, but it defaults to `false` via Zod and is taken from the request body at face value. There is no middleware or auth layer verifying this claim against the authenticated session. Any caller who sets `isAdmin: true` in the POST body gains admin-level search permissions (access to all private documents). This is the single blocking concern in the change.

The content extractor's PDF fallback path remains as-is: any 4+ character printable ASCII run that isn't a handful of PDF structure keywords gets indexed. For modern compressed-stream PDFs this will index font names, encoding tables, and Base64 fragments as searchable text, producing false-positive matches at search time. Not blocking for a v1, but worth a TODO or a stricter heuristic.

## isAdmin trust boundary in the router

The v1 review flagged the hardcoded `isAdmin: false`. The fix exposes it as a schema field:

```typescript
export const UniversalSearchRequestSchema = z.object({
  query: z.string().min(1),
  userId: z.string().min(1),
  isAdmin: z.boolean().default(false),
  // ...
});
```

The router passes this directly:

```typescript
permissions: { userId: validated.userId, isAdmin: validated.isAdmin },
```

There is no middleware or pre-handler that compares this claim against the session's actual role. In a typical deployment, `isAdmin` should be derived from the authentication token (JWT claim, session lookup, or RBAC middleware) rather than accepted as client input. As currently written, a non-admin user who knows the API contract can escalate their permissions by including `isAdmin: true` in the request body.

The test `'should pass isAdmin from request to permissions'` confirms the pass-through works — but it doesn't test that the claim is validated against a session. This is a privilege escalation vector if the router is exposed to authenticated-but-untrusted callers.

## PDF extractor fallback pollution

The fallback regex in `content-extractor.ts`:

```typescript
const printableRegex = /[\x20-\x7E]{4,}/g;
```

The exclusion list is limited to exact matches of `obj`, `endobj`, `stream`, `endstream`, `xref`, `trailer`, `startxref`. It does not filter:

- Font descriptors: `TimesNewRomanPSMT`, `ArialMT`, `/BaseFont`
- Encoding tables: `/Encoding /WinAnsiEncoding`
- Stream fragments: Base64-encoded image data (long printable runs)
- Cross-reference entries: `0000000015 00000 n`

For PDFs with compressed text streams (the majority in the wild), the BT/ET path yields nothing and the fallback indexes this noise. Searches for common font names or encoding keywords will produce false positives. A minimum run length of 10+ characters combined with an entropy check (reject runs that are mostly digits or have low letter diversity) would significantly reduce noise without losing real text.
