# LLM streaming pipeline with tool bridge and sentence splitter

This change wires an LLM streaming layer into `@quant/quant-live`, connecting the existing ASR pipeline through a streaming LLM provider, a sentence splitter (for TTS-sized chunks), and a tool execution bridge that routes tool calls to the agent-runtime registry with tier enforcement and timeout. The architecture mirrors the existing TTS streaming pattern (AsyncIterable-based), introduces a factory for provider selection (`quant-ai` vs `mock`), and adds a `processLLMResponse()` method to `LivePipeline` that chains ASR output into LLM streaming into sentence-split TTS synthesis.

Watch for: tool_call chunks from the LLM stream are never actually dispatched to the ToolBridge (confirmed); the sentence splitter regex splits on `.` inside abbreviations and decimal numbers (likely); the QuantAIProvider silently swallows all errors and falls back to mock responses with no logging or signal to callers (confirmed).

## High-level view

The `ToolBridge` is a standalone executor with tier gating, argument validation, and timeout, but it is not yet integrated into the pipeline's stream-processing loop. `processLLMResponse()` accepts the bridge as an optional parameter and immediately voids it.

The sentence splitter's regex is aggressive, splitting on any period regardless of context (abbreviations, decimals, URLs). This produces TTS fragments too short to synthesize well.

`processLLMResponse()` measures time-to-first-sentence (not time-to-first-token, since it sits after the splitter), dispatches sentences to response callbacks, and synthesizes each sentence sequentially. The sequential synthesis means total latency scales with sentence count rather than streaming in parallel.

<details>
<summary>Issues (5)</summary>

1. **tool_call chunks ignored in pipeline** — `processLLMResponse()` passes the LLM stream directly to `splitSentences`, which skips non-text chunks. Tool calls emitted by the LLM are silently dropped. Wire up ToolBridge dispatch for `tool_call` chunks before the sentence splitter, or split the stream into a text path and a tool path.
2. **Silent error swallowing in QuantAIProvider** — The catch block falls back to mock output with no logging, no error event, and no way for callers to distinguish a real mock response from a failure fallback. Add an error emission or at minimum a `type: 'error'` chunk so the pipeline can react.
3. **Sentence splitter splits on periods inside abbreviations/decimals** — The regex `/[.?!\u0964]/` matches any period, so "Dr. Smith" splits into "Dr." and "Smith", and "3.14" splits into "3." and "14". Use a lookbehind or heuristic (e.g., require whitespace or end-of-buffer after the punctuation).
4. **Sequential TTS synthesis blocks streaming** — Each sentence awaits `synthesizeResponse()` fully before the next sentence is dispatched. For multi-sentence responses, the user hears silence between sentences while the next TTS call completes. Consider prefetching the next sentence's TTS while the current one plays.
5. **Latency measurement captures time-to-first-sentence, not time-to-first-token** — The LLM latency `endMeasure` fires after the first complete sentence exits the splitter, not when the first token arrives from the LLM. This conflates LLM latency with sentence-accumulation time. Move the measurement to before the splitter or emit a timing event from the raw stream.

## Tool call dispatch gap

The `processLLMResponse()` method passes the raw LLM stream into `splitSentences()`, which processes only `type: 'text'` chunks and skips everything else:

```typescript
if (chunk.type === 'done') break;
if (chunk.type !== 'text' || !chunk.text) continue;
```

`LLMStreamChunk` defines `tool_call` and `tool_result` as valid types, the `ToolBridge` class is fully implemented with tier checks, validation, and timeout, and `processLLMResponse()` accepts an optional `toolBridge` parameter. But the bridge is never called; the parameter is consumed by `void toolBridge`. Any LLM that emits tool calls during streaming will have those calls silently dropped. The types promise a capability the pipeline doesn't deliver.

## Silent fallback in QuantAIProvider

When the injected `streamFn` throws, the provider falls to:

```typescript
catch {
  const mock = new MockLLMProvider();
  yield* mock.streamResponse(context);
}
```

The caller receives valid-looking text chunks ("Hello, how can I help you today?") with no indication that the real provider failed. In a live conversation, the user hears a canned response instead of their actual answer, and the system has no telemetry event to trigger alerts. The same pattern applies when `streamFn` is undefined (null check at the top). This is a fail-silent design that makes production debugging difficult.

## Sentence boundary regex

The splitter uses `const SENTENCE_ENDINGS = /[.?!\u0964]/` and splits on the first match in the buffer. This fires on:

- Abbreviations: "Dr. Smith" -> ["Dr.", "Smith"]
- Decimal numbers: "The value is 3.14 percent" -> ["The value is 3.", "14 percent"]
- Ellipses: "Wait..." -> ["Wait.", ".", "."]
- URLs or file paths: "Go to example.com" -> ["Go to example.", "com"]

For TTS, short fragments like "Dr." or "14" produce poor audio. A simple improvement: require the character after the period to be whitespace or end-of-buffer (`/[.?!]\s/` or lookahead).

## Sequential synthesis latency

Inside `processLLMResponse()`:

```typescript
for await (const sentence of sentences) {
  // ...
  await this.synthesizeResponse(sentence);
}
```

Each sentence blocks until `synthesizeResponse` completes (which itself iterates a TTS stream to completion). With a 3-sentence response, sentence 2's TTS doesn't start until sentence 1 finishes playing. The existing `PrefetchBuffer` in `synthesizeResponse` buffers _within_ a single sentence, but there's no cross-sentence prefetch. This adds inter-sentence gaps proportional to TTS startup latency.

## Dangling timeout in ToolBridge

The timeout `setTimeout` in `Promise.race` is never cleared on successful completion:

```typescript
const result = await Promise.race([
  tool.handler(toolCall.args),
  new Promise<never>((_, reject) =>
    setTimeout(() => reject(new Error('Tool execution timed out')), this.timeoutMs),
  ),
]);
```

The timer fires and rejects into a dead promise after the handler resolves. For high-throughput scenarios (many rapid tool calls), dangling timers accumulate until they fire and get GC'd. The type signature mismatch between ToolBridge's expected interface (`{ requiredTier, handler }`) and `TypedToolRegistry`'s `ToolDefinition` (which adds `category`, `parameters`, etc.) will also need an adapter when this is wired up.

## Test coverage gaps

Not tested:

- `processLLMResponse()` integration in pipeline (the new orchestration method has no test)
- tool_call chunks flowing through the system (would document the silent-drop behavior)
- Concurrent abort + in-flight tool execution in ToolBridge
- Sentence splitter behavior with abbreviations, decimals, or URLs
