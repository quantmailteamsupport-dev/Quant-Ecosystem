# ML moderation pipeline replacing regex placeholders (v2)

This change removes placeholder regex-based moderation and replaces it with an ML-based classification pipeline backed by injectable API client interfaces, adds trust/safety enforcement primitives (sybil detection, reputation, anti-spam, rate limiting, captcha), and an async BullMQ moderation worker. The v1 issues (worker main() crash, image handler using SimHash on URLs, cluster caching, whitelist/blacklist gap, rate limiter TTL, transparency report unbounded memory, and worker swallowing invalid jobs) have all been addressed.

**Watch for:** The image handler hashes the URL string bytes via `Buffer.from(job.content)` rather than fetching actual image data, so perceptual hash matching will fail for images delivered by URL (confirmed). The `createHandlerDeps()` factory wires placeholder API clients that throw on first call with no path to configure real ones at runtime (confirmed). The `AppealWorkflow.createFromAction` counts toward the per-user appeal limit, so automated moderation actions can exhaust a user's manual appeal quota (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The classifiers (TextClassifier, ImageClassifier) delegate to injected API clients and return a unified ModerationResult. The policy engine adds a second threshold layer on top. The image handler's hash computation operates on `Buffer.from(job.content)` where `job.content` is a URL string, not fetched image bytes. The pHash algorithm will produce stable hashes of the URL's ASCII bytes, which defeats the purpose of perceptual deduplication against a known-bad-hash database.

The worker's `createHandlerDeps()` factory constructs real classifier instances but with placeholder API clients that throw descriptive errors. There's no runtime configuration path: no env-var check that swaps in a real OpenAI client, no conditional logic. Every production job will hit the throw.

The appeal system's `createFromAction` adds records to the user's history, and `submitAppeal` enforces a max-appeals-per-user limit against that same history. A heavily-moderated user could hit the cap before ever manually appealing.

The security primitives all had their v1 issues fixed: rate limiter uses store TTL for `retryAfterMs`, anti-spam checks whitelist/blacklist at the top of `classify()`, abuse graph caches cluster results with a dirty flag. One v1 concern persists: `reportTimestamps` in AbuseGraph grows without bound.

<summary>Issues (5)</summary>

1. **Image handler hashes URL bytes, not image content** — `Buffer.from(job.content)` converts the URL string to a buffer of ASCII bytes, not the image's pixel data. Fetch the image from the URL before hashing, or accept that known-bad-hash matching only works for base64-encoded content passed inline.
2. **No runtime path to real API clients** — `createHandlerDeps()` always instantiates placeholder clients that throw. Add conditional logic (e.g., checking `MODERATION_API_KEY` env var) to construct real clients when credentials are available.
3. **Automated appeal records consume user appeal quota** — `createFromAction` pushes records into the user's appeal history. `submitAppeal` counts all records within the cooldown window. Separate automated audit records from user-initiated appeals in the quota check.
4. **Classifier action thresholds duplicated and unconfigurable** — Both TextClassifier and ImageClassifier have identical hardcoded thresholds (0.9/0.7/0.5) for action determination. Define in one shared location or make configurable.
5. **AbuseGraph reportTimestamps unbounded** — Each `addReport` pushes a timestamp that is never pruned. Add a sliding window or cap per user.

## Image handler hash computation on URL bytes

The v1 issue was using `computeSimHash` (text hashing) on a URL. The fix changed to `computeImageHash(Buffer.from(job.content))`, but `job.content` for image jobs is a URL string like `"https://cdn.example.com/img.jpg"`. `Buffer.from` produces a ~30-byte buffer of ASCII character codes. The DCT algorithm samples bytes at regular intervals from this buffer, producing a hash of the URL's byte representation rather than the image's pixel data.

```typescript
// image-handler.ts line ~44
const imageHash = this.hasher.computeImageHash(Buffer.from(job.content));
```

For the known-bad-hash check to work as intended, the handler needs to fetch the image bytes from the URL first. The test passes because it sets up a `knownBadHashes` set with the hash of the test string's bytes, creating a tautological match.

## Worker factory and runtime configuration gap

`createHandlerDeps()` fixes the v1 crash but the placeholder clients are unconditionally constructed:

```typescript
function createPlaceholderTextClient(): ModerationAPIClient {
  return {
    moderateText: () => {
      throw new Error('Text moderation API client not configured...');
    },
  };
}
```

There's no `if (process.env['MODERATION_API_KEY'])` branch that constructs a real HTTP client. The factory always produces a worker that will throw on the first real classification call, making the worker deployable only as a test harness.

## Appeal quota interaction with automated records

`createFromAction` adds a record and pushes its ID to `userRecords`:

```typescript
const userList = this.userRecords.get(userId) ?? [];
userList.push(record.id);
this.userRecords.set(userId, userList);
```

`submitAppeal` counts all records within the cooldown window:

```typescript
const recentAppeals = userList.filter((id) => {
  const record = this.records.get(id);
  return record && Date.now() - record.createdAt < this.config.cooldownDays * 86400000;
});
if (recentAppeals.length >= this.config.maxAppealsPerUser) {
  throw new Error(`Maximum appeals (${this.config.maxAppealsPerUser}) reached...`);
}
```

With `maxAppealsPerUser: 3`, a user who has 3 pieces of content auto-moderated within 30 days will be unable to submit any manual appeal. The fix is to either separate automated records from user-initiated appeals in the count, or only count records where the user explicitly submitted an appeal (e.g., filter by a `source` field).

## AbuseGraph reportTimestamps unbounded growth

While cluster caching was fixed (dirty flag works correctly), `reportTimestamps` still grows without bound. Each `addReport` pushes a timestamp, and these arrays are never pruned or windowed. In a long-running process with active reporters, this becomes a memory concern. The `getUserRisk` velocity check only needs recent timestamps (last few minutes), so a sliding window that evicts entries older than the velocity detection period would suffice.
