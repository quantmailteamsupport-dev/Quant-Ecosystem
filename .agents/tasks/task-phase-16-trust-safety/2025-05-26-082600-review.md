# ML moderation pipeline replacing regex placeholders

This change removes all placeholder regex-based moderation (PROFANITY_PATTERNS, HATE_PATTERNS, CATEGORY_KEYWORDS) and replaces them with an ML-based classification pipeline backed by injectable API client interfaces. It adds three new subsystems: ML classifiers in `packages/moderation/`, trust/safety enforcement primitives in `packages/security/`, and an async BullMQ worker in `services/moderation-worker/` that routes content through the pipeline. The architecture is interface-driven throughout, making every external dependency mockable.

**Watch for:** The moderation worker's `main()` bootstraps with `{} as ModerationHandlerDeps` meaning it will crash on the first real job in any environment (confirmed). The `AbuseGraph.getClusterForUser()` re-runs full cluster detection on every call with no caching (confirmed). The `ImageModerationHandler` uses `computeSimHash` (text hashing) on image URLs instead of `computeImageHash` on actual image bytes (confirmed).

## High-level view

The moderation package introduces a clean separation between classification (TextClassifier, ImageClassifier), policy enforcement (PolicyEngine), and lifecycle management (AppealWorkflow, TransparencyReportGenerator). Classifiers take injectable API clients and return a unified `ModerationResult`; the policy engine applies per-app threshold rules to determine the final action. This is a solid inversion-of-control design that makes the system testable and swappable between providers.

The security package adds five independent enforcement primitives: sybil cluster detection via a co-reporting graph, reputation scoring with weighted factors, Naive Bayes + feature-based spam classification, configurable rate limiting behind a storage interface, and reputation-gated captcha challenges. Each is self-contained with its own Zod schemas and in-memory state, designed for later Redis-backed production deployment.

The moderation worker consumes jobs from BullMQ, routes by content type to handler classes, and delegates action execution to an `ActionExecutor` that writes audit log entries. The routing and handler logic are properly extracted for testability, but the `main()` function's dependency wiring is a stub that will fail at runtime.

<summary>Issues (7)</summary>

1. **Worker main() crashes on first job** — `deps` is cast as `{} as ModerationHandlerDeps`, so any real job will throw a null-reference error. Wire real handler instances or add a factory function that constructs them from config.
2. **Image handler uses text hashing on URLs** — `ImageModerationHandler.handle()` calls `this.hasher.computeSimHash(job.content)` where `job.content` is a URL string, not image bytes. This means the known-bad-hash check compares URL SimHashes, not perceptual image hashes. Should fetch the image buffer and call `computeImageHash`.
3. **AbuseGraph re-detects clusters on every getUserRisk call** — `getClusterForUser` calls `detectSybilClusters()` with O(n^2) co-reporting graph construction each time. Cache cluster results or compute lazily.
4. **Anti-spam whitelist/blacklist never consulted during classify** — `classify()` never checks `this.whitelist` or `this.blacklist`. The `addToWhitelist`/`addToBlacklist` methods exist but have no effect on classification decisions.
5. **Rate limiter retryAfterMs always returns windowMs on deny** — The `retryAfterMs` calculation has dead logic (the ternary always evaluates to `rule.windowMs`), so the value doesn't reflect actual time remaining.
6. **Transparency report uses in-memory arrays with no eviction** — `moderationActions` and `appealRecords` grow unbounded; in a long-running process this is a memory leak.
7. **Worker swallows invalid job data** — When `ModerationJobSchema.safeParse` fails, the worker logs and returns `undefined`, causing BullMQ to mark the job as completed successfully rather than failing it.

## Classifier architecture and API client interfaces

TextClassifier and ImageClassifier both accept an injected client interface (`ModerationAPIClient` / `ImageModerationAPIClient`). The response types (`TextModerationResponse`, `ImageModerationResponse`) define the expected shape from the external API. Classification results are mapped to a unified `ModerationResult` with category scores, action decisions, and confidence values.

The action determination logic uses fixed thresholds: score >= 0.9 removes, >= 0.7 flags, >= 0.5 restricts, else warns. These thresholds are hardcoded in both classifiers independently and are not surfaced as configuration. If an operator wants to tune classifier-level thresholds separately from policy-engine thresholds, they'd need to modify source code. The policy engine adds a second layer of thresholds, which means the final action is the policy engine's decision, not the classifier's. This is fine for the worker pipeline, but `TextModerator.moderate()` and `ImageModerator.moderate()` bypass the policy engine entirely and return the classifier's raw action decision.

The `ContentClassifier` facade delegates to text or image classifiers but only handles those two types. Video, audio, and other content types go through the worker's handler classes instead, creating two different code paths for classification depending on whether content enters via the facade or via the worker.

## Image handler's hash mismatch

In `ImageModerationHandler.handle()`:

```typescript
const imageHash = this.hasher.computeSimHash(job.content);
```

`job.content` for image jobs is a URL string (e.g., `https://cdn.example.com/img.jpg`). `computeSimHash` tokenizes the string into words and produces a text fingerprint. This means the "known-bad hash" check is comparing URL text fingerprints, not perceptual image hashes. Two different URLs pointing to the same image will produce different hashes, and the same URL with a query parameter change will also differ. The `PerceptualHasher.computeImageHash` method expects a `Buffer` of image bytes, which would require fetching the image first. The handler never fetches image content.

## Worker bootstrap and job error handling

The `main()` function assigns `const deps = {} as ModerationHandlerDeps`, which means all four handler slots are `undefined`. The `buildHandlerMap` call wraps these in lambdas like `(job) => deps.textHandler.handle(job)`, so the first real job will throw `Cannot read properties of undefined (reading 'handle')`.

When `ModerationJobSchema.safeParse` fails, the handler returns `undefined`:

```typescript
if (!parsed.success) {
  logger.error({ errors: parsed.error.errors }, 'Invalid job data');
  return;
}
```

BullMQ treats a handler that returns without throwing as a successful completion. The malformed job disappears from the queue with no retry and no dead-letter routing. Throwing an error (or using `bullJob.moveToFailed`) would preserve the job for investigation.

## AbuseGraph performance characteristics

`detectSybilClusters()` builds a co-reporting graph by comparing every pair of reporters' target sets — O(n^2) in the number of reporters, with an inner loop over targets. `getUserRisk()` calls `getClusterForUser()` which calls `detectSybilClusters()` again. In a graph with thousands of reporters, calling `getUserRisk` in a request path would be expensive.

The `reportTimestamps` array in `AbuseGraph` also grows without bound. Each call to `addReport` pushes a timestamp, and these are never pruned or windowed.

## Anti-spam whitelist/blacklist gap

The `AntiSpamFilter` has `addToWhitelist`, `addToBlacklist`, `isWhitelisted`, and `isBlacklisted` methods. However, `classify()` never calls either check:

```typescript
classify(input: SpamInput): SpamResult {
  const parsed = SpamInputSchema.parse(input);
  // ... goes straight to feature scoring ...
}
```

A blacklisted sender still gets the normal Bayesian + feature pipeline instead of an immediate spam verdict. The whitelist/blacklist infrastructure exists but is inert.

## Rate limiter retryAfterMs logic

In `ConfigurableRateLimiter.checkLimit()`:

```typescript
if (count > rule.maxRequests) {
  const retryAfterMs = rule.windowMs - (count - rule.maxRequests > 1 ? 0 : rule.windowMs);
  return {
    allowed: false,
    remaining: 0,
    resetAt,
    retryAfterMs: rule.windowMs, // <-- always returns windowMs regardless of calculation above
  };
}
```

The `retryAfterMs` local variable is computed but never used; the returned object hardcodes `rule.windowMs`. The calculation itself is also suspect: when `count - maxRequests > 1`, it produces `windowMs - 0 = windowMs`, and otherwise `windowMs - windowMs = 0`. The intent appears to be reporting time until the window resets, but the implementation doesn't track when the window started.

## Appeal workflow in-memory storage

`AppealWorkflow` stores all records in a `Map<string, AppealRecord>` and a parallel `Map<string, string[]>` for user-to-record indexing. This is fine for tests but will lose all appeal state on process restart. The same applies to `TransparencyReportGenerator` which accumulates all actions and appeals in arrays.

The cooldown check in `submitAppeal` filters records where `Date.now() - record.createdAt < cooldownDays * 86400000`, but `createFromAction` (called for every moderation result) also adds to the user's record list. If the system generates automated appeals for every moderation action, users could hit the max-appeals limit without ever manually appealing.

## Test coverage

Tests cover the happy path well: classification with mocked API responses, policy evaluation, appeal lifecycle, sybil cluster detection, spam classification, rate limiting, and captcha triggering. The moderation worker's routing logic is tested independently of BullMQ via exported `buildHandlerMap` and `routeJob` functions.

Not tested: behavior when API clients throw errors (network failures, timeouts), the `main()` bootstrap path, BullMQ retry semantics, concurrent access to shared Maps, or the `ContentClassifier` facade's delegation logic.
