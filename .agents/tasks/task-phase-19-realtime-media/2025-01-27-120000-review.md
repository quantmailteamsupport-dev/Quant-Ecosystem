# LiveKit gateway, WebRTC components, call/matchmaking services, and TURN infrastructure

Phase 19 replaces the fake SFU service in QuantMeet with a real LiveKit gateway (livekit-server-sdk), adds a 12-component MeetingRoom React suite backed by livekit-client, introduces 1:1/group call support in QuantChat, builds a matchmaking service for QuantMax random video chat, deploys a coturn TURN server via Helm with daily credential rotation, and adds E2E encryption support. The approach centralizes all server-side LiveKit interaction in a single LiveKitGateway class, with existing services (recording, breakout) delegating to it. Frontend components use LiveKit's data channel for in-meeting features (polls, chat, knock-flow). All backend state is held in in-memory Maps.

**Watch for:** Partner token leaked to the initiator in the random-chat response (confirmed); all call/matchmaking state lost on restart with no persistence path (confirmed); webhook endpoint has no replay/idempotency protection (confirmed); TURN credential rotation CronJob lacks a ServiceAccount RBAC namespace constraint (confirmed); KnockFlow admission decisions are purely client-side with no server enforcement (confirmed).

## High-level view

The LiveKitGateway wraps RoomServiceClient and EgressClient, generates 6-hour JWTs, handles egress for recording to S3 via protobuf message classes, and surfaces SDK failures as typed application errors.

The webhook service validates signatures via WebhookReceiver and publishes domain events through a Node EventEmitter. There is no persistence or replay protection: a dropped webhook or duplicate delivery will silently miscount participants or double-trigger egress-ended handlers.

The frontend component suite communicates entirely through LiveKit's data channel for chat, polls, and knock-flow. All admission control (KnockFlow) and vote integrity (Polls) live purely in the client; any participant with SDK access can forge admit/deny or vote messages.

The random-chat route in quantmax returns the partner's token directly in the HTTP response to the match initiator. This gives one participant full access to impersonate the other in the LiveKit room.

All backend services (CallService, MatchmakingService, random-chat queue) hold state in per-process Maps. Horizontal scaling or a pod restart will lose active calls, matchmaking queues, and breakout room records.

The TURN helm chart uses `coturn/coturn:latest` without pinning, and the rotation CronJob's Role is not namespace-scoped to specific resource names.

<summary>Issues (8)</summary>

1. **Partner token leaked in /random-chat/join response** — The response includes both `token` (for the caller) and `partnerToken` (for the matched user). The partner token should be delivered to the partner through a separate channel (push notification, WebSocket), not returned to the initiator.
2. **No webhook idempotency or replay protection** — The webhook handler processes every delivery with no dedup key. LiveKit retries on non-2xx, so a slow response followed by a retry will produce duplicate domain events.
3. **All call/queue state in-memory Maps** — CallService, MatchmakingService, and the quantmax queue all lose state on restart. Any deploy or crash orphans LiveKit rooms that never get cleaned up.
4. **KnockFlow admit/deny is client-enforced only** — The admit response is a data-channel message from host to participant. A participant that ignores the deny (or spoofs an admit) can still publish/subscribe because the LiveKit room itself has no server-side gate.
5. **Poll vote integrity is client-side** — Any participant can broadcast arbitrary poll-vote messages (including multiple votes or votes on behalf of others). No server-side tally exists.
6. **TURN CronJob Role not namespace-constrained** — The Role allows `get` and `patch` on all secrets and deployments in the namespace, not scoped to specific resource names. A compromised CronJob pod could patch any secret.
7. **Token TTLs vary without documentation** — Gateway tokens are 6h, call tokens are 1h, matchmaking tokens are 30m. The differences are intentional (longer meetings vs. ephemeral chats) but undocumented, making future changes error-prone.
8. **Coturn image tag is `latest`** — Both the main coturn deployment and the kubectl rotation image use `latest`, making builds non-reproducible and exposing the cluster to breaking changes on image update.

## Partner token exposure in random-chat

In `apps/quantmax/backend/routes/random-chat.ts`, when a match is found, the response body includes both the caller's token and the matched partner's token:

```typescript
return reply.send({
  success: true,
  data: {
    matched: true,
    pairId,
    roomName,
    token: tokenA,
    partnerToken: tokenB, // <-- leaked to the initiator
  },
});
```

A LiveKit access token encodes the participant's identity and grants. Handing both tokens to one client allows impersonation: the initiator can join the room as the partner. The partner's token should be delivered out-of-band (push notification, WebSocket event from the matchmaking service, or a separate `/token` endpoint the partner calls).

## Webhook reliability gap

The webhook handler in `apps/quantmeet/backend/routes/webhooks.ts` validates the signature and emits a domain event, then returns 200. Two failure modes:

1. If the event handler (downstream of `this.emit('event', domainEvent)`) throws asynchronously after the 200 is returned, the event is lost.
2. LiveKit retries webhooks on timeout. If the handler is slow (e.g., egress-ended triggers an S3 upload), a retry produces a duplicate domain event. There is no idempotency key (e.g., `event.id` or `event.createdAt + event.room.name`) being checked against a dedup store.

For egress_ended events specifically, a duplicate could trigger double-marking a recording as completed.

## In-memory state and room orphaning

CallService, MatchmakingService, and the random-chat module-level `queue` Map all hold active-session state in memory. On pod restart:

- Active calls in CallService have LiveKit rooms that remain open (emptyTimeout of 60s will eventually close them, but the 60s window allows unauthorized access since the room still exists).
- The matchmaking queue silently drops everyone; users see no error but never get matched.
- Breakout rooms tracked by BreakoutService are forgotten; the LiveKit rooms persist until emptyTimeout (300s for breakout rooms).

The emptyTimeout partially mitigates orphaned rooms but only after all participants leave. If one participant is still connected, the room stays alive indefinitely with no service-side record of it.

## Client-enforced admission in KnockFlow

KnockFlow uses data-channel messages for knock/admit/deny. The LiveKit room itself does not enforce admission. When a room is "locked," participants can still join the LiveKit room (they have a valid token with `roomJoin: true`); the lock is purely a UI concept displayed by the KnockFlow component. A client that skips the KnockFlow logic and calls `room.connect()` directly will be a full participant.

Server-side enforcement would require either:

- Not issuing tokens until admission is granted (gateway-level gate), or
- Using LiveKit's `hidden` participant mode and only promoting after admission.

## TURN credential rotation RBAC

The CronJob Role (`infra/helm/turn/templates/cronjob.yaml`) grants:

```yaml
rules:
  - apiGroups: ['']
    resources: ['secrets']
    verbs: ['get', 'patch']
  - apiGroups: ['apps']
    resources: ['deployments']
    verbs: ['get', 'patch']
```

This permits the rotator to patch _any_ secret and _any_ deployment in the namespace. Scoping to `resourceNames: ["{{ .Release.Name }}-turn-credentials"]` and `resourceNames: ["{{ .Release.Name }}-coturn"]` would follow least-privilege.

## CallService leave vs. end asymmetry

The `/calls/:id/leave` endpoint calls `callService.endCall(callId)`, which deletes the LiveKit room entirely. In a group call with 8 participants, one person leaving destroys the room for everyone. There is no `leaveCall` method that removes a single participant; the only operation is the nuclear `endCall`.

The route naming suggests "leave" semantics (individual departure) but implements "end" semantics (room destruction). This will surprise callers of the group-call API.

## MeetingRoom reconnection and media permissions

MeetingRoom creates a `Room` instance via `useState(() => new Room())` and connects in a `useEffect`. If the `token` or `serverUrl` props change, the effect fires again on the same Room instance that was already disconnected. The Room SDK may not support reconnecting a disconnected Room object; if token refresh is needed, the component would need to create a new Room instance.

`enableCameraAndMicrophone()` is called immediately on connect with no user consent gate. On the first visit, this triggers a browser permission prompt. If denied, the error is caught but the participant joins without media and there's no UI to retry the permission request.
