# Monorepo integration fixes: auth enforcement, Prisma wiring, port corrections, router conflicts

A wide-net bug-fix pass across the Quant monorepo addressing build failures, runtime misconfigurations, and conflicting Next.js routing. The dominant theme is getting the pnpm/TypeScript/Prisma dependency chain to resolve without compile-time path mappings, using a dynamic-import trick in server-core. On the runtime side, a global auth enforcement hook now protects all routes by default, duplicate pages/app router files are removed, and several hardcoded port numbers are corrected.

**Watch for:**
- The dynamic import hack (`'@quant/' + 'database'`) silently swallows failures at startupâ€”there's no way to distinguish "database package missing in tests" from "database package present but broken" (confirmed).
- The global auth hook whitelist uses exact string equality on `request.url`, so any query-string or trailing-slash variation bypasses it (confirmed).
- Converting `console.warn` to comments in sso-service removes runtime observability for a security-critical stub (confirmed).

## High-level view

The Prisma resolution strategy hoists packages to root `node_modules`, points the generator output there, removes per-app path mappings, and has server-core import the database package via a string-concatenation trick that defeats TypeScript's module resolver. This solves cross-package type resolution at the cost of making the Prisma dependency invisible to the compiler and silently optional at runtime.

Global auth enforcement centralizes the security posture in one `onRequest` hook inside `createApp()`. Three paths are whitelisted by exact URL matchâ€”brittle against query strings, trailing slashes, or proxy-normalized paths that Kubernetes health probes might produce.

The AI SDK migration updates token-usage field names to the v5 API (`inputTokens`/`outputTokens`) and casts provider models to `any`, removing compile-time safety that would catch the next SDK breaking change.

The pages-router deletions resolve Next.js's prohibition against having both `pages/X.tsx` and `app/X/page.tsx` for the same route. The removed files are full-featured components (analytics dashboard with charting, audiences segment builder, campaigns list, camera, map, stories)â€”if the app-router equivalents are incomplete, those routes now 404.

## Silent Prisma bootstrap and the dynamic-import trick

The bare `catch {}` block in `app.ts`:

```typescript
try {
  const mod: string = '@quant/' + 'database';
  const db = await import(/* @vite-ignore */ mod);
  await fastify.register(prismaPlugin, { client: db.prisma });
} catch {
  // @quant/database not available (e.g., in unit tests without DB)
}
```

This swallows every failure mode: package genuinely absent (expected in unit tests), package present but `prisma` export undefined (bug), package present but connection fails (misconfigured DB URL), package present but wrong schema version. In a production deployment with a broken `DATABASE_URL`, the app starts successfully with no Prisma client, and every DB query throws `Cannot read property X of undefined` on `app.prisma`.

A conditional log when `config.env !== 'test'` would surface this. Alternatively, make the database package non-optional for non-test configs and let the error propagate.

## Auth enforcement whitelist brittleness

```typescript
fastify.addHook('onRequest', async (request, reply) => {
  if (request.url === '/healthz' || request.url === '/readyz' || request.url === '/metrics') {
    return;
  }
  await fastify.requireAuth()(request, reply);
});
```

In Fastify, `request.url` is the raw URL string including query parameters. A request to `/healthz?_=123` or `/metrics?format=prometheus` will not match and will require authentication. If a reverse proxy adds trailing slashes, health checks behind a load balancer start failingâ€”causing pod restarts when these paths are used for Kubernetes liveness/readiness probes.

Parsing the pathname before comparison (`new URL(request.url, 'http://x').pathname`) or checking `request.routeOptions.url` after routing would be more robust.

## SSO observability regression

The `console.warn` calls in `sso-service.ts` were the only runtime indicators that:
1. An assertion was rejected because verification isn't implemented (fail-closed path).
2. An assertion was accepted without verification because the insecure flag was set.

Both are security-relevant events. Converting them to comments means zero telemetry. In a production incident where "SSO stopped working", there's no log line to find. The lint fix is understandable (`no-console` rules), but the right fix is replacing with the project's structured logger, not removing observability entirely.

## AI SDK `any` cast

The `as any` cast on the provider model passed to `generateText` and `streamText` removes the type constraint entirely. If the next AI SDK release changes the `model` parameter shape, there's no compile-time signal. A more targeted approach would be defining a local interface matching the subset actually consumed, or using a type assertion with `satisfies`.

## Pages/app router conflict resolution

The diff removes seven full implementations:
- `quantads/src/pages/analytics.tsx` (316 lines), `audiences.tsx` (562 lines), `campaigns.tsx` (407 lines), `index.tsx` (70 lines)
- `quantchat/src/pages/camera.tsx` (223 lines), `map.tsx` (208 lines), `stories.tsx` (206 lines)

These are production-quality componentsâ€”the analytics page has charting and KPI cards, the audiences page has a segment builder with demographic/interest/behavior targeting. The assumption is that equivalent `app/` directory implementations exist at parity. If the app-router versions are stubs or partial implementations, user-facing functionality has been removed.
