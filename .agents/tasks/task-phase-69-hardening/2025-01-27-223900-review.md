# XSS sanitization, production logger, and security headers across the Quant ecosystem

This change hardens the frontend layer across all 7 Next.js apps. It adds DOMPurify-based HTML sanitization to every `dangerouslySetInnerHTML` call site (7 total), replaces scattered `console.log`/`console.error` with a centralized environment-aware logger that suppresses non-error output in production, and adds a standard set of security response headers (HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy) to every app's Next.js config.

The sanitization approach uses browser-only DOMPurify with an SSR bypass that returns unsanitized HTML during server-side rendering. The logger is a thin wrapper that gates on `NODE_ENV`. Security headers are copy-pasted identically across configs with one appropriate deviation: quantmeet permits `camera=(self), microphone=(self)`.

Watch for: SSR bypass in sanitize.ts passes unsanitized HTML through on the server (confirmed). No unit tests for either new utility (confirmed). Missing Content-Security-Policy header across all apps (confirmed).

## High-level view

The sanitization utility solves the XSS problem at the call site but introduces a server-side gap: during SSR, `typeof window === 'undefined'` causes the function to return raw HTML unmodified. React hydration replaces it client-side, so the sanitized version only takes effect after JavaScript loads. For quantmail's email rendering, this means crawlers, link preview bots, and edge caches may receive unsanitized payloads.

The logger replacement sweep touches 21 files, converting catch-block `console.error` calls to `logger.error` and removing pure-debug `console.log` statements. No `logger.warn` or `logger.debug` calls were introduced anywhere, meaning the production suppression logic for non-error levels is currently dead code. Prettier reformatting triggered by lint-staged inflates the diff by several thousand lines of whitespace changes.

Security headers are applied uniformly but omit Content-Security-Policy — the most impactful XSS mitigation header and the natural complement to DOMPurify. The headers chosen are low-risk but don't address the primary attack vector this task targets.

## SSR bypass in the sanitization layer

The core of FEAT-001 lives in `packages/shared-ui/src/utils/sanitize.ts`:

```typescript
export function sanitizeHtmlContent(html: string): string {
  if (typeof window === 'undefined') {
    return html; // <-- unsanitized on server
  }
  return DOMPurify.sanitize(html);
}
```

The findings note explains the trade-off: `isomorphic-dompurify` pulls in jsdom which depends on ESM-only packages (parse5, @exodus/bytes) that break Next.js webpack builds. The bypass is documented and intentional. The security implication is that any server-rendered HTML containing malicious content will be present in the initial HTML response. React hydration replaces it client-side, but:

- Search engine crawlers and link preview bots consume the SSR HTML directly
- Edge caches or CDN-cached pages may serve the unsanitized version
- If JavaScript fails to load (network issue, CSP block), the unsanitized content persists

For the quantmail `EmailThread` and `thread/[id]` components, which render user-provided email HTML, this is the highest-risk scenario. Email HTML is the most common XSS vector in productivity apps.

The `sanitizeCodeHighlight` function has the same bypass but lower risk since it processes syntax-highlighter output (developer-controlled strings, not user input).

## Logger: dead suppression paths

No `logger.warn` or `logger.debug` calls exist anywhere in the codebase after this change. The production suppression logic for those levels is exercised only by `logger.log`, which is also never called. Every replacement uses `logger.error` — the one level that always passes through regardless of environment. The gating infrastructure exists but has no practical effect today.

## Security headers — X-Frame-Options and CSP gaps

`X-Frame-Options: DENY` prevents all iframe embedding. If quantdocs or quantmeet ever need to be embedded in another app (e.g., a portal, an LMS, or each other), this will break silently. `SAMEORIGIN` would be safer for an ecosystem where apps may embed each other (possible).

The most significant absent header is `Content-Security-Policy`. A basic policy like `script-src 'self'` would prevent inline script execution even if DOMPurify is bypassed (mutation XSS, DOMPurify zero-day). The combination of DOMPurify + CSP provides defense-in-depth; either alone is a single point of failure (confirmed — no CSP present in any config).

## Test coverage gap

Neither `sanitize.ts` nor `logger.ts` has unit tests. The sanitize utility is security-critical infrastructure — it's the single function standing between user-provided HTML and XSS execution. Testing should cover at minimum: script tags, event handler attributes (onerror, onload), javascript: URLs, SVG-based XSS, and the SSR bypass behavior (verifying it returns input unchanged when window is undefined). The logger should test that production mode actually suppresses output and that error always passes through.
