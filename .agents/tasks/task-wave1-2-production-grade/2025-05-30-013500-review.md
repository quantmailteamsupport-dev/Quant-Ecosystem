# Production-grade fixes: dead builds, admin API, cross-app wiring

This change revives three dead frontend apps (quantmax, quantneon, quantedits) by adding full Next.js 15 infrastructure, replaces mock data in the admin panel's API routes with live service pings and Prisma queries, and wires cross-app services (notification dispatch, event bus, federated search) through the backend layer. The scope is broad: 69 files, ~1500 lines added, touching frontend configs, admin API routes, shared packages, and the WebSocket gateway.

**Watch for:** Every admin API route (users, restart, broadcast, search, logs) is completely unauthenticated — no middleware, no session check, no role guard (confirmed). The notification broadcast endpoint can be hit by anyone to push notifications to arbitrary user IDs. The EventBus singleton means handler leaks are possible in long-running processes if `registerEventHandlers` is called more than once (likely). PrismaClient is instantiated at module scope in multiple route files with no connection pooling or shutdown hook, which will exhaust database connections under load (confirmed).

## High-level view

The admin panel's new API surface is the highest-risk area. Seven routes expose Prisma queries, service health pings, notification broadcast, and cross-app search — all without authentication. The absence of any auth layer means the admin panel currently trusts its network boundary entirely.

The cross-app wiring introduces three new abstractions: CrossAppDispatcher wraps NotificationFanout with per-app context, EventBus provides in-process pub/sub for the ws-gateway, and CrossAppSearchApi wraps CrossAppSearchService with user-context hooks. The notification dispatch in quantchat passes the conversation ID where user IDs are expected, which means notifications route to nobody.

The EventBus is an in-process singleton that cannot relay events across multiple gateway instances. This is fine for single-replica development but becomes a scaling cliff in production. The wildcard subscription also creates duplicate delivery for events that match a named channel.

The Next.js configs relax TypeScript strictness (`noUnusedLocals`, `noUnusedParameters` disabled) to avoid touching pre-existing code. Pragmatic for a build-fix PR but these flags should be re-enabled once dead code is cleaned up.

## Admin API: unauthenticated privilege surface

None of the seven new admin routes check for a session, JWT, or role. There is no `middleware.ts` in the admin app and no per-route auth guard. The routes expose:

- `GET /api/users` — dumps paginated user records (email, username, role, login timestamps) from Prisma
- `POST /api/apps/[id]/restart` — triggers (simulated) app restart by ID
- `POST /api/notifications/broadcast` — pushes notifications to arbitrary user IDs via NotificationFanout
- `GET /api/search` — cross-app search (information-disclosure vector)
- `GET /api/services/[id]/logs` — fetches service health data

The broadcast route is the most dangerous: it accepts a `recipientIds` array and a `title`/`body` with no validation of who is calling it. In a deployed environment, this is a spam/phishing vector. The restart route currently only pings health endpoints, but the intent is to evolve it into an actual process signal — the lack of auth now sets a dangerous precedent.

The fix is straightforward: a `middleware.ts` that checks session + admin role, or a shared `requireAdmin()` helper called at the top of each route handler.

## PrismaClient connection leak

```typescript
// apps/admin/src/app/api/users/route.ts (line 4)
const prisma = new PrismaClient();

// apps/admin/src/app/api/stats/route.ts (line 4)
const prisma = new PrismaClient();
```

Two separate module-scope `PrismaClient` instances exist across the admin routes. In serverless or edge deployments, each cold start creates a new connection pool. In long-running processes, Next.js hot-reloads during development will accumulate instances without closing old ones. The standard pattern is:

```typescript
const globalForPrisma = globalThis as unknown as { prisma?: PrismaClient };
const prisma = globalForPrisma.prisma ?? new PrismaClient();
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;
```

## Notification wiring: recipient identity mismatch

In `apps/quantchat/backend/routes/messages.ts`, the notification dispatch passes the conversation ID as the recipient list:

```typescript
notifier.notifyNewMessage(
  [request.params.id], // this is the conversationId, not user IDs
  userId,
  parseResult.data.content.slice(0, 100),
  request.params.id,
);
```

`CrossAppDispatcher.notifyNewMessage` expects `recipientIds: string[]` — actual user IDs of conversation participants. Passing the conversation ID means notifications route to nobody (no user has that ID) or to the wrong person if IDs collide. The fix requires fetching conversation participants before dispatching.

## EventBus lifecycle and scaling

The wildcard subscription in `registerEventHandlers` means every event gets broadcast twice to WebSocket clients: once via the `*` handler (relayed to `event.channel`) and once via the channel-specific handler (e.g., `messages` handler relays to `conversation:{id}`). This creates duplicate delivery for events on named channels.

If `registerEventHandlers` is called during hot module reload, handlers accumulate without limit because there's no guard against double-registration. Adding `bus.clear()` before registration would prevent this.

For multi-instance deployments, the EventBus provides no cross-process delivery. Events published in one ws-gateway instance are invisible to clients connected to another. The typical solution is backing the bus with Redis pub/sub or NATS.

## Cross-app search: empty index on cold start

The admin `/api/search` route instantiates `CrossAppSearchService` at module scope. Since the underlying search is backed by an in-memory `Map`, it starts empty every cold start. Without a persistent index or indexing pipeline running, searches will always return zero results in production. The search infrastructure exists but nothing feeds it data at startup.

## `as never` type escapes

```typescript
this.fanout = new NotificationFanout(defaultPrefs as never);
```

This suppresses the type mismatch between the inline object and the full `PreferenceService` class interface. If `PreferenceService` adds a required method, this will compile fine and crash at runtime. A proper fix is to extract a narrower interface (e.g., `NotificationPreferences`) that `NotificationFanout` depends on instead of the full class.
