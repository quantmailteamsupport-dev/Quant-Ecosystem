# Rewrite 42 fake components to real React JSX with Tailwind CSS

Forty-two components across seven apps that previously returned plain JavaScript object trees (`{ type: 'div', children: [...] }`) are rewritten to proper React JSX with Tailwind CSS, semantic HTML, and ARIA attributes. The conversion follows a consistent pattern: destructure props, build the UI with Tailwind utility classes, add `aria-label`/`role` annotations, enforce 44px minimum tap targets, and preserve the original named + default export structure. State management is introduced where the former fakes had none (ChatInterface, LiveChat, ExportDialog gain `useState`), and expensive computations are memoized (EcosystemMap).

Watch for: SVG `<line>` elements use `calc()` in coordinate attributes, which is invalid SVG and will not render (confirmed). `role="img"` on a container with interactive buttons hides those buttons from assistive technology (confirmed). Several components mix `min-h-[44px]` with a smaller explicit size class, creating dead CSS (confirmed). Multiple interactive `<button>` elements are missing `type="button"` (confirmed).

## High-level view

The EcosystemMap SVG implementation uses `calc()` expressions in `x2`/`y2` attributes of `<line>` elements. SVG geometry attributes do not support `calc()`, so the connection lines will not render. This is the most significant functional bug in the batch.

EcosystemMap also uses `role="img"` on a container that holds interactive `<button>` elements for each app node. Assistive technology treats the entire subtree as a single image, hiding those buttons.

Buttons in ExploreGrid, PostGrid, some shop/games page buttons, and ContentGrid's GridItem (which uses a `<div>` with click handlers rather than a `<button>`) omit `type="button"`, defaulting to `type="submit"` and risking accidental form submissions.

The VoteButtons component applies both size-specific classes (`h-7 w-7`, `h-9 w-9`, `h-11 w-11`) and an unconditional `min-h-[44px] min-w-[44px]`. The minimum always wins below `large`, so the size prop is effectively dead for `small` and `medium`.

Six components suppress `noUnusedParameters` via `void _onProp` statements. The pattern works but signals that prop interfaces are wider than what the components currently implement.

PlacementSelector hides checkboxes with `sr-only` but the wrapping `<label>` has no `focus-within` ring, leaving keyboard users without a visible focus indicator.

## EcosystemMap SVG coordinate bug

The SVG `<line>` elements attempt to use CSS `calc()` for their `x2` and `y2` attributes:

```tsx
<line
  x1="50%"
  y1="50%"
  x2={`calc(50% + ${x}px)`}
  y2={`calc(50% + ${y}px)`}
  ...
/>
```

SVG presentation attributes for geometry (`x1`, `y1`, `x2`, `y2`) accept only lengths or percentages, not `calc()`. Browsers will discard these values and the lines will not render. The fix is to compute the absolute pixel positions (e.g., using the container's known dimensions and `useRef`) or position the lines via a `<g transform="translate(...)">` approach. Confidence: **confirmed** -- SVG spec does not support `calc()` in geometry attributes.

## Missing `type="button"` on buttons

Buttons without an explicit `type` attribute default to `type="submit"` per the HTML spec. If any of these components are ever placed inside a `<form>`, clicking them would trigger form submission. The affected components:

- ExploreGrid: `<button key={item.id} ...>` (no type)
- PostGrid: `<button key={post.id} ...>` (no type)
- ContentGrid GridItem: `<div>` used with `onClick`/`onKeyDown`/`tabIndex` instead of a native `<button>` -- loses built-in keyboard semantics and the implicit button role
- GamesPage, ShopPage: inconsistent -- some buttons have `type="button"`, others do not

Confidence: **confirmed** -- visible in the diff.

## VoteButtons size prop ineffectiveness

The component defines size-specific classes (`h-7 w-7`, `h-9 w-9`, `h-11 w-11`) but also unconditionally applies `min-h-[44px] min-w-[44px]`. Since 44px > 28px (`h-7`) and 36px (`h-9`), only the `large` variant (44px = `h-11`) is unaffected. The `small` and `medium` sizes render identically to `large`. Confidence: **confirmed**.

## `void _prop` unused-handler pattern

Six components use this pattern:

```tsx
export function EffectsPanel({ onUpdateParams: _onUpdateParams, ... }) {
  void _onUpdateParams;
  ...
}
```

The `void` statement satisfies `noUnusedParameters` but reads as dead code. The pattern appears in EffectsPanel (`onUpdateParams`), LayerPanel (`onReorder`, `onRename`), ToolPanel (`onToolAction`), LiveChat (`streamId`), VideoFeed (`onScroll`), DevicePreview (`onType`). If the handlers are genuinely not wired up yet, a `// TODO` comment would communicate intent. If the interface is wider than needed, the prop could be removed or marked optional. Confidence: **confirmed**.

## Accessibility contradiction in EcosystemMap

The root container has `role="img"`, which tells assistive technology to treat the entire subtree as a single image. The container includes interactive `<button>` elements for each app node with detailed `aria-label` attributes. Buttons inside `role="img"` are effectively hidden from screen readers -- they become part of the "image" and lose their interactive semantics. The container should use `role="region"` or no role at all. Confidence: **confirmed**.

## PlacementSelector hidden checkbox focus

The checkbox `<input>` is wrapped in a styled `<label>` and hidden with `className="sr-only"`. The label itself does not receive a `:focus-within` ring or any visual indicator when the checkbox gains keyboard focus. A keyboard-only user tabbing through placements would not see which item is currently focused. Adding `focus-within:ring-2 focus-within:ring-indigo-500` to the label's className would fix this. Confidence: **confirmed**.
