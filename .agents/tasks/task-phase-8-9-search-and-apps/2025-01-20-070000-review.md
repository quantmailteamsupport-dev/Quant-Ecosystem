# Universal Search and App-Specific Product Completion

Phase 8 adds four search-layer services (saved searches with alerts, observability metrics, reindex job management, NL query enhancement) and wires them into the search-indexer's Kafka pipeline. Phase 9 fills in core service methods across all 13 apps and introduces a CrossAppOrchestrator in QuantAI that operates across mail, chat, docs, calendar, and drive with a demo connector. The work is substantial (6,254 lines across 90 files) but structurally uniform: each app gets CRUD methods + tests following the same Prisma-backed pattern.

**Watch for:** Reindex API routes are mounted on the health server with no auth (confirmed). The orchestration routes use a wildcard `'*'` permission entry that grants all demo users full access without real auth middleware (confirmed). In-memory stores for SavedSearchService, ReindexJobManager, and SearchObservabilityService lose all state on restart (confirmed). The `copyFile` method in QuantDrive copies the raw encryption key into a new record, meaning two files share the same AES key material (confirmed).

## High-level view

The search package grows four new services, all using in-memory Maps as their backing store. SavedSearchService performs substring matching against indexed documents to trigger alerts. SearchObservabilityService accumulates latency records with no eviction policy. ReindexJobManager tracks job state transitions but never actually performs reindexing—it's a state machine without a worker. NLQueryEnhancer wraps the existing QueryParser with regex-based intent detection and entity extraction.

The search-indexer gains two integration points: saved search matching runs inline after every successful event handler invocation (wrapped in a try-catch so failures don't block indexing), and a set of reindex REST routes are registered on the health server's Fastify instance. The reindex routes have no authentication or authorization middleware.

The CrossAppOrchestrator defines an `AppConnectors` interface that abstracts five apps behind async methods, enforces per-user permission checks before any cross-app call, and produces citation-annotated results. The orchestration routes hard-code a wildcard permission map and fall back to `'demo-user'` when no auth context exists—an open-by-default pattern that needs explicit gating before production.

QuantDrive's FileService encrypts at rest with AES-256-GCM per file, but `copyFile` replicates the encryption key into the new record rather than re-encrypting with a fresh key. QuantChat's MessageService includes E2E encryption with per-recipient RSA-wrapped session keys, but `searchMessages` queries the `content` field which stores JSON ciphertext for encrypted messages—making encrypted message search non-functional.

The `ThreadService.muteThread` and `snoozeThread` use `as never` casts because the Prisma schema lacks those columns. QuantSync's `getBookmarks` method queries posts owned by the user rather than a bookmarks join table—the storage and retrieval mechanisms are disconnected.

<summary>Issues (9)</summary>

1. **Reindex API has no auth** — The reindex routes (`POST /api/v1/reindex/:indexName`, `DELETE /api/v1/reindex/jobs/:jobId`) are registered on the health server with zero authentication. Add auth middleware or move to a separate admin-only Fastify instance.
2. **Orchestration routes wildcard permissions** — The `'*'` key in the permissions map grants any user full access to all apps. When real auth is added, the fallback logic (`if (!permissions[userId]) permissions[userId] = permissions['*']`) will silently grant full access to any authenticated user not explicitly listed.
3. **In-memory stores lose state on restart** — SavedSearchService, ReindexJobManager, and SearchObservabilityService use in-memory Maps with no persistence. Saved searches, running reindex jobs, and metrics history vanish on process restart. Document whether this is intentional demo scope or needs persistence before production.
4. **Unbounded metrics accumulation** — SearchObservabilityService appends to an array with no eviction, TTL, or max-size constraint. Under sustained query load this will OOM the process.
5. **copyFile shares encryption key** — `FileService.copyFile` copies `encryptionKey` verbatim into the new record. Compromise of one file's key exposes the other. Re-encrypt on copy.
6. **Encrypted message search is non-functional** — `MessageService.searchMessages` does a `content: { contains: query }` filter, but E2E-encrypted messages store JSON ciphertext in that field. Search will never match encrypted messages meaningfully.
7. **Thread mute/snooze schema gap** — `ThreadService.muteThread` and `snoozeThread` use `as never` casts because `isMuted`/`snoozedUntil` don't exist in the Prisma schema. These writes will silently fail or throw at runtime depending on Prisma's strict mode.
8. **getBookmarks queries wrong table** — `FeedService.getBookmarks` queries `post.findMany({ where: { userId } })` which returns the user's own posts, not their bookmarked posts. Needs a bookmarks join table or a `bookmarkedBy` relation query.
9. **Saved search matching is substring-only** — `matchNewDocument` splits the saved query on whitespace and checks `content.includes(term)`. This will false-positive on partial word matches (e.g., query "art" matches "started") and cannot express field-scoped or boolean queries. Acceptable for v1 but worth noting.

## Reindex and admin API surface on the health server

The `registerReindexRoutes` function registers four routes (`POST /api/v1/reindex/:indexName`, `GET /api/v1/reindex/jobs`, `GET /api/v1/reindex/jobs/:jobId`, `DELETE /api/v1/reindex/jobs/:jobId`) directly on the Fastify instance returned by `startHealthServer`. This is the same server that exposes `/health` and `/ready` endpoints. Any network-reachable client that can hit the health port can trigger reindex jobs or cancel them. Since health ports are often exposed without ingress-level auth (they're targets for Kubernetes probes), this is a privilege escalation path in any environment where the health port is network-accessible.

`ReindexJobManager.startReindex` only creates a state-machine entry—no worker loop, no connection to MeiliSearch or Qdrant to trigger a rebuild. The route returns `201 Created` with a job in `pending` state, but nothing ever transitions it to `running`. The API contract implies functionality that doesn't exist yet.

## CrossAppOrchestrator permission model

In the route layer (`orchestration.ts`), if a user's ID isn't in the permissions map AND there's a `'*'` key, `ensurePermissions` copies the wildcard permissions into the user's entry. This means the first request from any user permanently grants them wildcard access for the lifetime of the process. The pattern is an open-by-default fallback that should be gated behind an explicit `DEMO_MODE` flag before real auth arrives.

The `scheduleMeeting` scenario computes `hasConflict` by checking if any existing event's time range overlaps `preferredTime`, but creates the event unconditionally regardless of conflict. The conflict boolean is returned in the response but never blocks creation—whether this is intentional "warn but proceed" should be explicit in the API contract.

## Saved search alert matching in the indexing pipeline

After each event is successfully handled, `routeEvent` calls `savedSearchService.matchNewDocument` inline. The matching logic extracts content from the event payload by checking multiple field names (`bodyPlain`, `content`, `description`, `title`) with fallback to empty string.

When matches are found, the only action is a `logger.info` call. No notifications are sent, no webhook is fired, no state is updated. The `matchNewDocument` call doesn't invoke `markAlerted`, so `getAlertsDue()` will keep returning the same saved searches as due on every poll. The alert pipeline is wired for detection but has no delivery mechanism.

The matching is synchronous and runs on the Kafka consumer's event loop. The iteration over all saved searches for every indexed document is O(searches \* terms) per event. Under heavy indexing load with many saved searches, this could become a latency concern for event processing throughput.

## NLQueryEnhancer regex-based classification

Intent detection checks action patterns first, then navigational, defaulting to informational. This ordering means "show me what changed since yesterday" classifies as `action` (matches "what changed") rather than `navigational` (matches "show me"). The overlapping patterns create priority-dependent classifications that may surprise callers.

Entity extraction uses single `match()` calls per pattern, so only the first occurrence is captured. A query like "from Alice and from Bob" will only extract "Alice". The deduplication via `seen` set prevents the same entity appearing twice but doesn't help with multiple distinct entities matching the same pattern type.

## QuantDrive encryption key propagation on copy

`FileService.copyFile` creates a new record with identical `encryptionKey`, `encryptionIV`, and `encryptionAuthTag`. Since AES-GCM security requires a unique (key, nonce) pair, reusing the same key and IV for what is effectively the same ciphertext isn't a cryptographic break (it's the same data), but it means:

1. If an attacker obtains the key for either file record, they can decrypt both.
2. If the copy is later updated (via `updateFile`), the original version stored in `fileVersion` for the copy will still reference the old shared key.

The mitigation: `copyFile` should decrypt and re-encrypt with a fresh key, or at minimum generate a new key+IV pair if the ciphertext is duplicated.

## QuantChat encrypted message search gap

`MessageService.sendMessage` with `encryption: 'e2e'` stores `JSON.stringify(encryptedPayload)` as the message content. `searchMessages` then does `content: { contains: query, mode: 'insensitive' }` against this field. The search will match against base64 ciphertext strings and JSON structure characters, never against the original plaintext. The service doesn't distinguish encrypted from unencrypted messages in the search path—at minimum, encrypted messages should be excluded from search results or a search index of message metadata (sender, timestamp, conversation) should be maintained separately.

## Thread schema debt with `as never` casts

`ThreadService.muteThread` updates with `{ isMuted: true } as never` and `snoozeThread` with `{ snoozedUntil: until } as never`. The `as never` cast bypasses TypeScript's type checking entirely. If the Prisma schema doesn't have these columns, the generated SQL will either include them (and fail at the database level) or Prisma will strip unknown fields silently depending on the version. Either way, the feature is non-functional until the schema migration adds these columns. The cast hides what should be a compile-time error.

## QuantSync getBookmarks implementation

`FeedService.getBookmarks` executes:

```typescript
this.prisma.post.findMany({
  where: { userId, deletedAt: null },
  ...
})
```

This returns posts created by the user, not posts bookmarked by the user. The `PostService.bookmark` method (also in this diff) stores bookmarks in the post's metadata JSON:

```typescript
data: { metadata: { ...metadata, bookmarkedBy: [...existing, userId] } }
```

The feed method and the bookmark storage mechanism are disconnected—`getBookmarks` would need to query posts where `metadata.bookmarkedBy` contains the userId, or use a separate bookmarks table.
