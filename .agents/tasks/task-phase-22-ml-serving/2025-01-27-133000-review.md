# Real ML Serving: Triton client, feature pipeline, hybrid search, A/B experimentation, and user agency

Phase 22 replaces the in-memory random-scoring NeuralCF with a Triton Inference Server client, builds a real-time feature aggregation pipeline over NATS, adds a hybrid search system (BM25 + vector + Cohere reranker), introduces an A/B experiment framework with sticky bucketing and guardrail auto-rollback, and ships a set of user-agency features (time-well-spent, following-mode, show-me-less, feed explainability). The approach is interface-based DI throughout: every external dependency (Triton, NATS, Redis, Qdrant, OpenAI, Cohere) is abstracted behind a TypeScript interface, so all production code is testable against mocked transports. The old NeuralCF stays alive as a test fixture and local-dev fallback inside the new `NeuralCF` class.

**Watch for:** FollowingMode fetches _all_ posts from all followed authors into memory before paginating (confirmed). The FeatureAggregator accumulates events in memory with no eviction, making it a slow memory leak in any long-running process (confirmed). NatsFeatureConsumer silently drops malformed messages with no observability hook (confirmed). The Triton Helm entry omits GPU resource requests and `nvidia.com/gpu` limits, so Kubernetes will never schedule GPU-backed inference (likely).

## High-level view

The `@quant/triton-client` package is a thin HTTP v2 protocol wrapper with Zod validation and a `TritonTransport` interface for DI. The NeuralCF refactor uses this client with a graceful fallback chain: Triton -> InMemoryNeuralCF -> constant 0.5. However, the production module unconditionally imports the ~300-line test fixture at the top level.

The feature store pipeline introduces three new classes (`FeatureAggregator`, `NatsFeatureConsumer`, `BackfillJob`) that form an event-sourced aggregation system. The aggregator computes sliding-window features per user but never prunes events older than its 7-day window, meaning unbounded memory growth. The NATS consumer silently drops parse failures with no dead-letter or logging mechanism.

The hybrid search pipeline (BM25 via MeiliSearch + vector via Qdrant + Cohere reranker) has multi-level fallback: if reranker fails, return fused results; if vector fails, try BM25 alone; if BM25 fails, try vector alone. The `EmbeddingService` routes English to OpenAI and Indic languages to a self-hosted multilingual-e5 on Triton, but the `embed()` method detects language from the concatenation of all input texts, misrouting mixed-language batches.

The A/B experiment service implements deterministic FNV-1a bucketing with sticky persistence. Sticky bucketing means that changing `trafficAllocation` on a live experiment only affects users who haven't been bucketed yet -- a non-obvious constraint for operators.

User-agency features (UserSignalProcessor, TimeWellSpent, FollowingMode, ScreenTimeMonitor, FeedExplainer) are self-contained classes with in-memory stores. FollowingMode fetches all posts from all followed accounts into memory on every page request, defeating the `ContentStore` interface's limit/offset contract.

<summary>Issues (7)</summary>

1. **FollowingMode unbounded fetch** -- `getFeed` passes `limit: Number.MAX_SAFE_INTEGER` to the content store, loading all posts into memory. Use the store's `limit`/`offset` natively and accept approximate `totalAvailable` or add a `count` method.
2. **FeatureAggregator memory leak** -- Events accumulate in `userStates` indefinitely. Add time-based or count-based eviction after the sliding window expires (>7 days) to prevent OOM in long-running services.
3. **NatsFeatureConsumer silent drops** -- Malformed messages are caught and swallowed. Add a counter/metric or dead-letter callback so operators can detect bad producers.
4. **Triton Helm missing GPU resources** -- The values.yaml sets CPU requests/limits but omits `nvidia.com/gpu` in resources. Without this, Triton will run CPU-only on a GPU cluster, or the scheduler won't place it on GPU nodes.
5. **FeatureAggregator flushToOnlineStore partial failure** -- If one user's `setFeatures` call fails, the loop breaks with an unhandled rejection. Users flushed before the failure have `dirty = false` but the remaining are skipped. Consider try/catch per-user or batch-atomic writes.
6. **EmbeddingService mixed-language batch** -- `embedBatch` detects language per-item but `embed()` detects language from the _concatenation_ of all texts. A batch of mixed-language texts routed through `embed()` (not `embedBatch`) will all go to whichever backend the concatenated text triggers.
7. **NeuralCF production import of test fixture** -- `neural-cf.ts` imports `InMemoryNeuralCF` from `../__tests__/fixtures/in-memory-ncf` at the module level. This pulls test fixture code into the production bundle. Make it a lazy dynamic import or move the fallback to a separate entrypoint.

## NeuralCF production import of test fixture

The `NeuralCF` class imports `InMemoryNeuralCF` from `../__tests__/fixtures/in-memory-ncf` at the top of the file:

```typescript
import { InMemoryNeuralCF } from '../__tests__/fixtures/in-memory-ncf';
```

This is a production module importing from a test-fixtures directory. Even if the class is only instantiated in fallback mode, the module is always loaded, pulling ~300 lines of random-seeded neural network simulation into the production bundle. In a tree-shaking-unfriendly monorepo (CommonJS or barrel re-exports), this will be included in the build output regardless of whether fallback mode is active.

The `hashId` function uses a simple left-shift hash that's adequate for distributing IDs across tensor indices but provides no collision guarantees. For a recommendation system where user/item ID collisions mean wrong scores, this will collide often for short numeric IDs (possible).

## FeatureAggregator memory growth and flush semantics

The aggregator stores every event in `userStates[userId].events` and only filters by time when _computing_ features. No events are ever removed from the array.

```typescript
processEvent(event: UserEvent): void {
  state.events.push(event);  // never pruned
  state.dirty = true;
}
```

In a service processing thousands of users with hundreds of events per day, this grows without bound. Adding a prune step in `processEvent` or `getAggregatedFeatures` that drops events older than `SEVEN_DAYS_MS` would cap memory per user.

The `flushToOnlineStore` method iterates the user map and calls `setFeatures` for each dirty user. If any call rejects, the async iteration throws and remaining users won't flush until the next invocation -- but already-flushed users have `dirty = false`, so their data is consistent. There's no error isolation between users.

## NatsFeatureConsumer observability gap

```typescript
private handleMessage(data: Uint8Array): void {
  try {
    // ...parse and process...
  } catch {
    // Malformed messages are silently dropped
  }
}
```

In a production event pipeline, silent drops make it impossible to detect schema evolution bugs, producer misconfiguration, or encoding issues. The class needs at minimum an error callback or metric increment hook in the constructor options.

## FollowingMode in-memory sort of entire following graph

```typescript
const posts = this.contentStore.getPostsByAuthors(following, {
  limit: Number.MAX_SAFE_INTEGER,
  offset: 0,
});
posts.sort((a, b) => b.timestamp - a.timestamp);
const items = posts.slice(offset, offset + pageSize);
```

For a user following 500 accounts each posting 10 times/day, this loads 5,000 posts into memory and sorts them on every page request. The `ContentStore` interface already accepts `limit` and `offset` parameters, so the pagination should be pushed down to the store layer. The `totalAvailable` count could either be fetched via a separate count query or returned as an estimate.

## EmbeddingService language detection asymmetry

`embed(texts, language?)` concatenates all texts to detect language when no explicit language is given:

```typescript
async embed(texts: string[], language?: string): Promise<number[][]> {
  const detectedLanguage = language ?? this.detectLanguage(texts.join(' '));
  const backend = this.routeToBackend(detectedLanguage);
  return backend.embed(texts);
}
```

If a batch contains mixed English and Indic text, the concatenation might or might not trigger Indic detection depending on proportion. `embedBatch` handles this correctly by detecting per-item, but `embed` does not. Callers who pass mixed-language batches through `embed()` will get all items misrouted to one backend.

## Triton Helm values and GPU scheduling

The Helm entry sets CPU requests/limits but omits `nvidia.com/gpu`:

```yaml
resources:
  requests:
    cpu: 500m
    memory: 1Gi
  limits:
    cpu: 4000m
    memory: 8Gi
```

The model config (`config.pbtxt`) specifies `KIND_CPU` instance groups, which is consistent for a CPU-only deployment. But the `tritonserver:24.01-py3` image is the full GPU-capable image (1.5GB+ pull). If CPU-only is intentional, consider `tritonserver:24.01-py3-min` to save image pull time and reduce attack surface. If GPU inference is the target, add GPU resource requests.

## ScreenTimeMonitor session overwrite

`startSession` overwrites any existing session state for a userId without checking whether one is already active. If a user has an active session and the client calls `startSession` again (e.g., on app foreground), the previous session's engagement-trap counter and duration are lost. There's no `resumeSession` or session-activity check -- the engagement-trap detection resets silently (possible).

## Test coverage gaps

Not tested: the production import path of `NeuralCF` when a real `TritonInferenceClient` is provided (all tests use the fallback/mock path). The `flushToOnlineStore` error-handling path. The `EmbeddingService.embed()` mixed-language batch scenario. FollowingMode behavior with large following lists (performance regression test). ScreenTimeMonitor `startSession` overwrite behavior.
