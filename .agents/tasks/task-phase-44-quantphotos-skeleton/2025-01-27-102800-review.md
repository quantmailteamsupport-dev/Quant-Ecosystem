# Photo library skeleton with face clustering, object detection, and multi-signal search

The `@quant/photos` package introduces an in-memory photo library with AI-backed face clustering (cosine similarity, threshold 0.7), COCO-style object detection with category mapping, and a multi-signal search engine that weights face matches (4), objects (3), location proximity (2), and time range (1). Editor capabilities (MagicEraser, Unblur, Cinematic, BestTake) are interface-only stubs awaiting real implementations. Hindi natural-language keyword mapping provides localized search entry.

Watch for: centroid drift in face clustering produces incorrect groupings as clusters grow (confirmed); `removePhoto` leaves orphan references in albums and face clusters (confirmed); search scans the entire store on every query via `listByDateRange(0, Date.now() * 2)` with an arbitrary upper bound (confirmed).

## High-level view

The face clustering algorithm uses a single-pass greedy approach where the first face in a group becomes the permanent centroid. This means the centroid never updates as more faces are absorbed into the cluster, so later faces are compared against an increasingly stale reference point. For a skeleton this is functional, but it will produce visibly wrong groupings once clusters exceed a handful of faces.

Photo removal deletes from the photos map but leaves stale IDs in `Album.photoIds`, `FaceCluster.faceIds`, and any cached search results. The album's `getAlbumPhotos` silently filters out missing photos, but `FaceEngine.getPhotosForPerson` will scan for face IDs that no longer exist, and cluster membership counts will be wrong.

Search iterates every photo in the store on every call, using a synthetic upper bound of `Date.now() * 2` to bypass the date range filter. The location proximity check uses Manhattan distance on raw lat/lng degrees, meaning "within 1 degree" is roughly 111km at the equator but varies significantly at other latitudes.

The `parseNaturalQuery` Hindi-to-English map covers five words and is entirely disconnected from the object label vocabulary in ObjectEngine's CATEGORY_MAP — "family" maps to no COCO category, so a Hindi query for "parivaar" produces a label that will never match any photo's detected objects.

<details>
<summary>Issues (6)</summary>

1. **Stale centroid in face clustering** — The cluster centroid is set to the first face's embedding and never recalculated. Recompute centroid as the mean of all member embeddings after each addition, or switch to incremental averaging.
2. **Orphaned references on photo removal** — `removePhoto` does not clean up `Album.photoIds` or `FaceCluster.faceIds`. Add cascade logic or document that callers must remove from albums/clusters first.
3. **Unbounded full-store scan in search** — `store.listByDateRange(0, Date.now() * 2)` materializes the entire photo collection into an array on every search call. Expose a direct iterator or `listAll()` method instead of abusing the date range filter.
4. **Hindi "family" label mismatch** — `parseNaturalQuery` maps "parivaar" to "family", but no COCO label or category matches "family". The query will always return zero results for that keyword.
5. **Location proximity uses raw degree distance** — Manhattan distance on latitude/longitude treats 1 degree equally everywhere. At 60N latitude, 1 degree of longitude is ~55km vs ~111km at the equator. Use haversine or at minimum apply a cos(lat) correction.
6. **No persistence or size bounding** — The in-memory maps grow without limit. Acceptable for a skeleton, but the type definitions and PhotoStore API should signal whether this is intended to be backed by persistent storage later (e.g., an async interface, or a capacity limit).

## Face clustering: greedy single-pass with static centroid

The clustering algorithm in `FaceEngine.clusterFaces` iterates the face list once. For each unassigned face, it creates a new cluster with that face's embedding as the centroid, then sweeps the remaining faces to find those within cosine similarity >= 0.7. The centroid is never updated:

```typescript
const cluster: FaceCluster = {
  id: crypto.randomUUID(),
  centroid: [...face.embedding],
  faceIds: [face.id],
};
// ... absorbs nearby faces, but centroid stays as first face's embedding
```

This means faces 3, 4, 5 absorbed into a cluster are compared against face 1's embedding, not the cluster's actual mean position. In high-dimensional face embedding spaces (typically 128-512 dims), this causes boundary faces to be assigned to the wrong cluster or left as singletons when they should belong to an existing group.

The method also mutates `this.clusters` as a side effect, appending to the internal map. Calling `clusterFaces` multiple times with overlapping face sets will create duplicate clusters without deduplication.

## Orphaned references on photo deletion

`PhotoStore.removePhoto` calls `this.photos.delete(id)` and returns. The photo's `albumIds` array is not consulted — albums referencing the deleted photo retain its ID in their `photoIds` array. `getAlbumPhotos` filters these out at read time:

```typescript
return album.photoIds.map((id) => this.photos.get(id)).filter((p): p is Photo => !!p);
```

But `album.photoIds.length` will be wrong, and any UI showing album counts will be inaccurate. The same issue exists in `FaceCluster.faceIds` — the face engine retains references to faces from deleted photos, causing `getPhotosForPerson` to search for face IDs that will never match.

## Search iteration strategy and location math

`PhotoSearch.search` begins by calling `store.listByDateRange(0, Date.now() * 2)`. The `Date.now() * 2` upper bound is a workaround to get "all photos" from an API that requires a range. This materializes the entire map into an array on every call and bakes in an assumption about timestamp magnitude.

The location proximity check:

```typescript
const dist = Math.abs(photo.location.lat - query.location.lat) +
             Math.abs(photo.location.lng - query.location.lng);
if (dist < 1) { ... }
```

This is Manhattan distance on degrees. One degree of latitude is always ~111km, but one degree of longitude varies from ~111km at the equator to 0km at the poles. A search centered at Mumbai (lat 19) treats "within 1 degree" as roughly 111km N-S but ~105km E-W, which is close enough. At Helsinki (lat 60), the E-W distance is only ~55km for 1 degree of longitude. The behavior is inconsistent and will surprise users at higher latitudes.

## Hindi keyword map dead ends

`parseNaturalQuery` maps five Hindi words to English equivalents, then checks if the resulting word is in a hardcoded list `['beach', 'food', 'home', 'travel', 'family']`. Two of these ("home" and "family") don't correspond to any label in ObjectEngine's CATEGORY_MAP or any COCO category name. A query for "parivaar" (family) will produce `{ objectLabel: 'family' }`, but no photo will ever have `objects.some(o => o.label === 'family')` because the object detector uses COCO labels like "person", not semantic concepts like "family".

## Test coverage gaps

Tests cover the happy paths well: CRUD, clustering with 2-3 faces, search signal combinations, editor interface satisfaction. Not tested:

- Duplicate face IDs passed to `clusterFaces`
- Empty inputs (zero faces, zero photos, empty search query)
- `removePhoto` followed by `getAlbumPhotos` (the orphan scenario)
- Multiple calls to `clusterFaces` with overlapping face sets
- `parseNaturalQuery` with mixed Hindi/English or unknown words
- Location edge cases (antimeridian, poles, negative coordinates)
- `listByDateRange` with start > end
