# Browser agent action system with trust gating

New `@quant/browser-agent` package providing a typed browser action vocabulary, an LLM-backed action planner, session management with timeout, a trust framework that gates purchase-tier actions behind explicit user confirmation, and an audit replay recorder. The package is interface-only for actual browser interaction (no Playwright dependency); real execution is deferred to downstream consumers.

Watch for: purchase-tier classification relies solely on regex matching against CSS selectors, which is trivially bypassed by sites that name their buy buttons differently (confirmed). The spending-cap check never actually debits `currentSpend`, so repeated small purchases will never exhaust the cap (confirmed). The module-level mutable `actionCounter` in browser-actions.ts creates cross-test coupling and would collide across concurrent sessions in production (confirmed).

## High-level view

The action vocabulary is a discriminated union on `type`, with factory functions that mint sequential IDs from a module-global counter. Classification into read/write/purchase tiers happens by regex-matching the selector string against purchase-related keywords — the weakest link in the safety model, since it cannot detect purchase intent from page semantics, only from developer-chosen selector names.

The trust framework enforces a hard gate (throws) for purchase-tier actions that haven't been explicitly confirmed via `confirmAction`. The spending cap tracks a limit but never decrements it after a successful purchase, making it advisory-only. Authorization matching uses a substring `includes` check against the URL, which is too permissive for production use.

The session manager uses in-memory storage with 30-minute timeout detection that callers must poll — no automatic enforcement. Sessions expose `lastActivityAt` as a mutable property with no encapsulation.

The action planner delegates intelligence to an injected `ActionPlannerStrategy`. The `replan` method discards the failed action and error context entirely, calling `plan` again unchanged — replanning cannot adapt to the specific failure.

## Purchase classification via selector regex

The `classifyAction` function tests the selector against:

```typescript
const PURCHASE_PATTERNS = /pay|buy|purchase|checkout|place.?order|confirm.?payment|submit.?order/i;
```

This is the sole mechanism determining whether an action triggers the confirm-before-pay gate. If a real-world checkout button uses `.primary-cta` or `#step-4-submit`, the action is classified as write-tier and proceeds without confirmation. The trust framework's safety guarantee is only as strong as this regex.

The `scroll` action is classified as `read_only`, which becomes debatable if scrolling triggers infinite-scroll loads that cost API calls or data transfer in downstream integrations.

## Spending cap is check-only, never debited

```typescript
checkSpendingCap(sessionId: string, amount: number): boolean {
  const cap = this.caps.get(sessionId);
  if (!cap) return true;          // no cap = always pass
  return cap.currentSpend + amount <= cap.sessionLimit;
}
```

`currentSpend` is initialized to 0 and never mutated anywhere in the codebase. The check will always compare `0 + amount <= limit`, meaning the cap is effectively infinite as long as individual transactions stay below the limit. A `debit` method or an internal increment on confirmed purchase actions is needed for the cap to function.

When no cap is set (`caps.get` returns undefined), `checkSpendingCap` returns `true` — uncapped sessions have no spending protection. Whether this fail-open behavior is intentional should be explicit.

## Module-global action ID counter

```typescript
let actionCounter = 0;
const nextId = () => `act_${++actionCounter}`;
```

Module-level mutable state shared across all sessions in the same process. IDs are non-deterministic for replay and debugging. In test suites with parallel workers, the counter leaks across files. Inject an ID generator or use crypto.randomUUID().

## Replan ignores failure signal

```typescript
async replan(
  goal: string,
  pageState: PageState,
  _failedAction: BrowserAction,
  _error: string,
): Promise<ActionSequence> {
  return this.plan(goal, pageState);
}
```

The strategy receives no information about the failure. An LLM-backed strategy would regenerate the same plan unless `pageState` happens to have changed externally. The underscore-prefixed parameters represent dead functionality.

## Authorization substring matching

```typescript
isAuthorized(url: string, userId: string): boolean {
  for (const [key, auth] of this.authorizations) {
    if (key.startsWith(`${userId}:`) && auth.granted && url.includes(auth.sitePattern))
      return true;
  }
  return false;
}
```

`url.includes(auth.sitePattern)` with a pattern like `"shop.com"` matches any URL containing that substring: `"https://evil-shop.com"`, `"https://x.com/shop.com-phishing"`. For a trust framework guarding browser navigation, this needs host-based matching (parse the URL, compare against the pattern as a domain suffix).

## Test coverage gaps

- No test validates spending cap denial after debiting (because debiting doesn't exist).
- No test verifies `classifyAction` for actions whose selectors lack purchase keywords but semantically represent purchases.
- `SessionManager` tests directly mutate `lastActivityAt` on the returned object, revealing that `BrowserSession` fields are mutable from outside the manager with no encapsulation boundary.
