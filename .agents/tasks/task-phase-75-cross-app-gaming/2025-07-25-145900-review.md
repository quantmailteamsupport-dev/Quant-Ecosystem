# Cross-app gaming framework (v2)

The six issues flagged in v1 have been addressed. The anonymous identity data model is properly split into internal/public types; `checkGameAccess` now performs real content-rating enforcement; the regional leaderboard filter correctly matches on the `region` field; consent revocation is implemented and wired into `getDisplayIdentity`; abandoned session cleanup exists via `cleanupAbandonedSessions`; and `getAvailableContexts` respects registered game capabilities. The fixes are structurally sound and well-tested.

**Watch for:** Unregistered games bypass the safety gate entirely — `checkGameAccess` returns `true` for any game not in the registry, making the content-rating system fail-open for games that haven't been explicitly registered (confirmed). The leaderboard accumulates every score submission per player per game without deduplication, growing without bound (confirmed). The `cleanupAbandonedSessions` method exists but is caller-driven with no built-in scheduling, meaning abandoned sessions still leak if nobody calls it (likely).

**Verdict**: APPROVED

## High-level view

The identity fix cleanly separates concerns: an `InternalAnonymousIdentity` interface holds `realUserId` privately within the service, and the public-facing `PublicAnonymousIdentity` type strips it. The method constructs a new object containing only public fields before returning, so property leakage is prevented at the type level and at runtime.

The game access control rework introduces a `GameRegistryEntry` type with `contentRating` and a numeric level comparison. The check is correct for registered games, but unregistered games default to allowed — the safety gate is fail-open for any game that hasn't been explicitly registered.

Session cleanup and consent revocation are implemented correctly. Cleanup is caller-driven (no auto-scheduling), and revocation properly breaks the mutual-consent check in both directions.

The leaderboard has a structural issue: `submitScore` appends unconditionally, meaning repeated submissions from the same player grow the array without bound. This isn't a correctness bug but an operational concern for any long-running instance.

## Content-rating enforcement and the fail-open default

The `checkGameAccess` implementation maps age groups to maximum allowed rating levels and compares numerically:

```typescript
const AGE_GROUP_MAX_RATING: Record<AgeGroup, GameContentRating> = {
  under13: 'everyone',
  teen: 'teen',
  adult: 'mature',
};

const RATING_LEVELS: Record<GameContentRating, number> = {
  everyone: 0,
  teen: 1,
  mature: 2,
};
```

The enforcement logic is correct: a game rated `teen` (level 1) is blocked for `under13` (max level 0) but allowed for `teen` and `adult`. The concern is the early return for unknown games:

```typescript
const game = this.gameRegistry.get(gameId);
if (!game) {
  return true; // Unknown game defaults to allowed
}
```

In a platform where third-party games can be loaded, an unregistered game is treated as universally accessible. The system is fail-open for the safety gate. If the expectation is that all games must be registered before they're playable, this should be fail-closed. If registration is optional and only used for restriction, the current behavior is intentional but creates a gap where a game with mature content can be accessed by minors if it hasn't been registered yet.

## Leaderboard accumulation without bounds

Every `submitScore` call pushes a new entry unconditionally:

```typescript
const entries = this.scores.get(key) ?? [];
entries.push(entry);
this.scores.set(key, entries);
```

There is no check for whether the player already has an entry for this game. `getPlayerRank` uses `findIndex` on the sorted array, which returns the first (highest-scoring) match — so rank calculation is accidentally correct. But all entries occupy memory and appear in `getLeaderboard` results, polluting the leaderboard with duplicate player rows.

For a casual gaming platform where players submit scores after each round, a trivia game with 100 players playing 10 rounds each accumulates 1000 entries. A keep-best or keep-latest policy per player per game would bound this and produce cleaner leaderboard output.

## Test coverage gaps

Not tested:

- Behavior when `checkGameAccess` is called for a game that was registered and then had its registry entry removed (would it revert to fail-open?)
- Leaderboard with duplicate player submissions — whether `getLeaderboard` returns multiple rows per player
- Concurrent cleanup while sessions are being created/abandoned
