# Cross-app gaming framework and Quant Studio SDK

This PR introduces two new packages: `@quant/cross-app-gaming`, a multiplayer gaming framework that adapts to five app contexts with identity protection, minor safety, and anti-cheat; and `@quant/quant-studio`, an SDK and build pipeline for sandboxed mini-apps with CSP enforcement, permission gating, and remix attribution. The gaming package covers session lifecycle with host migration, a leaderboard with threshold-based anti-cheat, anonymous-then-reveal identity for random matches, and age-gated communication restrictions. The studio package provides a manifest-validated build pipeline, an IPC bridge with rate limiting, and a permission-gated SDK surface.

**Watch for:** Anti-cheat relies solely on static score thresholds (confirmed) — trivially bypassed by submitting scores just under the ceiling. The identity bridge stores `realUserId` on every anonymous identity object and returns it from `createAnonymousIdentity` (confirmed), meaning any caller with access to the service can de-anonymize players without consent. `checkGameAccess` always returns true regardless of age group (confirmed), making the safety gate vacuous. The diff also bundles a completely unrelated `quant-studio` package with no cross-references between the two.

## High-level view

The gaming session service is a well-structured in-memory state machine: create, join, pause, resume, destroy, with host migration when the host leaves. The state transitions are correct, but there's no timeout or garbage collection for abandoned sessions, which will leak memory in any long-running process.

The leaderboard's anti-cheat is a static min/max threshold check on submitted scores. It catches obviously absurd values but offers no protection against replay, rate abuse, or gradual score inflation. The threshold boundaries are configured per-service instance, not per-game, which means a single config must accommodate games with wildly different scoring ranges. The `regional` scope filter is also buggy — it includes all entries when a region is specified rather than filtering by it.

The identity bridge implements anonymous-then-revealed semantics for random matches, requiring mutual consent before revealing. However, the data model exposes `realUserId` on the anonymous identity object itself, so any code path that receives an `AnonymousIdentity` can trivially extract the real user — the anonymity is display-layer only.

Minor safety enforces communication limits and purchase blocking by age group but the core access-control method (`checkGameAccess`) unconditionally returns `true`, making it impossible to block specific games for minors. The service also has no mechanism to verify that the age group passed in actually corresponds to a verified age.

The bundled `quant-studio` package is a separate concern — a sandboxed mini-app SDK with its own build pipeline, CSP builder, IPC bridge, and permission system. It shares no code or types with cross-app-gaming and appears to have landed in this branch by accident or for convenience.

## AnonymousIdentity data model exposes what it's meant to hide

The `IdentityBridgeService.createAnonymousIdentity` method returns an `AnonymousIdentity` object that contains `realUserId`:

```typescript
interface AnonymousIdentity {
  anonymousId: string;
  realUserId: string; // <-- the thing we're hiding
  displayName: string;
  createdAt: Date;
}
```

Any service that calls `createAnonymousIdentity` — or any code that holds a reference to the returned identity object — can read `realUserId` directly. In a system where games embed as third-party content (chat embeds, feed embeds), this is a real data-exposure vector. The anonymous identity lookup should be split: a public view (anonymousId + displayName) and a private internal mapping that only the bridge itself can query.

## Session lifecycle: cleanup gap for abandoned sessions

`destroySession` deletes from the map, but `leaveSession` with the last player sets state to `abandoned` without deleting. The `listActiveSessions` method filters by state, so abandoned sessions are invisible to queries, but they remain in the `sessions` Map and the `eventLog` Map indefinitely. In a server process that runs for hours with casual games being created and abandoned, this is unbounded memory growth.

The session ID generator uses `Date.now()` plus 8 random chars:

```typescript
private generateId(): string {
  return `gs_${Date.now()}_${Math.random().toString(36).slice(2, 10)}`;
}
```

This will collide under high concurrency. Not blocking for this PR's scope but worth noting for production use.

## Anti-cheat: static thresholds per service, not per game

The `UniversalLeaderboardService` validates scores against a single `maxScoreThreshold` and `minScoreThreshold` configured at construction time. The `gameId` parameter is accepted but never used in the threshold lookup. A trivia game scoring 0-10 and a rhythm game scoring 0-1,000,000 would share the same bounds. Any player who submits scores just below the threshold passes validation trivially. There's no rate limiting, no velocity detection, and no per-player history analysis.

The `regional` scope in `getLeaderboard` has a logic bug:

```typescript
case 'regional':
  filtered = entries.filter(() => options?.region !== undefined);
  break;
```

The arrow function ignores the entry entirely and just checks whether the option exists. This returns all entries if `region` is provided, or none if it isn't — the opposite of filtering by region.

## MinorSafetyService: access control is hollow

`checkGameAccess`:

```typescript
checkGameAccess(_playerId: string, _gameId: string, ageGroup: AgeGroup): boolean {
  if (!this.config.safetyConfigs[ageGroup]) {
    return true;
  }
  return true;
}
```

Both branches return `true`. The method exists in the public API, tests assert it returns `true` for all age groups, and consumers may rely on it for access gating — but it gates nothing. If a game has mature content, this service cannot block access.

The `maxPlayTimeMinutes` config field exists in `MinorSafetyConfig` but nothing reads it or enforces a session time limit.

## Consent model: reveal works, revocation doesn't

`IdentityRevealConsent` has a `revoked: boolean` field, and the `getDisplayIdentity` method checks `!hasConsent.revoked` before revealing. But there's no public method to set `revoked = true`. A player who revealed their identity to someone has no way to take it back.

## Unrelated quant-studio package in the same diff

The `@quant/quant-studio` package adds ~900 lines of source and ~500 lines of tests with zero shared types, imports, or functionality with `cross-app-gaming`. The two packages solve different problems at different layers. The studio package appears well-constructed (permission gating, path traversal prevention, rate-limited IPC, proper CSP generation), but it belongs in its own PR.

## Test coverage gaps

The 72 tests cover happy paths and basic error cases. Not tested:

- Concurrent session operations (two players joining simultaneously)
- Host migration chains (host leaves, new host leaves, third player promoted)
- Leaderboard behavior with duplicate player submissions to the same game
- Consent revocation effects (can't test what doesn't exist)
- `maxPlayTimeMinutes` enforcement (not implemented)
- Memory growth from abandoned sessions
- Regional leaderboard filtering (the bug means it's untestable in current form)
