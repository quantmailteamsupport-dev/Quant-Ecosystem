# Wire all frontend pages to backend APIs

Fourteen pages and components across three apps (admin, QuantAI, QuantChat) have been converted from hardcoded mock arrays to live `fetch()` calls against their respective API routes. Three new API routes were created to support this (`/api/ai/stats`, `/api/database/stats`, `/api/settings`). The pattern is consistent: each page initializes with default data in state, fetches on mount, replaces state on success, and falls back to defaults on failure. The approach is sound for a wiring pass, but several structural issues need attention before this ships to anything resembling production.

**Watch for:** Settings API stores feature flags in module-level mutable state that resets on every serverless cold start (confirmed). The settings PUT endpoint has no authentication, meaning any browser can toggle maintenance mode or disable signups (confirmed). The observability components all independently fetch `/api/health` on mount rather than sharing a single subscription, creating N+1 request amplification on the observability page (confirmed).

## High-level view

The admin dashboard and apps page wire to the existing `/api/stats` and `/api/health` endpoints with a straightforward map from API shape to UI shape. The mapping in the apps page indexes into the default array for `lastDeploy` values, which will silently produce `'N/A'` if the API returns apps in a different order or count than the default array.

The settings page implements optimistic UI updates with rollback on failure, which is the one place in this diff that handles the write-then-fail case. However, the backing API route stores state in a module-scoped `let` variable, making every toggle ephemeral across serverless invocations and invisible across replicas.

The four observability components (ServiceHealthGrid, ErrorTracker, LatencyChart, ServiceMap) each independently call `fetch('/api/health')` on mount. When rendered together on the observability page, this produces four parallel requests to the same endpoint returning the same data. This is a performance concern for the health endpoint and a waste of network, though not a correctness bug.

The QuantAI ask and voice pages both POST to `/api/assistant/chat`. The response parsing uses a fallback chain (`json.response || json.message || json.data`) and degrades to `JSON.stringify(json)` if none match, which will render raw JSON blobs to users when the API contract drifts.

The camera page fires a POST to `/api/stories` on every `mouseUp` and `touchEnd` event after a shutter press. Because `touchEnd` fires alongside `mouseUp` on touch devices, `handleCapture` will be called twice per tap, creating duplicate stories.

## Settings API: in-memory state and unprotected writes

The new `/api/settings/route.ts` declares `let flags: FeatureFlag[]` at module scope. The GET handler returns this array; the PUT handler mutates it. In a serverless environment (Vercel, Lambda, etc.), this state lives only for the lifetime of a single container instance. Two consequences: flags toggled by an admin will silently revert after a cold start, and concurrent instances will serve different flag states to different users.

More critically, the PUT handler performs no authentication or authorization check. The request body is validated for shape (`id` must be a string, `enabled` must be a boolean) but not for caller identity. Any client that can reach the route can toggle `maintenance_mode` or `signup_enabled`. For a feature flags API that controls ecosystem-wide behavior, this is a significant access control gap.

```typescript
// No session check, no role check â€” direct mutation
export async function PUT(request: NextRequest) {
  const body = await request.json();
  const { id, enabled } = body as { id: string; enabled: boolean };
  // ... validates shape, then mutates module state
  flags = flags.map((f) => (f.id === id ? { ...f, enabled } : f));
}
```

The settings page itself handles the write case well: it applies the toggle optimistically, then reverts on failure. This is the strongest error-handling pattern in the entire diff.

## Duplicate story creation on camera capture

The camera page attaches `handleCapture()` to both `onMouseUp` and `onTouchEnd`:

```tsx
onMouseUp={() => {
  setIsRecording(false);
  handleCapture();
}}
// ...
onTouchEnd={() => {
  setIsRecording(false);
  handleCapture();
}}
```

On touch devices, both `touchEnd` and the synthesized `mouseUp` fire for a single tap. This means `handleCapture` executes twice, sending two POST requests to `/api/stories` for one shutter press. The status indicator will flash through `saving -> saved` twice in rapid succession, and two stories will be created server-side.

Fix: call `handleCapture` only from `onMouseUp` and add `e.preventDefault()` in `onTouchEnd` to suppress the synthesized mouse event, or switch to pointer events which unify touch and mouse.

## Observability page: four independent fetches to the same endpoint

ServiceHealthGrid, ErrorTracker, LatencyChart, and ServiceMap each contain their own `useEffect` that fetches `/api/health`. When the observability page renders, all four fire simultaneously. The health endpoint does the same work four times and returns the same response shape.

This isn't a correctness bug â€” the data is consistent â€” but it's wasteful. If the health endpoint does any real work (database queries, service pings), this multiplies that cost by four on every page load. The standard fix is to lift the fetch into the parent observability page component and pass the result as props, or to use a deduplicating data layer (SWR, React Query with a shared cache key).

## Response parsing fragility in QuantAI pages

Both the ask page and voice page parse the AI chat response with a fallback chain:

```typescript
if (json.response || json.message || json.data) {
  setResponse(json.response || json.message || json.data);
} else if (json.error) {
  setError(json.error);
} else {
  setResponse(JSON.stringify(json));
}
```

The final `else` branch renders raw serialized JSON into the UI. If the `/api/assistant/chat` route ever returns a success response with a differently-named field (e.g., `json.content`, `json.text`), users will see `{"content":"Hello!","metadata":{...}}` rendered as plain text. This should be an error state ("Unexpected response format") rather than a raw dump.

## Error state rendering contradicts its own message

Across all admin pages, the pattern is:

```tsx
if (error) {
  return (
    <div>
      <p>Error: {error}</p>
      <p>Showing cached data</p>
    </div>
  );
}
```

The component says "Showing cached data" but returns early, never rendering the actual data grid/table that holds the fallback state. The state variables (`usersData`, `appsData`, etc.) are initialized with defaults and would render correctly if the component didn't bail out on the error path. Either remove the early return and show data with an error banner, or change the copy to "Could not refresh data" without claiming cached data is visible.

## Apps page: positional fallback for lastDeploy

The apps page maps API responses to the UI model using array index to pull `lastDeploy`:

```typescript
json.apps.map((app, index) => ({
  // ...
  lastDeploy: defaultAppsData[index]?.lastDeploy ?? 'N/A',
}));
```

This assumes the API returns apps in the same order and count as `defaultAppsData`. If the health endpoint adds a new app, removes one, or sorts differently, `lastDeploy` values will be silently misattributed. A name-based lookup (`defaultAppsData.find(d => d.name === app.name)?.lastDeploy`) would be more robust.
