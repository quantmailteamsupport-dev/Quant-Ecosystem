# Agent Swarm Runtime and Dock UI

A full AI agent orchestration framework (`@quant/agent-runtime`) plus companion React UI (`shared-ui/src/agent/`). The runtime introduces a supervisor/worker model: an Orchestrator decomposes natural-language tasks via an AI adapter, dispatches subtasks to WorkerAgent implementations gated by a 5-level permission model, and wraps everything in safety scaffolding (kill switch, approval queue, spending caps, undo engine, conflict locks, sandbox mode). Twelve pilot agents cover email, code review, scheduling, shopping, finance, social, content, travel, research, health, meetings, and learning. A marketplace layer allows publishing/installing community agents. The UI surfaces this system through a responsive dock, approval dialog, audit timeline, agent creator, and status widget.

**Watch for:**
- Kill switch singleton has no timeout enforcement and never cleans up dead agents (confirmed).
- Sandbox mode is completely disconnected from agent execution: no agent actually routes through `AgentSandbox` (confirmed).
- Orchestrator dispatches tasks using trust-score-derived permissions rather than the worker's `defaultPermission`, meaning a brand-new agent (trust 20 = OBSERVE) can never be dispatched for even SUGGEST-level work until manually boosted (confirmed).
- Conflict resolver and approval queue exist as standalone classes but are never integrated into the Orchestrator or WorkerAgent lifecycle (confirmed).

## High-level view

The Orchestrator sits above a pool of WorkerAgents, each governed by a state machine and trust score. Task decomposition relies on a pluggable AI adapter, validation uses Zod schemas throughout, and the worker lifecycle handles kill-switch registration. The safety components remain library primitives rather than wired-in constraints.

The permission model defines five tiers mapping trust score bands (0-20 OBSERVE, 21-40 SUGGEST, etc.) but the Orchestrator uses the agent's current trust-derived permission when selecting workers. Since all agents start at trust 20 (OBSERVE), no fresh agent can be dispatched for any subtask requiring SUGGEST or above. The 30-day graduation curve means agents are functionally inert until externally promoted.

The kill switch calls halt callbacks in parallel but stores no mechanism to time out a stalled callback, nor does it clear its registry after activation. The <500ms guarantee claimed in the spec is not architecturally enforced.

The sandbox, approval queue, and conflict resolver are independently testable units that nothing else calls. If an agent is installed from the marketplace sandboxed, nothing prevents it from executing real side effects since the WorkerAgent base class has no sandbox gate.

Permission levels in `AgentCreator` ("read-only"/"standard"/"elevated"/"admin") do not correspond to the runtime's `PermissionLevel` enum, creating an unmapped integration seam.

1. **Sandbox not integrated into execution path** — `AgentSandbox.execute()` exists but no WorkerAgent or Orchestrator routes actions through it. Marketplace-installed agents marked `sandboxed: true` still execute via the normal `run()` path. Wire a sandbox gate into `WorkerAgent.run()` or into the Orchestrator dispatch.

3. **Orchestrator worker selection uses trust score, not defaultPermission** — `findSuitableWorker` checks `worker.trustScore.getPermissionLevel()`, which starts at OBSERVE for all new agents. Either seed agents at a trust level matching their `defaultPermission` or use `defaultPermission` as a floor for dispatch eligibility.
4. **Approval queue not wired into high-risk execution** — `ApprovalQueue` is tested in isolation but the Orchestrator never submits ACT_HIGH subtasks for approval. Add an approval gate when `requiredPermission >= ACT_HIGH`.
5. **Conflict resolver not used during parallel dispatch** — The Orchestrator runs subtask groups via `Promise.all` but never acquires resource locks. Two subtasks touching the same resource will race without conflict detection.
6. **UI permission levels don't map to runtime enum** — `AgentCreator` uses "read-only"/"standard"/"elevated"/"admin" while the runtime uses OBSERVE/SUGGEST/ACT_LOW/ACT_HIGH/FULL_AUTO. Add a mapping constant or align the strings.
7. **KillSwitch singleton leaks across tests** — `resetInstance()` is called in `beforeEach`, but if a test registers agents and throws before cleanup, subsequent tests inherit stale state. The pattern works today but is fragile for concurrent or randomized test runs.

## Orchestrator dispatch logic vs. trust-score bootstrapping

The Orchestrator's `findSuitableWorker` iterates registered workers and selects the first whose `trustScore.getPermissionLevel()` meets the subtask's `requiredPermission`. Because `TrustScore` initializes at 20 (OBSERVE), every freshly-constructed pilot agent can only satisfy OBSERVE-level subtasks. The `defaultPermission` field on each WorkerAgent (e.g. `ACT_LOW` for EmailPilot, `ACT_HIGH` for TravelPilot) is never consulted during dispatch and serves only as metadata.

```typescript
// orchestrator.ts
private findSuitableWorker(requiredPermission: PermissionLevel): WorkerAgent | undefined {
  for (const worker of this.workers.values()) {
    if (canAct(worker.trustScore.getPermissionLevel(), requiredPermission)) {
      return worker;
    }
  }
  return undefined;
}
```

This means `executeTask` will throw "No suitable worker found" for any decomposed subtask requiring SUGGEST or above unless an external caller has already driven the trust score up manually. Two options: initialize `TrustScore` at the level matching `defaultPermission` to represent built-in pilot agents as pre-trusted, or add a `defaultPermission` floor check in `findSuitableWorker`.

## Kill switch: missing timeout and post-activation state

```typescript
async activate(): Promise<void> {
  this.active = true;
  const callbacks = Array.from(this.agents.values());
  await Promise.all(callbacks.map((fn) => fn()));
}
```

If any registered halt callback hangs (e.g. waiting on I/O), `activate()` never resolves, violating the <500ms halt spec. A `Promise.allSettled` with a 500ms race would make the guarantee enforceable. After activation the agents map is not cleared, so a subsequent `activate()` calls the same callbacks again on potentially-dead agents.

The `reset()` method clears the `active` flag but not the agents map, and `deregister()` is only called by `WorkerAgent.stop()`. An agent whose halt callback throws will remain registered forever.

## Sandbox isolation is structural but not enforced

`AgentSandbox` logs actions with a `wouldHaveExecuted: true` flag when in sandbox mode, but no WorkerAgent references it. The `AgentInstaller` marks installed agents as `sandboxed: true`, but this flag is never checked during execution. `promoteFromSandbox()` flips the flag but has no connection to the `AgentSandbox` class itself.

The sandbox layer needs either a decorator pattern wrapping WorkerAgent's `run()` method or a check in the Orchestrator's dispatch path that routes sandboxed agents through `AgentSandbox.execute()` instead of real execution.

## Safety components as unused library code

Neither `ApprovalQueue` nor `ConflictResolver` is referenced by `WorkerAgent`, `Orchestrator`, or any pilot agent:

- An ACT_HIGH subtask dispatched to TravelPilot (if trust allowed it) would execute immediately without human approval.
- Two agents modifying the same calendar slot via `Promise.all` in `groupByDependencyLevel` would race without lock acquisition.

These are infrastructure pieces waiting for wiring. The framework currently provides no enforcement of the safety model it defines.

## UI permission and state model mismatch

`AgentCreator` surfaces four levels: "read-only", "standard", "elevated", "admin". The runtime defines five: OBSERVE, SUGGEST, ACT_LOW, ACT_HIGH, FULL_AUTO. If the creator submits a config with `permissionLevel: "elevated"`, nothing in the codebase translates that to a runtime `PermissionLevel`. The `AgentDock` and `AgentCard` components use status fields ("running"/"paused"/"stopped"/"completed"/"error") that don't map to the runtime's `AgentState` enum ("idle"/"planning"/"executing"/"waiting_approval"/"done"/"failed"). Explicit mapping utilities will be needed when the UI connects to the runtime.

## Test coverage gaps

Not tested: integration between components (Orchestrator + ApprovalQueue + ConflictResolver), kill switch timing under load, marketplace install-to-execution flow, and the UI-to-runtime contract.
