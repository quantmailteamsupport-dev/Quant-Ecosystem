# Agent Swarm Runtime and Dock UI (v2)

The seven issues from v1 have all been addressed: sandbox gates execution in `WorkerAgent.run()`, kill switch uses `Promise.race` with 500ms deadline and clears agents post-activation, the orchestrator uses `defaultPermission` as a floor via `getEffectivePermission()`, approval queue is wired into `dispatchSubtask()` for ACT_HIGH+ work, conflict resolver acquires locks during parallel group dispatch, `AgentCreator` now uses the runtime's `PermissionLevel` string values, and `reset()` clears registered agents.

**Watch for:**
- Approval queue submits requests but never awaits approval before executing the subtask — the gate is fire-and-forget, not blocking (confirmed).
- Resource locks use `subtask.id` as the resource identifier rather than an actual shared resource, so two subtasks touching the same calendar slot still race without conflict detection (confirmed).

## High-level view

The kill switch now enforces the <500ms guarantee by racing halt callbacks against a deadline and clearing the registry afterward.

The orchestrator's `getEffectivePermission()` takes the higher of trust-derived and `defaultPermission`, solving the bootstrapping problem. The `permissionLevelToScore()` helper initializes trust at the boundary value for each level (21 for SUGGEST, 41 for ACT_LOW, etc.), so the floor logic is effectively a no-op until trust degrades below `defaultPermission`.

The approval queue is now called during dispatch, but the call is non-blocking: the request is submitted and execution proceeds immediately. High-risk subtasks still execute unattended — the queue serves as a post-hoc record rather than a gate.

Resource locking uses `subtask.id` as the key. Since subtask IDs are unique by construction, no two subtasks ever contend for the same lock. The conflict resolver is mechanically integrated but cannot detect real resource conflicts.

The lock needs to key on the actual resource being modified.

## Approval queue: record vs. gate

The approval queue is called in `dispatchSubtask` but execution is never conditioned on approval:

```typescript
if (
  subtask.requiredPermission === PermissionLevel.ACT_HIGH ||
  subtask.requiredPermission === PermissionLevel.FULL_AUTO
) {
  const approvalId = `approval-${subtask.id}`;
  this.approvalQueue.submit({
    id: approvalId,
    agentId: worker.id,
    action: subtask.description,
    riskLevel: subtask.requiredPermission === PermissionLevel.FULL_AUTO ? 'critical' : 'high',
  });
}

// ... immediately proceeds to:
worker.stateMachine.reset();
worker.start();
await worker.run(agentTask);
```

There is no `await` on an approval resolution between the submit and the execution. The `ApprovalQueue` class has no async `waitForApproval()` method. The fix would add a blocking wait:

```typescript
const approved = await this.approvalQueue.awaitResolution(approvalId, timeoutMs);
if (!approved) throw new Error(`Subtask ${subtask.id} rejected or timed out`);
```

The test asserts that an approval entry exists after execution, which passes because submit is called — but it doesn't test that execution was blocked pending approval, because it wasn't.

## Conflict resolver: structural vs. semantic locking

The orchestrator acquires a lock per subtask before dispatching a group:

```typescript
for (const subtask of group) {
  const resourceId = subtask.id;  // <-- unique per subtask
  const worker = this.findSuitableWorker(subtask.requiredPermission);
  if (worker) {
    this.conflictResolver.acquireLock(resourceId, worker.id);
  }
}
```

Since `subtask.id` is generated uniquely, `acquireLock` always succeeds and the contention logic is never triggered. For meaningful conflict detection, the lock key would need to come from a resource descriptor in the subtask (e.g. `targetResource: "calendar:2024-01-15"`), requiring an extension to the `SubTask` schema.
