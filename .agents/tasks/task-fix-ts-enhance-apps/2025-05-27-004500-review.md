# TypeScript fixes and micro-feature services across 9 apps

This change resolves ~200 TypeScript compilation errors across three problem areas (broken TSX files, shared-ui strict null checks, backend Prisma type gaps) and then adds 46 new in-memory service classes with 4 React components spread across quantchat, quantmail, quantsync, quantube, quantneon, quantedits, quantmax, quantai, and quantads. Five cross-app integration services land in shared packages (notifications, search, media, common). The approach is consistent throughout: typed in-memory Maps as data stores, stateless service classes with no persistence, comprehensive unit tests using vitest.

Watch for:

- **No persistence layer** (confirmed) — every service uses `Map<string, T>` as its data store. Server restart or page reload destroys all state.
- **Prisma stub declarations shadow real types** (confirmed) — the two new `prisma-stub.d.ts` files declare `@prisma/client` as an ambient module; when a real Prisma client is generated, these stubs will conflict or silently override generated types.
- **Unbounded in-memory growth** (confirmed) — none of the 46 new services implement eviction, size limits, or TTL on their internal Maps.

## High-level view

The TypeScript error fixes fall into three tiers. The shared-ui/advanced fixes are mechanical strict-null appeasement (non-null assertions, nullish coalescing, unused-var prefixing). The TSX formatting fixes expand compressed single-line JSX into multi-line, replacing raw Unicode escape sequences in JSX positions with expression containers. The backend fixes introduce hand-rolled Prisma type stubs and scattered `as` casts to satisfy the compiler without a generated Prisma client present — these two stub files define conflicting `PrismaClient` interfaces and will break once a real schema is generated.

The 46 micro-feature services share an identical architecture: a class with a private `Map`, CRUD methods, and no I/O. Services that model time-based behavior (scheduler, undo-send, snooze) have no timer or event-loop integration — they're pure state containers that never fire their scheduled actions.

Test coverage is broad (682 new tests) but uniformly tests the in-memory happy path. No component tests for the 4 new React components, no integration tests between services, and no tests against the backend Prisma stubs with actual database queries.

## Prisma stub approach and its collision risk

Both `quantchat/backend/types/prisma-stub.d.ts` and `quantmail/backend/types/prisma-stub.d.ts` use `declare module '@prisma/client'` to provide interfaces and a `PrismaClient` interface with a generic `ModelDelegate<T>`. The quantchat stub defines `PrismaClient extends TransactionClient` with `conversation`, `conversationMember`, `message` delegates. The quantmail stub defines a separate `PrismaClient` with `email`, `emailThread`, `repository`, `pullRequest`, etc. These two declarations conflict: a consumer importing from `@prisma/client` in a workspace where both stubs are visible will see a merged, incoherent `PrismaClient` type.

The backend service changes to `conversation.service.ts` annotate the transaction callback as `import('@prisma/client').TransactionClient` — a type that only exists in the stub. The `thread.service.ts` changes add explicit `(e: Email)` and `(e: { threadId: string | null })` annotations where the compiler couldn't infer from the stub's generic `findMany`. These are brittle: they'll break silently if the Prisma schema adds fields or renames types.

## Time-dependent services with no timers

`MessageSchedulerService` accepts `sendAt` timestamps but has no scheduler loop to fire messages. `UndoSendService` implements a delay window but no cancellation mechanism once the period expires. `EmailSnoozeService` stores snooze-until timestamps but never resurfaces snoozed items. These services model temporal behavior as pure data containers — they fail open (action never executes) rather than fail closed.

## Orphan state creation in ReadReceiptsService

`getOrCreateState` is called by `markAsSent`, `markAsDelivered`, `markAsRead`, and `markAsFailed`. For unregistered messages, it creates:

```ts
state = { status: 'sending', receipts: new Map(), conversationId: '' };
```

The empty `conversationId` means this message will never appear in `getConversationReadState` results because `conversationMessages` was never updated. A consumer calling `markAsDelivered('msg-x', 'user-y')` before `registerMessage` gets a functioning receipt that's invisible to conversation-level queries.

## AnalyticsService mutation asymmetry

```ts
private getOrCreateMetrics(postId: string): PostMetrics {
  const existing = this.postMetrics.get(postId);
  if (existing) {
    return { ...existing }; // COPY on subsequent calls
  }
  this.postMetrics.set(postId, fresh);
  return fresh; // REFERENCE on first call
}
```

`trackImpression` and `trackClick` correctly call `set` after mutation, so the code works today. But the inconsistent return semantics are a maintenance trap — any future method that mutates the returned value without calling `set` will silently lose changes.

## Non-null assertions in shared-ui

- `easingFunctions.easeOutBounce!(1 - t)` — safe because `easeOutBounce` is in the same literal, but throws if the engine is extended with dynamic easing name lookup.
- `animations[currentIndex]!` in `sequence()` — safe given the preceding bounds check.
- `points[0]!`, `points[i]!` throughout charts-engine — safe given `if (points.length < 2) return ''` guards.

These are maintenance risks rather than current bugs: future refactors that separate the guard from the access will introduce crashes.

## Test coverage gaps

Not tested:

- Race conditions (multiple async callers modifying the same Map)
- Memory pressure (large Map sizes)
- Integration between services (reactions + notifications, search + activity feed)
- React component rendering (no component tests for LinkPreview, MessageReactions, ReadReceipts, VoiceMessage)
- Backend Prisma stubs against actual database queries
- Time-based service behavior (scheduler firing, undo-send expiry, snooze resurfacing)
