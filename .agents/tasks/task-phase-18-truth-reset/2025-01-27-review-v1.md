# Phase 18 Truth Reset: audit, stub elimination, and grep-gate renames

Phase 18 audits the entire codebase for honesty (44 files classified as REAL/NAIVE/FAKE/STUB), deletes 7 empty packages (~37k lines), replaces the no-op CSAM matcher with a fail-closed guard, renames "placeholder" terminology to pass a grep gate, tags 27 mock-data pages with FIXME comments, and adds ARCHITECTURE.md documenting the monolith-per-app decision.

The CSAMGuard is the most security-relevant change: it flips the CSAM matcher from fail-open (always returns not-matched) to fail-closed (throws unless `UGC_MEDIA_ENABLED=true`). The deleted packages had no consumers and no package.json, so removal is safe. The variable renames are mechanical and consistent within their scope.

**Watch for:** CSAMGuard returns `{matched: false}` when enabled, meaning it still performs no actual detection тАФ the guard only blocks the unconfigured path (confirmed). One "Placeholder" comment survives inside `sync-protocol.ts` line 243 that the rename missed (confirmed). The moderation-worker's image handler doesn't wire CSAMGuard, so CSAM checking is not active in the actual pipeline (confirmed).

## High-level view

The CSAMGuard change is a meaningful safety improvement: media uploads will now throw at ingestion time unless the env var is explicitly set, preventing accidental deployment without CSAM detection. The guard itself is not a detector тАФ when enabled it returns `{matched: false}` тАФ so the system still has no real CSAM matching. The naming is misleading since `UGC_MEDIA_ENABLED=true` implies detection is active when it merely disables the kill-switch. Critically, the guard is exported but never wired into the moderation-worker's `ImageModerationHandler`, so its protection only works if a consumer explicitly instantiates it upstream of media acceptance.

Deleting 7 packages removes ~37k lines of dead code with zero remaining imports anywhere in the monorepo. The variable renames are consistent within each package; one inline comment in `sync-protocol.ts:243` was missed by the rename pass.

## CSAMGuard: fail-closed but detection-free when enabled

The concern is the enabled path:

```typescript
async checkHash(_hash: string): Promise<{ matched: boolean; reportId?: string }> {
  if (!this.enabled) {
    throw new Error('CSAM matching not configured...');
  }
  return { matched: false };
}
```

When `UGC_MEDIA_ENABLED=true`, the guard becomes a passthrough that claims nothing matches. There's no provider injection point, no constructor parameter for a delegate, no configuration for which provider to use. A developer could flip the env var to unblock media uploads and unknowingly run without any CSAM detection.

The `ImageModerator` class in `packages/moderation` accepts an optional `csamMatcher?: CSAMMatcherInterface` and calls `checkHash` during `moderateWithHash()`. But the moderation-worker's `ImageModerationHandler` doesn't instantiate or pass a `CSAMGuard` instance тАФ it has its own separate `knownBadHashes: Set<string>` mechanism. The guard is exported from `@quant/moderation` but not wired into the actual moderation pipeline that processes BullMQ jobs. Its protection exists only if a consumer explicitly instantiates it and calls `checkHash` before accepting uploads.

Two gaps compound:

1. Setting `UGC_MEDIA_ENABLED=true` disables the kill-switch without requiring a real provider
2. The pipeline that actually moderates images doesn't use the guard at all
