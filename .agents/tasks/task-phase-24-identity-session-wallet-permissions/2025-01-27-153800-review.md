# Unified identity, wallet, and permission model across 13 apps

Phase 24 introduces a platform-wide ABAC+RBAC permission engine, extends auth with cross-app SSO and federated "Sign in with Quant" for third-party developers, adds a unified wallet with India-market payment gateways (Razorpay, UPI), and layers on a single Quant Pro subscription with Apple/Google IAP validation. The approach is services-in-memory with Zod validation at the boundary, consistent with the rest of the monorepo. All 13 apps are now registered as first-party OAuth clients, and the SSO middleware enforces scope-per-app restrictions for cross-app token reuse.

**Watch for:** The Sign-in-with-Quant SDK never validates `clientSecret` (confirmed), meaning any caller with a valid auth code and PKCE verifier can exchange tokens regardless of the secret they present. ABAC policies are silently bypassed when `context` is omitted from `can()` calls (confirmed). Account purge only clears local lifecycle state without coordinating with wallet, sessions, or permissions (confirmed).

## High-level view

The permission engine is a synchronous `can(subject, action, resource, context?)` function that first checks role-to-action contracts per resource type (16 resource types defined), then evaluates ABAC conditions on any matching policies. Contracts deny by default, which is the right posture. The risk is that omitting the optional `context` parameter entirely skips all ABAC restrictions, making policy-defined time/location/device/trust-score gates unenforceable if callers are not disciplined about passing context.

Auth gains five new services: FederatedIdentityService and SignInWithQuantSDK together power the external developer OAuth2 flow with PKCE; SSOMiddleware provides cross-app token validation and logout propagation; TravelModeService gates access by device and region; AccountLifecycleService manages deletion with a 14-day grace period. The SSO middleware correctly propagates logout by revoking all user sessions and tokens, but the SDK's `handleCallback` ignores the `clientSecret` parameter entirely, which breaks the confidential-client security model for server-side integrations.

The payments package adds RazorpayGateway and UPIPaymentService as India-market primitives, UnifiedWalletService as the single cross-app wallet, and QuantProService managing subscription lifecycle with IAP receipt validation. IAP validation is simulated (receipt length > 10 = valid), which is fine for the in-memory phase but has a structural concern: the signature verification in RazorpayGateway is a public method that reveals the expected signature format, making it trivial to forge in the current shape.

## ABAC context bypass in permission engine

The `PermissionEngine.can()` function treats `context` as optional. When present, it evaluates ABAC conditions on applicable policies. When absent, the entire ABAC evaluation block is skipped:

```typescript
if (applicablePolicies.length > 0 && context) {
  // conditions evaluated only here
}
// If context is undefined, this block is never entered -> returns true
```

A policy that restricts `CREATOR` write access on `video` resources to `trustScore >= 80` will be silently bypassed by any code path that calls `can(subject, 'write', 'video')` without passing context. The engine cannot distinguish "context not available" from "caller forgot to pass it." This is a fail-open design for what is supposed to be a restrictive layer. (confirmed)

## clientSecret validation absent in Sign-in-with-Quant SDK

Both `handleCallback` and `refreshToken` accept a `clientSecret` parameter but prefix it with underscore and never use it:

```typescript
async handleCallback(
  code: string,
  clientId: string,
  _clientSecret: string,  // never validated
  redirectUri: string,
  codeVerifier: string,
): Promise<SDKTokenResponse | null> {
```

For public clients (SPAs, mobile apps) this is acceptable because PKCE alone secures the flow. But the SDK is documented as powering "Sign in with Quant" for external developers, which includes server-side integrations (confidential clients). Those clients expect their secret to be verified. Currently, any party that intercepts an auth code and knows the PKCE verifier can exchange it without knowing the client secret. (confirmed)

## Account purge scope is incomplete

`AccountLifecycleService.purgeAccount` does this:

```typescript
purgeAccount(userId: string): boolean {
  const request = this.deletionRequests.get(userId);
  if (request) { request.status = 'purged'; }
  this.vacationResponders.delete(userId);
  this.purgedAccounts.add(userId);
  return true;
}
```

It does not call `SessionService.revokeAllSessions`, `TokenService.revokeAllForUser`, `WalletService` (balance remains), `ConsentManager` (consents remain active), or `PermissionEngine` (role assignments persist). The integration test (`account-deletion-gate.test.ts`) tests purge by checking `isAccountPurged()` returns true, but does not verify cross-service cleanup. A purged account can still hold wallet funds and valid auth tokens. (confirmed)

## Razorpay signature is publicly forgeable

`RazorpayGateway.generateSignature` is a public method:

```typescript
generateSignature(orderId: string, paymentId: string): string {
  return `sig_${orderId}_${paymentId}`;
}
```

The tests use this same method to compute the expected signature for `verifyPayment`. In production, Razorpay uses HMAC-SHA256 with a shared secret. The current simulation makes `verifyPayment` a no-op security check since any caller can compute a valid signature from public IDs. Even for in-memory simulation, the method should be private to avoid test code that normalizes calling it externally. (confirmed)

## Wallet summary performance and atomicity

`getWalletSummary` calls `getTransactionHistory(userId, { limit: 1000 })` to compute `totalAdded` and `totalSpent` by iterating all returned transactions. The limit of 1000 means the totals are wrong for users with more than 1000 transactions (silent truncation). It also means every summary call does a full scan. The fix is straightforward: maintain running `totalAdded`/`totalSpent` counters on the wallet itself, updated on each credit/debit. (confirmed)

## SSO scope validation is permissive by default

`SSOMiddleware.validateCrossAppToken` checks that "at least one token scope is allowed in the target app." This means a token with scopes `['profile:read', 'agent:execute']` is valid for `quantchat` because `profile:read` is in quantchat's allowed list, even though `agent:execute` is not. The token retains all its original scopes once validated. The middleware does not downscope the token for the target app context, so downstream code must independently filter scopes. (likely)

## Third-party client scope registration unrestricted

`QuantMailProvider.registerThirdPartyClient` accepts any `PermissionScope[]`:

```typescript
registerThirdPartyClient(
  name: string,
  redirectUris: string[],
  allowedScopes: PermissionScope[],
  createdBy: string,
): OAuthClient { ... }
```

There is no validation that the requested scopes are from a safe subset for third-party apps. An external developer could register a client with `agent:execute`, `workspace:manage`, or `email:send` scopes. First-party apps have these scopes because they're trusted; third-party apps should be restricted to a curated subset (e.g., `profile:read`, `email:read`). (confirmed)
