# Universal AI Memory & Context Graph (v2)

Phase 26 adds an 8-category memory schema with anti-creep enforcement, a candidate approval pipeline, encrypted export, and typed edges on the context graph. This v2 review confirms that 6 of the 7 issues from the first pass were addressed adequately. One fix (export double-encoding) resolved the server-side problem but introduced a new client-server contract mismatch.

**Watch for:** The export endpoint now sends raw content without the `{ success, data }` envelope, but the UI's `apiFetch` helper still extracts `json.data` — the export download will silently fail (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The ownership checks on PUT, DELETE, approve, and reject routes are correctly implemented: each fetches the memory via `service.getMemory()`, verifies `existing.userId !== userId`, and returns 403 before proceeding. The pattern is consistent across all four mutating routes.

The `createCandidate` fix is clean: `writeSignal` is now `'pending-review'` (a third value added to the union type), and the `Omit<..., 'writeSignal' | 'status'>` signature prevents callers from overriding it.

The export route bypasses the response envelope and sends raw content with correct `Content-Type` headers. This fixes the double-encoding at the protocol level, but the UI still calls this endpoint via `apiFetch` which does `res.json()` then `.data` extraction — on a raw JSON response with no `.data` property, this returns `undefined`, so the download never fires.

## Export contract mismatch between route and UI

The v1 review flagged that `GET /export` wrapped a JSON string in `{ success: true, data: "..." }`, causing double-encoding in the download. The fix correctly sends raw content:

```typescript
// routes/memory.ts - GET /export
if (format === 'json') {
  return reply.header('Content-Type', 'application/json').send(exported);
}
```

But the UI's generic `apiFetch` helper still expects the envelope:

```typescript
async function apiFetch<T>(path: string, options?: RequestInit): Promise<T | null> {
  const res = await fetch(...);
  const json = await res.json();
  return json.data as T;  // <-- export response has no .data property
}
```

In `handleExport`:

```typescript
const data = await apiFetch<string>('/export?format=json');
if (data) {  // data is undefined → condition is falsy → download never happens
  const blob = new Blob([data], { type: 'application/json' });
  ...
}
```

The fix: `handleExport` should call `fetch` directly and use `res.text()` to get the raw export content, bypassing `apiFetch`.

## Ownership check implementation

The fix follows fetch-then-verify in all four mutating routes:

```typescript
// PUT /:id
const existing = service.getMemory(request.params.id);
if (!existing) throw createAppError('Memory not found', 404, 'NOT_FOUND');
if (existing.userId !== userId) throw createAppError('Forbidden', 403, 'FORBIDDEN');
```

The `getMemory` method delegates to `store.get()` which returns any entry regardless of status, so the ownership check works for both active and pending entries. The 404-before-403 ordering avoids leaking which IDs exist to unauthorized users. No test currently exercises this path at the route level.
