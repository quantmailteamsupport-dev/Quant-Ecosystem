# Universal AI Memory & Context Graph

Phase 26 introduces an 8-category memory schema with anti-creep enforcement, a candidate/digest approval pipeline, encrypted export, and typed edges on the context graph. The approach leans on a single writeSignal gate to prevent autonomous memory writes, uses AES-GCM for encrypted export with IV prepended to ciphertext, and adds BFS-based path-finding to the context graph. The implementation spans the `ai-memory` package (types, store, export), the `identity-permissions` context graph, a Fastify API layer in QuantAI, and a React inspector UI.

**Watch for:** Missing ownership checks in mutating API routes allow any authenticated user to modify/delete another user's memories (confirmed). The `createCandidate` method sets `writeSignal: 'explicit'` on pending entries, which misrepresents the actual signal source (confirmed). The UI lock/unlock feature is purely client-side state with no persistence or enforcement (confirmed). The Memory Inspector UI tests provide no meaningful coverage of component behavior (confirmed).

## High-level view

The memory schema expansion is clean: 8 categories validated by a zod enum, `accessScopes` as a string array per memory, `writeSignal` as the anti-creep gating field, and `status` distinguishing active from pending. The core store correctly refuses non-explicit writes and hides pending/expired entries from `getUserMemories`. The candidate lifecycle (create -> pending -> approve -> active) works as intended, with approved entries getting `writeSignal: 'digest-approved'`.

The API layer in QuantAI exposes full CRUD, candidate management, export/import, and a disclosure endpoint. Every route extracts `userId` from `request.auth` but then delegates to service methods that operate on memory IDs without verifying the memory belongs to the requesting user. This is the most significant security gap in the change.

The context graph gains typed edges (a 5-value union replacing raw strings), BFS path-finding with configurable depth, edge-type filtering, and metadata-based node lookup. Edges are bidirectional by default (addEdge pushes to both nodes), which is correct for undirected relationships but semantically wrong for directed ones like "edited" or "mentioned-in".

The test suite includes 40 dedicated gate tests covering schema validation, export roundtrip, anti-creep red-teaming, candidate flow, TTL expiration, and graph traversal. The backend service tests are solid. The UI tests are essentially no-ops that verify the component is a function and produces a React element.

<summary>Issues (7)</summary>

1. **No ownership check on mutating routes** — PUT `/:id`, DELETE `/:id`, approve/reject candidates all accept any authenticated user. Add ownership verification by checking `memory.userId === request.auth.userId` before mutation.
2. **createCandidate sets writeSignal to 'explicit'** — Pending candidates created by pattern detection should not claim an explicit user signal. Introduce a third writeSignal value (e.g., `'pending'`) or leave it undefined until approval, then set `'digest-approved'` on approve.
3. **Lock/unlock is client-only** — `lockedIds` lives in React state and disappears on refresh. Either persist to the backend (add a `locked` field or `accessScopes: []` override) or document it as a UI-only soft-pause.
4. **Bidirectional edges for directed relationships** — `addEdge` creates symmetric links. "Alice edited Doc" also means "Doc edited Alice" in the current graph. Consider asymmetric edge insertion or a reverse relationship type.
5. **MemoryService instantiated per route registration** — Each `memoryRoutes` call creates a new `MemoryService` with an empty in-memory store. This is fine for tests but means server restarts lose all data. The singleton pattern would prevent accidental multi-instance bugs in production wiring.
6. **UI tests are no-ops** — `memory-page.test.tsx` never renders the component to DOM. The assertions check that `MemoryPage` is a function and `React.createElement` succeeds, which adds no behavioral confidence.
7. **Export endpoint wraps JSON string in another JSON envelope** — `GET /export` returns `{ success: true, data: "<stringified JSON>" }`. The client then tries to create a Blob from `data` directly, but `data` is a string not a JSON object. This will produce a double-encoded file on download.

## Missing authorization on memory mutations

The memory routes extract `userId` for creating and listing memories, but PUT, DELETE, approve, and reject operate solely on the memory ID parameter without verifying it belongs to the requesting user. In `memory.service.ts`, `updateMemory` and `deleteMemory` call directly into the store's `update`/`delete` methods which accept any ID:

```typescript
// routes/memory.ts - PUT /:id
const memory = service.updateMemory(request.params.id, parseResult.data);
// No check: memory.userId === userId
```

Any authenticated user who guesses or enumerates memory IDs can modify or delete another user's memories. The same gap applies to `/candidates/:id/approve` and `/candidates/:id/reject`. The fix is straightforward: fetch the memory first, compare `userId`, return 403 if mismatched.

## Candidate writeSignal misrepresentation

`createCandidate` in `memory-store.ts` hardcodes `writeSignal: 'explicit'`:

```typescript
const full: MemoryEntry = {
  ...entry,
  writeSignal: 'explicit', // <-- this is pattern-detected, not explicit
  status: 'pending',
};
```

The anti-creep contract says `writeSignal: 'explicit'` means the user explicitly asked the system to remember something. Candidates are pattern-detected inferences that haven't been user-confirmed yet. After approval, `approveCandidate` correctly sets `writeSignal: 'digest-approved'`, but during the pending phase the entry claims explicit provenance. If any code path reads pending entries and trusts `writeSignal`, it would incorrectly treat these as user-confirmed. The type system allows only `'explicit' | 'digest-approved'`; a third value like `'pending-review'` would be more honest.

## Export download path is double-encoded

The route handler returns the export string wrapped in the standard response envelope:

```typescript
// routes/memory.ts
const exported = service.exportMemories(userId, format);
return reply.send({ success: true, data: exported });
```

`exported` is already a JSON string. The response becomes `{"success":true,"data":"{\"version\":\"1.0.0\",...}"}`. The UI's `handleExport` then does:

```typescript
const data = await apiFetch<string>('/export?format=json');
if (data) {
  const blob = new Blob([data], { type: 'application/json' });
  // ...
}
```

`apiFetch` extracts `json.data`, which is the stringified JSON. The downloaded file will contain a raw JSON string (with escaped quotes if the browser serializes further), not a clean JSON object. The fix: either have the route return the raw content with `Content-Type: application/json` bypassing the envelope, or have the UI parse the string before blobbing it.

## Bidirectional edges on directed relationships

`addEdge` always pushes the relationship to both nodes:

```typescript
fromNode.relationships.push({ targetId: toId, relationship });
toNode.relationships.push({ targetId: fromId, relationship });
```

For symmetric relationships like `related-to` and `shared-with`, this makes sense. For `edited` and `mentioned-in`, the reverse doesn't carry the same semantic meaning. When calling `getEdgesByType(doc.id, 'edited')`, you'll get back the person who edited it, which is useful. But semantically, `getEdgesByType(person.id, 'edited')` should yield documents the person edited, while `getEdgesByType(doc.id, 'edited')` should yield nothing (the doc didn't edit anything). The current design conflates these. This isn't a bug in the current usage (tests work around it), but it will cause confusion when agents traverse the graph for grounding.

## Access scope enforcement is absent at query time

The `accessScopes` field is stored per memory and preserved through export/import, but no code path actually enforces it. `getUserMemories` returns all active, non-expired entries for a user regardless of which app is requesting. The `listMemories` service method has no `requestingApp` parameter. The API routes don't pass any app identity for scope filtering. The field is currently documentation-only metadata. This is acceptable for a first pass if scope enforcement is planned for a later phase, but the gate tests assert scopes are "stored correctly" without testing enforcement.

## UI test coverage gap

The five tests in `memory-page.test.tsx` verify:

1. `React.createElement(MemoryPage)` doesn't throw
2. The module exports a default
3. Expected category names is an array of 8
4. `typeof MemoryPage === 'function'`
5. `React.isValidElement(element)` is true

None of these render the component to DOM or test user interactions, API integration, state transitions, or accessibility. The 8-category assertion checks a hardcoded constant in the test file, not the component's actual output. These tests will pass even if the component is replaced with `() => <div/>`.
