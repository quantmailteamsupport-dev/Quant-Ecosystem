# Data pipeline, edge config, and defense-in-depth security (Wave 6)

Three new infrastructure packages — event streaming over Redis Streams, Next.js edge/CDN configuration, and advanced security primitives — plus comprehensive documentation and a root README overhaul. All implementations rely on in-memory Maps for state, making them self-contained library code that defers persistence decisions to the consuming application.

Watch for: All security and session state lives in process memory with no persistence or replication story (confirmed). The `EventStream.subscribe` loop swallows handler errors silently — a failing processor loses events without dead-lettering them (confirmed). The CSP policy ships `'unsafe-inline' 'unsafe-eval'` by default, which substantially weakens XSS protection (confirmed).

## High-level view

The data-pipeline package models Redis Streams semantics faithfully — publish with MAXLEN trimming, consumer groups, XREADGROUP blocking reads, batch acknowledgment. The DeadLetterQueue exists as a standalone in-memory structure, but nothing in the subscribe loop actually routes failed events into it — that wiring is left to the caller, and the `ProcessorConfig.maxRetries` field is never read.

The security-advanced package provides correct crypto throughout (`timingSafeEqual` at every comparison point, 12-byte IVs for GCM, HMAC-signed CSRF tokens). The fundamental limitation is that all state (sessions, API keys, IP blocks) lives in a process-local Map — a single restart loses all sessions and revocations, and the documentation doesn't flag that production deployments need an external store.

The edge-config package generates configuration objects for Next.js middleware — cache policies, security headers, bot detection rules, A/B test definitions. The default Permissions-Policy denies camera and microphone access, which will break QuantMeet and QuantMax without per-app overrides.

Documentation covers architecture, deployment (local/Docker/K8s), API reference, development guide, security posture, and operational runbook. Badge URLs reference `your-org/Quant-Ecosystem` which needs replacement before publishing.

5. **API key validation is O(n)** — `APIKeyManager.validate` iterates all stored keys for every validation call. Index by hash for O(1) lookup. [confirmed]

6. **No error events in EventStream** — No metrics hooks, no logging, no error events. Processor failures are invisible to the operator. [confirmed]

7. **Permissions-Policy blocks camera/microphone** — `camera=(), microphone=()` denies these APIs entirely. QuantMeet and QuantMax require them. The `getSecurityHeaders()` function hardcodes this without an override mechanism. [confirmed]

## Redis Streams consumer loop and the dead-letter gap

The subscribe loop's core problem:

```typescript
// event-stream.ts, subscribe method:
if (events.length > 0) {
  await handler(events);
  await this.acknowledge(config.stream, config.group, entryIds);
}
```

If `handler(events)` throws, entries remain unacknowledged in Redis (they'll appear in the PEL), but the loop continues reading new messages with no retry, no dead-letter routing, and no backoff. The `DeadLetterQueue` class exists in the same package with full replay/purge/stats capabilities, but nothing connects it to the subscribe loop. The `ProcessorConfig` type has a `maxRetries` field that's never read.

The three processors don't throw in normal operation — they aggregate or filter quietly — so this gap manifests only under unusual conditions (serialization errors, downstream failures). In a production event pipeline, silent event loss is the worst failure mode.

## API key validation performance and timing leakage

```typescript
validate(key: string): APIKeyRecord | null {
  const hashedKey = this.hashKey(key);
  for (const [, record] of this.keys) {
    // timingSafeEqual per record...
  }
}
```

O(n) in the number of keys. The `timingSafeEqual` per-key prevents timing attacks against individual keys, but the iteration count itself leaks whether a key exists early or late in the map. More practically, this won't scale beyond a few hundred keys. A hash-indexed lookup (keyed on `hashedKey`) would be O(1) and still timing-safe since the comparison happens after the lookup.

## Session fingerprint fragility

The fingerprint hashes `IP | User-Agent | Accept-Language`. Any of those three changing invalidates the session — IP rotation on mobile is the common case. `SessionConfig.fingerprintFields` exists at the type level suggesting configurability, but `generateFingerprint()` hardcodes all three fields.

## CSP and Permissions-Policy conflicts with app requirements

`getSecurityHeaders()` bundles `Permissions-Policy: camera=(), microphone=(), geolocation=()` into a fixed array. QuantMeet and QuantMax require camera/microphone. The `getCSPHeader(overrides)` function allows CSP customization, but `getSecurityHeaders()` provides no equivalent override for Permissions-Policy — callers would need to filter the returned array manually.

The CSP defaults (`'unsafe-inline' 'unsafe-eval'`) are appropriate for Next.js dev mode but ship as the production default with no environment-aware switching.

## Test coverage gaps

Not tested: the Fastify plugin integration (CSRF `createFastifyPlugin` returns a function but hook behavior isn't exercised), error propagation in the subscribe loop (because there isn't any to test), and behavior under concurrent access patterns (relevant for session concurrency enforcement where `enforceConcurrencyLimit` and `create` could race).
