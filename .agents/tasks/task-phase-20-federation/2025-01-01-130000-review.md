# Federation package (v3): signature verification wired through server layer

The v2 issue — `FederationServer` not passing request context to `InboxProcessor` for signature verification — is now resolved. The server constructor accepts an optional `InboxSignatureVerifier`, `handle()` accepts an optional `RequestMetadata` parameter, and `handlePostInbox` constructs the `requestContext` and passes it through. Verification fires only when both a verifier is configured and request context is present; without either, the server remains backwards compatible. Four new test cases exercise the reject, accept, no-verifier, and verifier-without-metadata paths.

**Watch for:** No blocking concerns remain.

**Verdict**: APPROVED

## High-level view

`FederationServer`'s constructor threads `InboxSignatureVerifier` through to `InboxProcessor`. The public `handle()` method accepts optional `RequestMetadata` (headers, method, url, body), and `handlePostInbox` forwards it as `requestContext`. The guard `if (this.signatureVerifier && requestContext)` means both must be present for verification to trigger — the right default for a protocol skeleton where not all transports provide raw HTTP context.

Signature verification runs before moderation checks, which is the correct ordering — reject unauthenticated requests before spending cycles on content policy.
