# Federation package v2: ActivityPub, Matrix bridge, and moderation

Six of the seven v1 issues have been properly resolved. The outbox now signs per-recipient, the delivery queue enforces backoff via `nextAttemptAfter` timestamps, moderation handles object-typed actors, the Matrix bridge returns structured results for unmapped events, and WebFinger supports an existence check hook. The inbox processor accepts a signature verifier callback — but `FederationServer.handlePostInbox` never passes request context through to it, so signature verification remains unwired at the HTTP layer.

**Watch for:** `FederationServer` still processes inbox POSTs without signature verification because it doesn't forward HTTP headers/method/url to the `InboxProcessor` (confirmed). The `InboxProcessor` machinery is correct; the server just doesn't use it.

**Verdict**: NEEDS_CHANGES

## High-level view

The inbox signature verification infrastructure is complete at the processor level — the `InboxSignatureVerifier` interface, the conditional check in `process()`, and tests covering both accept/reject paths all work correctly. The gap is purely at the integration layer: `FederationServer.handlePostInbox` calls `this.inboxProcessor.process(parsed.data, senderDomain)` with only two arguments, never constructing or forwarding the `requestContext` parameter that would trigger verification. Any client that can reach the inbox endpoint can still forge activities from any remote actor.

The remaining five fixes (outbox per-recipient signing, delivery queue backoff, moderation object actors, Matrix bridge result status, WebFinger existence check) are all properly implemented with corresponding tests that exercise the fixed behavior.

## Server-level signature verification gap

The `InboxProcessor` correctly gates on signature verification when both `signatureVerifier` and `requestContext` are provided:

```typescript
// inbox.ts
process(activity: Activity, senderDomain: string, requestContext?: { headers, method, url, body }) {
  if (this.signatureVerifier && requestContext) {
    const valid = this.signatureVerifier(requestContext.headers, ...);
    if (!valid) return { accepted: false, error: 'Invalid signature' };
  }
  // ...
}
```

But the server never exercises this path:

```typescript
// server.ts line ~95
const result = this.inboxProcessor.process(parsed.data, senderDomain);
// requestContext is never constructed or passed
```

The `FederationServer` constructor creates an `InboxProcessor` without a verifier (`new InboxProcessor(moderation)`), and `handle()` doesn't receive raw HTTP headers. The fix requires: (1) accepting an `InboxSignatureVerifier` in the `FederationServer` constructor and passing it to the `InboxProcessor`, and (2) extending `handle()` to accept request metadata (headers, body string) so it can forward them as `requestContext`.
