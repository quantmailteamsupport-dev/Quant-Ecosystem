# Monorepo bootstrap and Prisma persistence layer

This change converts a skeleton monorepo into a functioning pnpm + Turborepo workspace (Phase 0) and replaces the in-memory `BaseModel`/`Map<>` persistence with a 1028-line Prisma schema, 7 repository classes, a Docker Compose development stack, and a faker-based seed script (Phase 1). The deleted code is the ad-hoc migration system and in-memory models; the new code is a proper database layer with typed repositories, pagination, transactions, and soft-delete patterns.

**Watch for:** Tests only verify method existence, not behavior, so regressions in query logic would go undetected (confirmed). The `UserRepository` leaks `passwordHash` and `twoFactorSecret` in every read path (confirmed). The CI pipeline's `build` command for the database package is a no-op echo, so Prisma generation is never validated (confirmed). Docker Compose uses default credentials with no check preventing them from reaching shared environments (confirmed).

## High-level view

The lint-staged config only runs Prettier, not ESLint, so lint violations slip past the pre-commit hook and are only caught in CI.

The Prisma schema covers 36 models across 9 product domains with appropriate composite unique constraints, cascade deletes, and indexes on foreign keys and query-hot columns. Two structural gaps stand out: the `Match` and `Swipe` models have no FK relations to `User`, leaving referential integrity to the application layer; and `CommunityMember` has no `User` relation, making it impossible to eagerly load member profiles through Prisma.

The 7 repositories are thin wrappers around Prisma calls with consistent offset-based pagination. They all return full entity objects including sensitive fields. There is no cursor-based pagination despite the `cursor` field on `PaginationOptions`, and no authorization boundaries are enforced at the repository level.

The test suite (36 tests) is almost entirely "does this method exist" checks. Only two tests exercise actual behavior (findById delegation and findMany pagination math). No test calls `markAsRead`, `softDelete`, `incrementLikeCount`, or any mutation method with assertions on the arguments passed to the mock.

Default credentials are hardcoded in both `docker-compose.dev.yml` and `.env.example`, and SECURITY.md claims Trivy scanning in CI that doesn't actually exist in the workflow.

## Sensitive field leakage in UserRepository

Every method on `UserRepository` returns the full `User` entity, which includes `passwordHash`, `twoFactorSecret`, `lastLoginIp`, and `failedLoginAttempts`. Unless the repository applies a `select` clause or a post-query projection, every caller that touches user data must remember to strip sensitive fields manually. This is a fail-open pattern: any new consumer of `findById` silently gets secrets.

```typescript
// current
async findById(id: string): Promise<User | null> {
  return this.prisma.user.findUnique({ where: { id } });
}
```

A `select` whitelist or a separate `UserPublic` type returned from read methods would push the security boundary into the repository rather than relying on every caller to handle it. `findMany` (paginated user listing) is the clearest exposure point since it's the most likely method to feed API responses directly.

## Test coverage is structural, not behavioral

The test file creates a Prisma mock and instantiates each repository, then asserts that methods are defined and callable. Only `UserRepository` has two tests that actually invoke a method and check the result:

```typescript
it('should call prisma.user.findUnique for findById', async () => {
  (mockPrisma as any).user.findUnique.mockResolvedValue({ id: '1', email: 'test@test.com' });
  const result = await repo.findById('1');
  expect((mockPrisma as any).user.findUnique).toHaveBeenCalledWith({ where: { id: '1' } });
});
```

For the remaining 6 repositories, there is no test that verifies the `where` clause includes `deletedAt: null`, that pagination math is correct, or that `markAsRead` passes the right data. A bug like removing the `deletedAt` filter from `findByUser` would pass all tests.

The `withTx` transaction helper tests verify `$transaction` is called but don't check that the return value propagates back to the caller.

## Prisma schema: missing relations on dating/matching models

`Match` and `Swipe` reference `user1Id`/`user2Id` and `swiperId`/`targetId` as plain `String` fields with unique constraints and indexes, but no `@relation` directive connects them to `User`. This means:

- Prisma won't generate type-safe includes (`match.user1` won't exist).
- `onDelete: Cascade` is not enforced; deleting a user leaves orphan matches/swipes.
- The seeder doesn't create any matches or swipes, so this gap isn't visible in dev either.

Similarly, `CommunityMember` has a relation to `Community` but not to `User`, making `include: { user: true }` impossible.

## CI pipeline gaps

The CI workflow runs `pnpm turbo lint`, `typecheck`, `test`, and `build`. For the `@quant/database` package, the `build` script is:

```json
"build": "echo 'prisma generate requires DATABASE_URL - skipping in CI'"
```

`prisma generate` does NOT require `DATABASE_URL`; it only needs the schema file to produce the client typings. Running `prisma generate` in CI would catch schema syntax errors and ensure the generated client matches what repositories import. The current setup means a broken schema could merge without detection.

The `typecheck` step depends on `^typecheck` in turbo.json (packages first), but since the database package never generates its Prisma client in CI, `tsc --noEmit` against repository files that import `@prisma/client` will either fail (no generated client) or succeed vacuously (if `node_modules/.prisma/client` happens to exist from `pnpm install`). The behavior is environment-dependent.

## Unused cursor field and pagination boilerplate

`PaginationOptions` declares a `cursor?: string` field, but no repository reads it. Every paginated method uses offset-based (`skip`/`take`) pagination. For chat messages and notification feeds, cursor-based pagination is typically preferred because new inserts don't shift pages. The interface promises cursor support that doesn't exist.

The pagination logic is copy-pasted identically across 7 methods (same `totalPages` calculation, same response shape). A shared utility or a generic `paginate<T>()` helper on `BaseRepository` would reduce the surface area for bugs in the math.

## Docker Compose credentials and SECURITY.md discrepancy

`.env.example` ships `JWT_SECRET=development-only-change-in-production` as a placeholder that could be cargo-culted into staging. There's no validation (a startup check or CI lint) that confirms secrets have been rotated before deployment.

SECURITY.md claims "All dependencies are scanned with Trivy in CI," but the CI workflow has no Trivy step, no `npm audit`, and no dependency scanning of any kind. The documentation makes a security promise the infrastructure doesn't deliver.
