# QuantDrive: encrypted cloud storage with AI file services

QuantDrive adds a full-stack encrypted file storage application to the ecosystem: 11 backend services covering file CRUD with AES-256-GCM encryption, folder tree management, sharing via encrypted key exchange, versioning, soft-delete trash, tiered storage quotas, and five AI-powered services (organize, duplicate detection, content search, summarization, data extraction). The implementation follows the established monorepo patterns with Fastify routes, Prisma-shaped DI interfaces, and Zod validation. Tests cover all 11 services with 82 cases.

**Watch for:** Server-side encryption key storage defeats the purpose of E2E encryption (confirmed). Sharing grants access to encrypted content without verifying the recipient can actually decrypt it (confirmed). Folder move operation doesn't update child paths (confirmed). Duplicate detection hashes ciphertext instead of plaintext, so it can never find duplicates (confirmed). No quota enforcement at upload time in the route layer (confirmed).

## High-level view

The encryption design uses AES-256-GCM correctly at the primitive level, but the key is stored in the same database row as the ciphertext. This is server-side encryption at rest, not end-to-end encryption. A database breach exposes both the key material and the data. The `encryptedFileKey` field in the share service is the right shape for actual E2E (recipient-encrypted key exchange), but the file service itself never uses public-key cryptography.

The sharing model passes an `encryptedFileKey` string through to the database without validation or use. The share service doesn't verify the owner actually has access to the file/folder being shared, and the recipient accepting a share gains no actual access to the decrypted content through the current flow.

Folder operations maintain a materialized `path` string, but `moveFolder` only updates the moved folder's path without cascading to its descendants. Any nested folder tree becomes inconsistent after a move. The delete operation prevents deleting folders with child folders but doesn't check for files inside.

The duplicate detection service hashes the `encryptedContent` field directly from the database. Since each file gets a unique IV and key, identical plaintext files produce different ciphertexts and the duplicate detector will never find them. The content search service has no indexing hook in the upload/update flow, so search will always return empty results.

Storage quota exists as a service but is never called from the file upload route. A user can exceed their tier limit freely.

<summary>Issues (9)</summary>

1. **Encryption key stored alongside ciphertext** — The file record stores the AES key in the same row as the encrypted content. Move key management to a separate keystore or implement client-side key wrapping so a database dump doesn't expose plaintext.
2. **Version history loses encryption key** — `updateFile` saves old encrypted content to `fileVersion` but not the old encryption key. Versions become undecryptable. Store the key in the version record.
3. **Duplicate detection operates on ciphertext** — `AIDuplicateService.findDuplicates` hashes `encryptedContent` from the DB. Since each file has a unique IV, identical files produce different ciphertexts and won't be detected. Decrypt before hashing, or store a plaintext-derived hash at upload time.
4. **Folder move doesn't cascade path updates** — `moveFolder` updates only the target folder's `path` field. All descendant folders retain stale paths. Add a recursive path update or switch to a closure-table model.
5. **Quota not enforced at upload** — `StorageQuotaService.checkQuota` exists but is never called from the files route or `FileService.uploadFile`. Wire it into the upload path.
6. **Share doesn't verify file ownership** — `shareFile` and `shareFolder` never verify the caller actually owns the referenced file/folder. Any authenticated user who knows a file ID can create a share record for it.
7. **Folder delete ignores contained files** — `deleteFolder` checks for child folders but not for files in the folder. A folder with files but no subfolders can be deleted, orphaning those files.
8. **Folder rename not implemented** — The PUT route accepts a `name` field but the handler only processes `parentId` (move). If only `name` is provided, the route returns the folder unchanged.
9. **No search indexing hook** — `AISearchContentService.indexFile` is never called from the upload or update flow. Content search will always return empty results until indexing is wired in.

## Encryption key lifecycle and storage

`FileService.uploadFile` generates a random 32-byte AES key and 12-byte IV per file, encrypts the content, then stores the key as hex in the same database row as the ciphertext:

```typescript
const key = crypto.randomBytes(32);
const iv = crypto.randomBytes(12);
// ...encrypts...
data: {
  encryptedContent: encrypted.toString('base64'),
  encryptionIV: iv.toString('hex'),
  encryptionAuthTag: authTag.toString('hex'),
  encryptionKey: key.toString('hex'),  // key lives next to the data
}
```

This provides encryption at rest only against filesystem-level access (e.g., stolen disk), not against database-level breaches. For a product positioned as E2E encrypted, the key must never touch the server in plaintext. The typical pattern: client generates the file key, encrypts content client-side, wraps the file key with the user's public key, sends only wrapped keys to the server.

The `updateFile` method generates a fresh key for new content but stores the old encrypted content in `fileVersion` without its encryption key. The version record preserves IV and authTag but not the key, making historical versions permanently undecryptable.

## Duplicate detection on ciphertext

`AIDuplicateService.findDuplicates` fetches all user files and computes a perceptual hash on `Buffer.from(record.encryptedContent, 'base64')`. Since each upload generates a unique IV and key, two identical plaintext files produce completely different ciphertexts. The perceptual hash comparison will never detect them as duplicates.

Fix options: (a) compute and store a content-derived hash (e.g., SHA-256 of plaintext) at upload time and compare those — leaks content identity; (b) decrypt before hashing at detection time — expensive but correct.

## Folder tree inconsistency on move

`FolderService.moveFolder` sets the moved folder's `path` to `${parentRecord.path}/${folder.name}` but does not update any descendant folders. After moving `/Documents/Work` to `/Archive/Work`, a child at `/Documents/Work/Reports` retains its stale path.

`deleteFolder` checks `folder.count({ where: { parentId: folderId } })` to prevent deleting folders with subfolder children but does not check for files. A folder can be deleted while still containing files, orphaning them with a `folderId` pointing to a nonexistent record.

## Share service ownership gap

`ShareService.shareFile` receives `ownerUserId` from the route handler (set to the authenticated user's ID) but never verifies that user actually owns the file referenced by `fileId`. It only checks that the recipient user exists. Any authenticated user who knows a file ID can create a share record for a file they don't own. The service should look up the file/folder record and verify ownership before creating the share.

## Content search indexing model

`AISearchContentService.indexFile` stores plaintext content in a `fileIndex` table. The summarize and extract routes correctly decrypt via `downloadFile` first, but no route calls `indexFile`. Content search depends on files being indexed, but no indexing hook exists in the upload or update flow.

## Test coverage gaps

Not tested: route-level integration (auth extraction, prisma wiring, request validation), the file content indexing workflow, quota enforcement during upload, folder path cascading on move, version restore interaction with encryption keys, and the folder-delete-with-files scenario.
