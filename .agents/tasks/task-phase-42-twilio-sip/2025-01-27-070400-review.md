# Twilio SIP phone provider with call session management

This change adds a dual-mode Twilio phone provider (real REST API calls when credentials are present, mock mode when absent), a call session state machine, a TwiML XML builder, a fire-and-forget native phone fallback, and a CallAgent that mediates voice intents through a permission tier system. Together these give Quant Live the ability to place, hold, transfer, and end calls either through Twilio infrastructure or through a platform `tel:` URI.

**Watch for:** The CallAgent's permission tier check is logically inverted — the `phone` capability maps to tier 3 in `CAPABILITY_TIER_MAP`, meaning `getTier('phone')` always returns 3, and the agent requires tier 2 for allowed contacts and tier 3 for unknown numbers, so the tier gate never blocks anything (confirmed). The TwiML builder does not escape XML-special characters in user-supplied values like phone numbers or conference names, creating an injection surface (confirmed). Twilio credentials (`authToken`) are read from `process.env` and stored in-memory on the config object with no sanitization or redaction for logging (likely).

## High-level view

The `TwilioPhoneProvider` implements `PhoneCapability` and forks on construction: if account SID, auth token, and from-number are all present (via config or env vars), it enters live mode and proxies calls through a `TwilioClientLike` interface; otherwise it enters mock mode and generates synthetic SIDs. However, in production the constructor never instantiates the real Twilio SDK client — `this.twilioClient` stays null even in live mode unless the caller injects one, so live mode silently degrades to mock behavior.

The `CallSessionManager` is a local-only in-memory state machine with no un-subscribe mechanism for listeners, no persistence, and no eviction of terminal sessions — the Map grows without bound in long-running processes.

The `CallAgent` gates voice actions through `PermissionManager`, but the tier-based check compares a capability-level constant (always 3 for phone) against the required tier, so the gate never triggers a denial. The tiered-contact-list design is dead code as implemented.

The `TwiMLBuilder` constructs XML documents for Twilio webhook responses through string concatenation without escaping inputs, exposing an injection surface.

1. **Permission tier gate is a no-op** — `getTier('phone')` returns the static value 3 from `CAPABILITY_TIER_MAP`, so the check `currentTier < requiredTier` in `CallAgent` can never be true (required is 2 or 3, current is always 3). The intent was likely to check a per-user or per-session tier, not the capability's intrinsic tier.
2. **TwiML XML injection** — `TwiMLBuilder.dial`, `say`, `conference`, and `gather` interpolate raw strings into XML without escaping `<`, `>`, `&`, or `"`. A phone number or conference name containing these characters produces malformed or exploitable TwiML. Escape all interpolated values or use the `xmlbuilder` package (already a transitive dep of `twilio`).
3. **Live mode silently degrades to mock** — When credentials are supplied but no `TwilioClientLike` is injected (the normal production path), `liveMode` is true but `twilioClient` is null. The guard `if (this.liveMode && this.twilioClient && this.config)` falls through to mock mode. The constructor should instantiate the real Twilio SDK client when no override is provided.
4. **No listener cleanup on CallSessionManager** — `onStateChange` pushes callbacks with no removal mechanism. Long-running processes accumulate dead listeners.
5. **Credential handling in memory** — `authToken` is stored on the plain `config` object. If serialized for logging or error reporting, the token leaks. Consider a secret-fetching function or redaction strategy.

## Permission tier logic inversion

The `CallAgent.handleIntent` method for the `place` action decides the required tier: 2 for allowed contacts, 3 for unknown numbers. It then calls `this.permissionManager.getTier('phone')` to get the "current" tier:

```typescript
const requiredTier = this.allowedContacts.includes(intent.target) ? 2 : 3;
const currentTier = this.permissionManager.getTier('phone');
if (currentTier < requiredTier) { ... }
```

But `getTier` simply looks up the capability in `CAPABILITY_TIER_MAP`, which maps `phone` to the constant `3`. The comparison is `3 < 2` (false) or `3 < 3` (false) — the branch is dead code. The test "places a call to an unknown number (tier 3 required)" passes precisely because the gate never fires, not because the permission was correctly granted.

To actually enforce tiered access, the tier needs to come from a per-user/per-session authority level, not from the capability's static sensitivity classification.

## TwiML XML injection surface

Every method in `TwiMLBuilder` interpolates arguments directly into XML strings:

```typescript
static dial(number: string, callerId?: string): string {
  const callerAttr = callerId ? ` callerId="${callerId}"` : '';
  return `<Response><Dial${callerAttr}>${number}</Dial></Response>`;
}
```

If `number` is `+1555""><Reject/><!--`, the resulting XML breaks out of the `<Dial>` element. The `gather` method puts `options.action` into an attribute value — same pattern. This is a confirmed injection risk in any code path where user input flows into TwiML generation (which is the designed purpose of the builder).

Standard mitigation: escape `&`, `<`, `>`, `"`, `'` in all interpolated values, or use the `xmlbuilder` package already pulled in as a transitive dependency of `twilio`.

## Live mode silent degradation

The constructor sets `this.twilioClient = client ?? null` when credentials are present. If no client is passed (normal production path), `liveMode` is true but `twilioClient` is null. The `placeCall` method guards with `if (this.liveMode && this.twilioClient && this.config)`, so it falls through to mock mode rather than crashing — but this means live mode silently degrades to mock mode in production unless the caller remembers to inject a client. The constructor should instantiate the real Twilio client from the SDK when no override is supplied:

```typescript
import Twilio from 'twilio';
this.twilioClient = client ?? Twilio(accountSid, authToken);
```

Without this, the `twilio` dependency declared in `package.json` is unused at runtime. Similarly, `endCall` and `transferCall` have live-mode branches guarded by `this.twilioClient` that will never execute.

## Session lifecycle and unbounded growth

The `CallSessionManager` accumulates sessions indefinitely — there is no eviction of terminal (`ended`/`failed`) sessions. For a long-running server handling many calls, the Map grows without bound.

`TwilioPhoneProvider.dispose()` iterates active sessions and ends them locally, but in live mode it never calls the Twilio REST API to terminate the remote call legs. This means server-side disposal leaves active Twilio calls running (and billing).

## NativePhoneProvider coverage gap

`NativePhoneProvider` throws on every method except `placeCall`. The `CallAgent` catches those throws and returns `success: false`, but the error messages are generic ("Not supported in native dialer mode") rather than capability-specific.

No tests cover `NativePhoneProvider`. Its interaction with `CallAgent` — particularly the throw-to-error propagation path — is untested.

## Test coverage gaps

Not tested: the `NativePhoneProvider` (no test file); the `CallAgent` when `permState` is `'prompt'` (it passes through without requesting permission); error propagation from the Twilio client (e.g., network timeout on `calls.create`); the `transferCall` live-mode path where the conference is created but the original call is also ended (potentially racing); XML injection via `TwiMLBuilder`.
