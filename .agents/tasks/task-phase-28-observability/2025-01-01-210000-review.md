# Observability package expansion + quality gates testing infrastructure

This branch introduces two major packages: an expanded `@quant/observability` with instrument decorator, OTel setup, SLO definitions, burn-rate alerting, chaos experiment templates, runbook generation, synthetic monitoring, dashboard config generation, and PagerDuty integration; plus a `@quant/testing` package with coverage/mutation gates, Playwright E2E journey skeletons, k6 load test scripts, OWASP ZAP config, and Snyk integration. A GitHub Actions security-scan workflow ties the security tooling together. The implementation delegates heavily to pre-existing in-package core classes.

**Watch for:** The `@instrument()` decorator uses legacy TC39 property-descriptor syntax which won't work as a native decorator at call sites (confirmed); the `OTelSetup.instrumentFastify` leaks span references in a `Map` that is never bounded or pruned on error paths (confirmed); the `SyntheticMonitor.executeCheck` always returns 200, making the entire probe system a no-op against real endpoints (confirmed); the PagerDuty integration stores incidents in an unbounded in-memory map with no eviction (confirmed).

## High-level view

The observability package adds nine modules, each wrapping or extending existing `core/` classes (DistributedTracer, MetricsCollector, SLOTracker, ChaosEngine) with higher-level APIs. The decorator and OTel setup are the two pieces intended for direct integration into app services. The remaining modules (SLO definitions, burn-rate calculator, chaos runner, runbook generator, synthetic monitor, dashboard config, PagerDuty) are operational tooling ‚Äî config generators and simulators rather than runtime production code. Because none of these depend on external OTel SDKs or real HTTP clients, they remain internal simulations of the real operational surface.

The testing package adds enforcement logic (coverage gate, mutation gate) and declarative test definitions (30 E2E journey skeletons, 3 k6 load scripts, ZAP + Snyk configs). These are config/definition files, not runnable tests against live services.

The SLO YAML files and runbook markdown files are static documentation artifacts ‚Äî they're not consumed by any code in this diff, and their schema doesn't match the TypeScript validator.

## `@instrument()` decorator and tracer coupling

The decorator relies on a module-level `globalTracer` variable set via `configureInstrument()`. If any decorated method runs before `configureInstrument` is called, it silently falls through to the untraced path with no warning. Production use should at minimum log a warning on the fallthrough path to make misconfiguration visible.

The implementation uses the legacy TypeScript experimental decorators (`(target, propertyKey, descriptor) => PropertyDescriptor`). The TC39 stage 3 decorator proposal uses a different signature. The test file acknowledges this by manually applying the decorator via `Object.getOwnPropertyDescriptor` rather than using `@` syntax. Any consuming service that enables TC39 decorators (the default in newer TypeScript/esbuild configurations) will get a runtime error or silent breakage.

## OTel Fastify span lifecycle leak

`instrumentFastify` maintains a `Map<string, Span>` keyed by `request.id`:

```typescript
const spanMap = new Map<string, Span>();

app.addHook('onRequest', (...args) => {
  spanMap.set(requestId, span);
});

app.addHook('onResponse', (...args) => {
  spanMap.delete(requestId);
});
```

If `onResponse` never fires (client disconnects, server crash, unhandled error in middleware that skips the response hook), the span entry remains in the map indefinitely. In a high-traffic service, this is a memory leak. Fastify has an `onError` hook and `onClose` that could serve as cleanup points. There's also no way to inspect or drain the map externally.

## SLO definitions and YAML drift

The TypeScript `parseSLOConfig` validates against a zod schema expecting `{ services: [{ serviceName, availability: { name, target, metric, window, burnRateThresholds, description }, ... }] }`. The YAML files use a different schema:

```yaml
service: quantmail
slos:
  availability:
    target: 0.999
    window_days: 30
    burn_rate_thresholds:
      critical: { burn_rate: 14.4, short_window: '5m', long_window: '1h' }
```

Fields like `window_days` (vs `window` in ms), string-encoded durations (`'5m'`), nested object burn rate thresholds (vs array), and missing `name`/`metric`/`description` fields mean the YAML cannot be parsed by `parseSLOConfig` without a translation layer that doesn't exist in this diff.

## Synthetic monitor as simulation-only

```typescript
private async executeCheck(config: ProbeConfig): Promise<{ statusCode: number }> {
  void config;
  return { statusCode: 200 };
}
```

This hardcodes success regardless of the probe URL, timeout, or headers. The class handles probe result tracking and journey step orchestration, but the actual checking capability is a no-op. For production use, `executeCheck` would need to accept an injected HTTP client. Tests pass because they're testing bookkeeping, not probe logic.

## Chaos experiment semantic mismatch

```typescript
diskFill(targetService: string, fillPercentage: number): ChaosExperiment {
  return this.engine.createExperiment(
    `disk-fill-${targetService}`,
    'resource',
    { resourceType: 'memory', resourceAmount: fillPercentage, ... },
    ...
  );
}
```

The underlying `ChaosEngine` only supports `'memory' | 'cpu'` resource types, so `diskFill` was mapped to memory as a workaround. The experiment name says "disk-fill" but simulates memory pressure ‚Äî consumers relying on the method name semantics will be misled.

## Test coverage gaps

Not tested: the `onRequest` hook when `request.id` is undefined (falls back to timestamp-based ID, potential key collision under concurrent requests), Prisma middleware error path, SLO burn rate with concurrent event streams, journey step timeout races, dashboard config with empty service list, the `@instrument()` decorator via actual `@` syntax.
