# Federation, Performance, Observability, and Security Hardening (v2)

All eight issues from the v1 review have been addressed. The OAuth2 timing attack is closed via `timingSafeEqual` with a self-compare pad on length mismatch. API key validation no longer leaks the raw secret. The semantic cache uses content-addressable keys. PII pattern ordering and anchoring have been tightened. The IMAP adapter supports an `AuthenticationDelegate`. DKIM is clearly marked as simulation. WAF rules sort by action severity before first-match evaluation. Work queue tests now cover retry-to-completion and dead letter paths.

**Watch for:** The work queue advertises `retryDelayMs` and `rateLimit` config but the `processNext` implementation enforces neither — jobs retry immediately and concurrency is unchecked (confirmed). The semantic cache key is a 32-bit integer hash (`(hash << 5) - hash + char`) which has high collision probability for prompts differing only at the end or by case changes (confirmed). The `constantTimeEquals` length-mismatch branch leaks secret length via `Buffer.from()` allocation timing — marginal risk given fixed-length system-generated secrets (possible).

## High-level view

The `constantTimeEquals` helper pads the length-mismatch case with `timingSafeEqual(bufA, bufA)`, which takes time proportional to the stored secret rather than `max(a,b)`. The timing differential from `Buffer.from(b)` allocation could fingerprint the stored secret's length class, but since client secrets are system-generated 64-char hex strings, this reveals nothing an attacker doesn't already know from the generation algorithm.

The work queue's `retryDelayMs` and `rateLimit` fields are stored in options but `processNext()` immediately re-processes failed jobs without delay and permits unlimited concurrent processing. The new test named "retries with configurable delay" is misleading — it asserts retry counts work but doesn't prove any delay occurs.

The semantic cache deduplication fix is correct: same prompt maps to the same key, and `set()` replaces existing entries. The underlying hash is Java's `hashCode` truncated to 32 bits. For a cache sized at 1000 entries, random collisions are rare, but the hash has weak avalanche properties for natural language prompts sharing long prefixes.

## Work queue — delay and rate limit are dead config

The `processNext` method finds the first `waiting` job and immediately executes it:

```ts
const job = queue.find((j) => j.status === 'waiting');
if (!job) return null;

job.status = 'active';
job.startedAt = Date.now();
job.attempts++;

try {
  await definition.handler(job.data);
} catch (error) {
  if (job.attempts >= definition.options.maxRetries) {
    job.status = 'dead-letter';
  } else {
    job.status = 'waiting'; // immediately re-eligible
  }
}
```

No `setTimeout` or `nextRetryAfter` timestamp is applied. A caller invoking `processNext` in a loop would retry the same job instantly N times until it either succeeds or dead-letters. The `rateLimit.maxConcurrent` field is never checked — the queue doesn't track how many jobs are currently `active`.

The test "retries with configurable delay (retryDelayMs config is respected)" calls `processNext` three times synchronously and asserts on attempt counts. It proves retry semantics work but doesn't prove delay enforcement — because there is none to enforce. Acceptable for a behavioral contract layer if documented as "caller is responsible for scheduling delays", but as written the config silently does nothing.

## Semantic cache — 32-bit hash collision characteristics

```ts
private generateKey(prompt: string): string {
  let hash = 0;
  for (let i = 0; i < prompt.length; i++) {
    const char = prompt.charCodeAt(i);
    hash = (hash << 5) - hash + char;
    hash |= 0;
  }
  return `sem_${hash}`;
}
```

This is Java's `String.hashCode()` with known weaknesses: strings like `"Aa"` and `"BB"` hash to the same value. A collision means `set()` silently overwrites the previous entry. The `get()` method uses cosine similarity on embeddings for retrieval, so a key collision doesn't affect read correctness (the embedding comparison rejects wrong entries below threshold). The only impact is premature eviction of one entry by another via key collision in the `Map`. For production use the key should be a cryptographic hash prefix.

## OAuth2 constant-time comparison — length leak

```ts
private constantTimeEquals(a: string, b: string): boolean {
  const bufA = Buffer.from(a);
  const bufB = Buffer.from(b);
  if (bufA.length !== bufB.length) {
    timingSafeEqual(bufA, bufA);
    return false;
  }
  return timingSafeEqual(bufA, bufB);
}
```

When lengths differ, the method runs `timingSafeEqual(bufA, bufA)` then returns false. The comparison itself is constant-time, but `Buffer.from(b)` allocation scales with `b`'s length. An attacker sending 10-byte vs 1000-byte secrets would see ~microsecond allocation differences that could fingerprint the stored secret's length. Since client secrets are `randomBytes(32).toString('hex')` (always 64 chars), an attacker learns only what the generation algorithm already implies.
