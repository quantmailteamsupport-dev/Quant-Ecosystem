# Federation, Performance Engineering, Observability, and Security Hardening

Four new infrastructure-layer packages spanning open protocol federation (CalDAV/CardDAV, IMAP/POP3/SMTP, OAuth2 developer platform), performance cost engineering (semantic LLM cache, AI cost router, work queue, DB optimizer), observability and SRE (browser RUM, PII scrubber, alert generators, disaster recovery, game day chaos), and security hardening (STRIDE threat modeling, OWASP scanner, API fuzzer, WAF, secret manager, container security, compliance framework). The implementation is all in-memory simulation — no real network calls, no persistent state — establishing the behavioral contracts and type surface for each subsystem. Infra artifacts include Grafana dashboards, Prometheus alerts, Helm mTLS policies, Trivy config, a hardened Dockerfile, and compliance/runbook documentation.

**Watch for:** OAuth2 provider validates PKCE and checks redirect URIs but performs client secret comparison with `===` on raw strings rather than constant-time comparison (confirmed). The APIKeyManager returns the full secret key in validation results, which risks logging the key downstream (confirmed). The PII scrubber runs all regex replacements in a fixed order where earlier patterns can partially destroy text that later patterns need to match, producing incomplete redaction on overlapping patterns (likely). WAF rule engine short-circuits on first match — a "log" rule shadows subsequent "block" rules (confirmed).

## High-level view

The federation package delivers CalDAV/CardDAV servers, mail protocol adapters simulating IMAP state machines, and a full OAuth2 authorization server with PKCE support. The OAuth2 provider is the most production-realistic component with authorization codes, client credentials, refresh token rotation, and introspection. The security gap is client secret comparison without timing-attack protection, and the API key manager exposing raw secrets in validation responses.

The performance package introduces a semantic LLM cache using cosine similarity on embeddings, an AI cost router that classifies prompt complexity and selects model tiers under budget/latency constraints, a priority work queue with dead letter semantics, and a database optimizer providing index recommendations and partition strategies. The semantic cache's key-generation scheme uses a hash combined with `Date.now()`, preventing deduplication of identical prompts stored across different milliseconds.

The observability package layers browser RUM (Web Vitals scoring with batched export), PII scrubbing for log pipelines, programmatic Prometheus alert rule generation following RED/USE/SLO-burn-rate patterns, disaster recovery planning with RPO-driven backup scheduling, and a game day runner for chaos engineering drills. The infra layer pairs this with ready-to-import Grafana dashboards and a comprehensive Prometheus alert rules YAML.

The security package implements STRIDE threat modeling, an OWASP Top 10 pen test scanner, an API fuzzer, WAF rule engine with OWASP Core Rule Set patterns, secret management with vault adapter abstraction, container Dockerfile validation, mTLS configuration, and a multi-framework compliance auditor (GDPR, CCPA, PCI-DSS, SOC 2, DPDP, COPPA). The compliance framework models six regulatory regimes with control-level audit scoring but lacks consent withdrawal validation for GDPR Article 7(3).

<summary>Issues (8)</summary>

1. **OAuth2 client secret timing attack** — `handleClientCredentials` compares `client.clientSecret !== request.clientSecret` with `===`. Use `timingSafeEqual` from `node:crypto` to prevent timing-based secret extraction.
2. **API key leaked in validation response** — `APIKeyManager.validate()` returns the full `APIKey` object including the raw `key` field. Callers logging the result will persist the secret. Return a redacted view or omit the key field.
3. **Semantic cache key collision** — `generateKey` combines a 32-bit hash with `Date.now()`. Two `set()` calls for the same prompt in different milliseconds produce duplicate entries; two calls in the same millisecond may collide. Use the prompt hash alone or a content-addressable key.
4. **PII scrubber ordering sensitivity** — Patterns are applied sequentially; an IP like `192.168.1.100` could be partially matched by the phone regex first (e.g., `168-1-100` substring). Add a test for mixed-format strings or anchor patterns more tightly.
5. **IMAP adapter accepts all credentials** — `authenticate` always returns OK. If the adapter is ever wired to a real transport, it fails open. Add a hook or flag to enforce credential checks.
6. **DKIM signature uses HMAC not RSA** — `SMTPRelay.signDKIM()` uses HMAC-SHA256 with the private key as HMAC secret, but real DKIM uses RSA (or Ed25519) signatures verified against a DNS public key. The naming implies standard compliance that isn't present.
7. **WAF evaluateRequest short-circuits on first match** — The `for` loop returns on the first matching rule. If a request triggers both a "log" rule and a later "block" rule, the log rule's action (`allow`) takes precedence and the block is skipped. Rules need priority ordering or multi-match semantics.
8. **No test coverage for work queue retry backoff or rate limiting** — The `WorkQueueManager` defines `rateLimit` and `retryDelayMs` config but tests never exercise timing-dependent retry logic or rate enforcement.

## OAuth2 provider — secret comparison and key exposure

The client secret comparison at line ~220 (`client.clientSecret !== request.clientSecret`) uses JavaScript string equality, which exits on the first mismatched byte. For a developer platform where client secrets are transmitted over TLS, this is a low-severity but real timing oracle. The fix is a single-line change to `timingSafeEqual(Buffer.from(client.clientSecret), Buffer.from(request.clientSecret))` with a length check guard.

The `APIKeyManager.validate()` returns the complete `APIKey` object including the `key` field (the raw secret). Any middleware that logs validation results will persist the secret in plain text. A safer pattern:

```ts
return { valid: true, key: { ...apiKey, key: undefined } };
```

Or better, a separate `APIKeyMetadata` type without the secret field.

## Semantic LLM cache — duplicate entries and O(n) lookup

The key generation function appends `Date.now()` to a 32-bit hash:

```ts
private generateKey(prompt: string): string {
  let hash = 0;
  for (let i = 0; i < prompt.length; i++) {
    const char = prompt.charCodeAt(i);
    hash = (hash << 5) - hash + char;
    hash |= 0;
  }
  return `sem_${hash}_${Date.now()}`;
}
```

The same prompt stored twice produces two distinct keys. The `get()` method finds matches by iterating all entries and computing cosine similarity, so duplicates don't break retrieval — but they waste capacity and cause unpredictable eviction behavior (the LFU eviction picks the entry with fewer hits, which may be the newer duplicate). If the intent is deduplication (newer response replaces older), the key should be hash-only.

The O(n) linear scan in `get()` will become expensive at the configured `maxSize: 1000` in production. An approximate nearest neighbor index would be needed, but that's future work for what's currently a behavioral contract.

## WAF rule engine — first-match short-circuit

```ts
for (const rule of this.rules) {
  if (!rule.enabled) continue;
  if (regex.test(content)) {
    const decision: WAFDecision = {
      action: rule.action === 'log' ? 'allow' : rule.action,
    };
    return decision;
  }
}
```

Rule ordering determines behavior. A "log" rule inserted before a "block" rule will shadow the block. The built-in rules are all "block", so this only manifests with custom rules added via `addCustomRule` (which appends). But if someone removes and re-adds a builtin rule, a custom "log" rule would then precede it. Either document first-match-wins semantics explicitly or implement severity-based priority sorting.

## PII scrubber — pattern interference

The scrubber applies regex replacements sequentially: email, phone, SSN, credit card, IP, JWT. A string like `"user123-456-7890@192.168.1.1"` could have the phone pattern match the `123-456-7890` portion before the email pattern gets a clean run (depending on replacement order). The current tests cover clearly separated PII types but don't exercise adversarial overlaps where one pattern partially consumes text needed by another.

## SMTP DKIM — incorrect cryptographic primitive

`SMTPRelay.signDKIM()` uses HMAC-SHA256 with the "private key" as HMAC secret. Real DKIM uses RSA or Ed25519 asymmetric signatures verified against a DNS public key record. The method names its parameter `privateKey` and produces output formatted like a DKIM-Signature header, implying standard compliance. Either rename to something like `signHMAC` to avoid confusion, or note in the type/docs that this is a simulation placeholder.

## Compliance framework — consent withdrawal gap

The consent record checker (`checkConsentRecords`) validates presence of purpose, version, and timestamp but doesn't check for consent withdrawal. A user who withdrew consent for a purpose that was later re-versioned would still pass validation. GDPR Article 7(3) requires withdrawal to be as easy as granting consent and to be trackable. The fix is to add a `withdrawn` field check and validate that no active processing occurs against a withdrawn purpose.

## Test coverage gaps

Key untested paths:

- Work queue: retry timing, dead letter after max retries with delay, rate limit enforcement
- OAuth2: expired authorization code rejection (code exists but no test)
- WAF: rule ordering/shadowing between log and block actions
- Semantic cache: hash collision behavior when same prompt stored twice
- PII scrubber: adversarial overlapping patterns
