# Conversational multimodal engine skeleton (quant-live)

New package `packages/quant-live/` scaffolds a streaming ASR pipeline with voice activity detection, session state management, turn-taking with barge-in, and per-stage latency tracking. The approach is contract-first: interfaces and a factory pattern define the ASR provider boundary, with a fetch-based Whisper server implementation and a WebGPU stub that falls back to the server provider. The pipeline itself is a thin orchestrator connecting audio input to ASR results via callbacks, with a LatencyTracker instrumenting each stage.

**Watch for:** The pipeline starts latency measurements on each audio chunk but never completes them (confirmed) â€” `endMeasure` is never called, so the pending map grows unbounded and no percentile data is ever produced. The WhisperServerProvider sends raw Float32Array samples serialized as JSON arrays (confirmed), which is bandwidth-inefficient and incompatible with the OpenAI Whisper API's expected multipart/form-data format.

## High-level view

The pipeline's latency tracking is structurally broken: `startMeasure` is called per chunk in `feedAudio`, but the corresponding `endMeasure` never fires because the ASR result callback doesn't close the measurement. In tests, this is masked by the test manually calling `endMeasure` on the tracker â€” a pattern that won't exist at runtime. Even with the call added, there's no correlation mechanism to match async ASR results back to the chunks that produced them.

The WhisperServerProvider fires one HTTP POST per audio chunk with no batching, buffering, or streaming upload. Each request carries raw PCM samples serialized to a JSON number array. For 16kHz mono audio at 20ms chunks, this is 50 requests/second with no backpressure. The request format (JSON body) doesn't match OpenAI's `/v1/audio/transcriptions` contract (multipart form-data with file upload).

The VAD uses RMS energy detection with configurable thresholds and speech duration gates. Detection correctness depends on callers providing monotonically increasing timestamps on AudioChunk â€” if timestamps don't advance, `minSpeechDuration` can never be satisfied regardless of energy levels. This contract isn't documented or enforced.

The SessionManager, LivePipeline, VoiceActivityDetector, and TurnManager are independent classes with no wiring. VAD events don't trigger turn changes, session state transitions don't pause/resume the pipeline, and ASR results don't update session state. The integration surface is where the real complexity will live.

## Pipeline latency measurement gap

`LivePipeline.feedAudio` starts a measurement on every chunk:

```typescript
const measureId = `chunk-${this.chunkId++}`;
this.latencyTracker.startMeasure('asr', measureId);
this.asrProvider.feedAudio(chunk);
```

The ASR result callback (`handleASRResult`) invokes transcript callbacks but never calls `endMeasure`. The pending map accumulates entries indefinitely. The pipeline test verifies this works by manually calling `tracker.endMeasure('asr', 'chunk-0')` after feeding audio â€” the test is testing the tracker's API, not the pipeline's integration with it.

Even if `endMeasure` were added to the result handler, there's no correlation mechanism: the ASR provider receives chunks and emits results, but nothing ties a specific result back to the chunk ID that produced it. A streaming ASR provider may aggregate multiple chunks into a single result, making 1:1 correlation impossible without a different measurement strategy (e.g., measuring round-trip from first chunk of a speech segment to final result).

## WhisperServerProvider: per-chunk HTTP and JSON-serialized PCM

Each call to `feedAudio` triggers an independent `fetch` POST with body:

```json
{ "audio": [0.1, 0.2, ...], "sample_rate": 16000, "model": "whisper-1", "language": "en" }
```

At 16kHz with 20ms chunks, this fires 50 requests/second, each carrying 320 floating-point numbers as JSON text. The OpenAI `/v1/audio/transcriptions` endpoint expects multipart form-data with a file upload (WAV/MP3/etc.), not JSON with raw samples.

There's no request queuing, backpressure, or cancellation. If the network is slow, requests pile up with no bound. The `processChunk` promise rejection is caught and forwarded to error callbacks, but a failing endpoint generates one error callback per chunk (50/sec) with no circuit breaker.

## VAD timestamp dependency

The VAD tracks `speechStartTime` to enforce `minSpeechDuration` before emitting `speech-start`. When the first above-threshold chunk arrives, `speechStartTime` is set to `chunk.timestamp`. On subsequent above-threshold chunks, it checks `now - speechStartTime >= minSpeechDuration` where `now` is `chunk.timestamp`. If chunks arrive with non-advancing timestamps (e.g., all zero, or real-time but with the same timestamp repeated), speech detection silently never triggers. The tests use incrementing timestamps so this works, but the contract isn't documented or validated.

## Test coverage gaps

What's not tested:

- Integration between VAD events and TurnManager (the barge-in path end-to-end)
- Pipeline behavior when ASR provider errors (the pipeline registers no error callback on the provider)
- Session lifecycle driving pipeline start/stop
- TranscriptManager (no dedicated test file)
- Memory growth under sustained load (pending measurements accumulating)
