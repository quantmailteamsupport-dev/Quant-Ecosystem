# Conversational multimodal engine skeleton (v2)

New package `packages/quant-live/` implements a streaming ASR pipeline with voice activity detection, session state management, turn-taking with barge-in, and per-segment latency tracking. The fix commit addressed all four v1 concerns: latency tracking now operates per-segment instead of per-chunk, audio is serialized as WAV via multipart form-data, request backpressure is enforced with batching and a concurrency limiter, and the VAD's timestamp contract is documented with out-of-order rejection.

**Watch for:** The pipeline does not register an error callback on the ASR provider (confirmed) — if the WhisperServerProvider emits errors (e.g., fetch failures), the pipeline silently drops them, leaving callers with no visibility into ASR failures. The `stop()` method on WhisperServerProvider calls `flushBuffer()` synchronously but the resulting fetch is fire-and-forget (likely) — callers cannot await completion of the final batch.

## High-level view

The pipeline's segment-based latency strategy opens a measurement when the first chunk of a segment arrives and closes it when the ASR result comes back. One gap remains: the pipeline ignores errors from the ASR provider entirely, so fetch failures surface only through error callbacks registered directly on the provider, not through any pipeline-level notification.

The WhisperServerProvider now batches audio into WAV blobs and sends multipart form-data at a configurable interval, with a concurrency cap that prevents unbounded parallel requests. The concurrency mechanism is simple — when `inFlightCount >= maxConcurrency`, the batch timer skips the flush — but buffered chunks accumulate without bound during backpressure. Under sustained load with a slow endpoint, memory grows linearly until the endpoint recovers.

The VAD now explicitly documents and enforces its timestamp contract: out-of-order chunks are silently discarded, and two new tests verify both the reject-path and the equal-timestamp accept-path.

## Pipeline error propagation gap

The pipeline's `start` method wires exactly one callback on the ASR provider:

```typescript
this.asrProvider.onResult((result: ASRResult) => {
  this.handleASRResult(result);
});
```

It never calls `this.asrProvider.onError(...)`. The WhisperServerProvider catches promise rejections in `flushBuffer` and forwards them to registered error callbacks, but the pipeline doesn't participate in that channel. The only way to observe ASR failures would be to register error callbacks directly on the provider instance before passing it to the pipeline, breaking the encapsulation the pipeline is supposed to provide.

The pipeline exposes `onResponse` and `onAudioOut` callback registrations but has no `onError` equivalent. Adding one and wiring it to the provider's error channel would close this gap.

## Buffer growth during backpressure

The backpressure mechanism in `WhisperServerProvider`:

```typescript
private flushBuffer(): void {
  if (this.audioBuffer.length === 0) return;
  if (this.inFlightCount >= this.maxConcurrency) return;  // skip flush
  const chunks = this.audioBuffer.splice(0);
  this.processBatch(chunks).catch(...);
}
```

When concurrency is saturated, `flushBuffer` returns without draining the buffer, but `feedAudio` continues to push chunks. At 16kHz mono with 20ms frames, that's ~50 chunks/sec accumulating. If the endpoint is slow for 10 seconds, that's 500 chunks (~160KB of audio data) held in memory. For a single session this is manageable, but multi-session deployments or longer stalls would need a cap.

## Stop semantics and the final flush

```typescript
stop(): void {
  this.running = false;
  if (this.batchTimer !== null) {
    clearInterval(this.batchTimer);
    this.batchTimer = null;
  }
  this.flushBuffer();  // synchronous call, but processBatch is async
}
```

Production callers have no mechanism to await the flush — `stop()` returns `void`. If a session ends and the caller immediately discards references, the final fetch may or may not complete depending on event loop state. A `drain(): Promise<void>` or async `stop()` would give callers deterministic shutdown.
