# Realtime event backbone, offline-first sync, and reliability primitives

Phase 10 and Phase 11 add eleven new modules across five packages, establishing the event validation layer (`EventSchemaRegistry`), notification routing (`NotificationFanout`), write deduplication (`IdempotencyKeyStore`), a real WebSocket gateway (`ws-gateway`), dead letter tracking (`QueueDeadLetter`), and a full offline-first client stack (optimistic updates, offline queue, retry with backoff, conflict store, data retention, sync status indicator). The approach is consistent: in-memory implementations with clean interfaces, subscriber patterns for UI binding, and no external infrastructure required at dev time.

**Watch for:** Idempotency store does not guard against concurrent in-flight executions of the same key (confirmed). The WS gateway silently degrades to health-only mode without structured logging or metrics when JWT_SECRET is missing (confirmed). The `withIdempotency` helper caches `null` return values but `get()` also returns `null` for "not found", creating an ambiguity that the test suite itself comments on (confirmed).

## High-level view

The event schema registry is a straightforward Zod-based validation layer. It registers 18 event types at construction and exposes runtime `validate()` for boundary enforcement on WebSocket messages and queue events. The schemas are static and the registry is not distributed — a single instance per process, no cross-service coordination.

The notification fanout is a pure routing layer that delegates all preference logic to the existing `PreferenceService`. It determines per-recipient channel routing and applies mention-based priority escalation but does not dispatch notifications itself — it returns routing decisions for something upstream to act on.

The idempotency key store is an in-memory TTL cache with a periodic sweep. The critical gap is the `withIdempotency` helper: it checks cache, then executes, then stores — so two concurrent calls with the same key can both execute the operation before either stores the result.

The WS gateway wires the realtime package's `WebSocketServer` with JWT auth, presence, and channel management into a standalone service. The 32-character minimum on JWT_SECRET is a sensible guard, but the service offers no signal to operators when it's running in degraded mode beyond a console.log.

The offline-first stack (optimistic updates, offline queue, retry backoff, conflict store, sync status indicator) provides a complete apply/confirm/rollback lifecycle for mutations, priority-ordered queuing with deduplication, exponential backoff with jitter, and a subscriber-driven status model for UI. These are all single-process, in-memory abstractions suitable for client-side use.

Data retention policy evaluates records against configurable per-entity-type rules and produces archive/delete batches. It treats `soft-delete` identically to `hard-delete` in its output classification, which may surprise consumers who expect soft-delete semantics to differ.

<summary>Issues (5)</summary>

1. **Idempotency race on concurrent calls** — `withIdempotency` does not lock the key during execution. Two concurrent calls with the same key will both execute the operation. Add a pending-promise cache (check-or-insert pattern) so the second caller awaits the first's result.
2. **Null ambiguity in IdempotencyKeyStore.get()** — Storing `null` as a valid result is indistinguishable from "key not found" via `get()`. Either wrap stored values in a container (`{ value: T }`) or change the return type to `{ found: boolean; result: T | undefined }`.
3. **WS gateway silent degradation** — When JWT_SECRET is absent or too short, the gateway logs a plain string and runs health-only. Add a structured warning (or metric emission) so operators can detect misconfiguration in monitoring systems.
4. **Soft-delete treated as hard-delete** — `DataRetentionPolicy.evaluate()` routes `soft-delete` strategy records into `toDelete[]` alongside `hard-delete`. Consumers that want to distinguish (e.g., set a `deletedAt` flag vs. actually remove) receive no signal of the intended strategy.
5. **OfflineOperationQueue.replayAll leaves failed ops in queue** — Failed operations remain in the map after replay, but there's no retry count or backoff. Repeated `replayAll()` calls will re-attempt the same permanently-failing operations with no circuit-breaking.

## Idempotency race window in `withIdempotency`

The helper function implements a check-then-act pattern without any concurrency guard:

```typescript
export async function withIdempotency<T>(
  store: IdempotencyKeyStore,
  key: string,
  ttlMs: number,
  operation: () => Promise<T>,
): Promise<T> {
  const cached = store.get(key);
  if (cached !== null) {
    return cached as T;
  }
  const result = await operation();
  store.set(key, result, ttlMs);
  return result;
}
```

Between the `get` returning `null` and the `set` after `await operation()`, any number of concurrent callers will see the key as absent and execute the operation. For a helper whose purpose is deduplication of writes, this is the primary failure mode. The standard fix is a pending-promise map: on first call, store a promise in a separate map; concurrent callers await that promise instead of re-executing.

The `null` ambiguity compounds this: if the operation legitimately returns `null`, `get()` will return `null` (indistinguishable from "not found"), so subsequent calls re-execute even when the result is cached. The test file itself notes this ambiguity in a comment: "null is a valid stored result, but get() returns null for 'not found' too".

## WS gateway configuration and degradation posture

The gateway makes JWT_SECRET availability a runtime branch:

```typescript
if (jwtSecret.length >= 32) {
  // full WS server
} else {
  console.log('ws-gateway: JWT_SECRET not configured, running health-only mode');
}
```

This design means the health check endpoint reports healthy even when the service is not actually performing its primary function. An operator monitoring only health endpoints will see green for a gateway that cannot accept WebSocket connections. In production this should either fail startup (if the secret is required) or emit a metric/log at warning level with structured context that monitoring can alert on.

The `void server.getPresenceManager()` / `void server.getChannelManager()` / `void server.getDeliveryManager()` calls appear to exist solely to suppress unused-variable lint rules or verify importability. They don't connect these managers to any routing logic within this file — the actual message routing happens internally in the `WebSocketServer` class.

## Soft-delete conflation in DataRetentionPolicy

The `evaluate()` method collapses two semantically different strategies into one output bucket:

```typescript
if (rule.strategy === 'archive') {
  toArchive.push(record);
} else {
  toDelete.push(record); // both 'hard-delete' AND 'soft-delete' land here
}
```

A consumer calling `evaluate()` receives `toDelete[]` but cannot distinguish between records that should be permanently removed and records that should merely be marked with a `deletedAt` timestamp. The `RetentionRule.strategy` field offers three options but the evaluation output only distinguishes two. Either add a `toSoftDelete[]` output category or include the strategy on each record in the output array.

## OfflineOperationQueue replay without circuit-breaking

`replayAll()` iterates all queued operations and attempts each one. Successful operations are removed; failed ones stay. There is no retry counter, no backoff between replay attempts, and no way to mark an operation as permanently failed. If a consumer calls `replayAll()` on a timer (as is common for offline-first reconnection), a single poison operation will be re-attempted indefinitely until it either succeeds or hits `maxAge` expiry (default: 7 days).

The deduplication-by-key behavior means a newer operation for the same key will evict the failing one, which provides some natural healing. But for unique-key operations (non-overwriting writes), there's no escape hatch.

## Test coverage shape

Tests exist for all eleven modules and exercise the primary paths: happy-path validation, expiration, priority escalation, dedup, backoff calculation. What's not covered:

- Concurrent `withIdempotency` calls (the race condition described above)
- WS gateway startup/shutdown (no integration test, which is understandable given it depends on network)
- `OfflineOperationQueue.replayAll` with partial failures across multiple batches
- `SyncStatusIndicator` interaction with `OptimisticUpdateManager` (they're tested in isolation)
- `NotificationFanout` with quiet hours active (tested in preference-service but not through fanout)
- `DataRetentionPolicy` with mixed strategies in a single evaluate call distinguishing soft from hard delete behavior
