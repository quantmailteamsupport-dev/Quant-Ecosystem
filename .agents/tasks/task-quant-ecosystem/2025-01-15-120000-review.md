# 9-app TypeScript monorepo for the Quant Ecosystem

A greenfield monorepo scaffolding 9 consumer apps (QuantMail, QuantChat, QuantSync, QuantAds, QuantTube, QuantNeon, QuantEdits, QuantMax, QuantAI) with 6 shared packages (common, database, auth, ai, shared-ui, realtime). The architecture positions QuantMail as the OAuth2 identity provider and QuantAI as the central AI orchestrator, with all apps following a consistent server/router/middleware/service/routes layering. The shared packages define ecosystem-wide types, database schemas, real-time infrastructure, and a centralized AI engine with per-app service specializations. This is structural code — it cannot be compiled or tested due to INTEGRATIONS_ONLY network constraints — but it defines complete application logic rather than stubs.

**Watch for:** Duplicated Router and middleware implementations across all 9 apps instead of a shared `@quant/server` package (confirmed); inconsistent frontend patterns — some apps use real React components while others use a JSON-tree virtual DOM representation (confirmed); `corsMiddleware` in QuantTube takes a different argument signature than other apps (confirmed); AI services in apps reimplement capabilities that the `packages/ai` domain services already provide, bypassing the central engine's queuing/caching/budgeting (likely); JWT signature not verified at the app level — tokens are only decoded, not validated (confirmed).

## High-level view

Every app follows the same backend layering: a custom `Router` class that parses routes into regex patterns, a middleware stack (rate limiter, CORS, security headers, request ID, logging, error handler), and domain services behind controllers. The pattern is immediately recognizable across all 9 apps. The problem is that the Router class, middleware implementations, and token-decode logic are copy-pasted into each app rather than shared. A `@quant/server` package would eliminate ~1500 lines of duplicated infrastructure and prevent the drift that's already appearing between copies.

The frontend is fractured. QuantChat's pages are fully-formed React functional components with hooks, state, and JSX. QuantSync, QuantNeon, QuantMax, and others return plain JSON objects describing a virtual component tree — no JSX, no hooks, no React lifecycle. These two approaches are architecturally incompatible. The `@quant/shared-ui` package exports React hooks and components that can only be consumed by the React-style apps, leaving the JSON-tree apps unable to use the shared UI layer.

Each app's AI service re-implements capabilities (smart replies, moderation, translation, sentiment analysis) locally rather than delegating to `packages/ai/services/*`. The package-level AI services route through the central `AIEngine` with its queuing, caching, budgeting, and model routing. The app-level services bypass all of that, creating two parallel AI stacks.

Token validation at the app level only does base64url decode and expiry check — it skips JWT signature verification entirely. The `TokenService` in `packages/auth` implements proper signing, but no app imports or calls it for request authentication.

## Router and middleware copy-paste

Every app defines the same `Router` class with identical logic: `register(routes)`, `match(method, path)`, and `pathToRegex(path)`. The regex generation is the same across all 9 apps (`:param` → `([^/]+)`). Similarly, `RateLimiter`, `corsMiddleware`, `securityHeaders`, `requestIdMiddleware`, `loggingMiddleware`, `errorHandler`, and `AppError` are reimplemented per app.

The differences between copies reveal drift:
- QuantMail's `RateLimiter` has a `skipFailedRequests` option; QuantChat's does not.
- QuantMail's CORS middleware exposes rate-limit headers via `Access-Control-Expose-Headers`; QuantTube's does not.
- QuantTube's `corsMiddleware` takes `string[]` instead of `CorsOptions` — an incompatible signature.
- QuantMail and QuantChat include `X-RateLimit-Reset` headers; QuantTube does not.
- QuantNeon/QuantTube set `X-Frame-Options: SAMEORIGIN`; others set `DENY`.

A bug fix to token decoding or rate limiting requires changes to 9 files. If these were a shared package, the drift would not exist.

## Dual frontend architectures

QuantChat pages are standard React:

```tsx
export const ChatListPage: React.FC<ChatListProps> = ({ currentUserId }) => {
  const [conversations, setConversations] = useState<Conversation[]>([]);
  useEffect(() => { loadConversations(); }, []);
  // ... real JSX with event handlers, conditional rendering
};
```

QuantSync, QuantNeon, QuantMax return JSON-tree objects:

```tsx
export function FeedPage({ posts }: FeedPageProps) {
  return {
    type: 'div',
    className: 'feed-container',
    children: [ ... ]
  };
}
```

The JSON-tree approach has no way to manage state, trigger effects, or call APIs without an external runtime that interprets these trees. The `@quant/shared-ui` package exports React hooks (`useAuth`, `useRealtime`, `useTheme`) and component props interfaces, which are useless to the JSON-tree apps. This means 6-7 of the 9 apps cannot consume the shared UI package as designed.

## Token validation gap at the application boundary

Each app server has a private `decodeToken` method:

```typescript
private decodeToken(token: string): Record<string, any> | null {
  const parts = token.split('.');
  if (parts.length !== 3) return null;
  const payload = JSON.parse(Buffer.from(parts[1], 'base64url').toString());
  if (payload.exp && payload.exp < Math.floor(Date.now() / 1000)) return null;
  return payload;
}
```

No signature verification. The `TokenService` in `packages/auth` signs tokens with HMAC and verifies signatures, but apps don't import or use it for request authentication. In production, any HTTP client could forge a token with arbitrary `sub`, `role`, and `scopes` claims. The fix: apps should import `TokenService.validateAccessToken()` from `@quant/auth` instead of rolling their own decode.

## AI service layering confusion

The `packages/ai` package exports domain-specific services: `ChatAIService`, `MailAIService`, `ContentAIService`, `RecommendationAIService`. These are instantiated with the central `AIEngine` and meant to be called from apps.

Each app also has `api/services/ai-service.ts` that reimplements the same features independently:
- QuantChat's AI service has its own `SmartReplyEngine`, `ModerationEngine`, `TranslationEngine`, and `AIChatbot` — none delegate to `packages/ai`.
- QuantSync's AI service has its own sentiment analysis, content classification, and fact-checking.

The `packages/ai` services route through the `AIEngine` (queuing, caching, budgeting, model routing). The app-level services bypass all of that. Two parallel AI stacks would produce different behavior and different cost tracking in production.

## Package boundary enforcement

App-level code references `@quant/common` types (`PermissionScope`, `QuantApp`) in the auth and realtime packages, but apps define their own local type files for domain types (QuantChat defines `MessageType`, `DisappearMode`, `Conversation` locally rather than importing from `@quant/common` or `@quant/database`). The `packages/ai/core/engine.ts` imports `@quant/common`, but app-level AI services don't import from `@quant/ai` at all. The monorepo's package boundaries exist in the tsconfig path aliases but are not enforced by actual cross-package imports in most app code.
