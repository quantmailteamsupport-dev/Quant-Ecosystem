# Quality gates pipeline for the Quant monorepo

This change adds a multi-layer quality gate system to the monorepo: coverage enforcement at 80% on critical packages, Stryker mutation testing at 60%, a Playwright e2e framework with 30 skeleton journeys across all 13 apps, k6 load test scripts for chat fanout/feed ranking/search, and OWASP ZAP + Snyk security configs. A new GitHub Actions workflow (`quality-gates.yml`) orchestrates everything, and a separate `security-scan.yml` adds Trivy/CodeQL/ZAP/Snyk scheduled scanning. The implementation also bundles a full `packages/observability` package (Phase 28) which appears to have been committed on the same branch.

**Watch for:** The quality gates workflow has a coverage-gate job that re-implements the logic from `coverage-gate.ts` as inline Node.js in the YAML rather than importing the TypeScript module, creating a divergence risk (confirmed). The ZAP config contains hardcoded plaintext credentials (confirmed). Load tests and e2e tests all use `continue-on-error: true`, meaning the pipeline will never actually fail on these gates (confirmed). Phase 28 observability code is bundled into this PR unrelated to quality gates (confirmed).

## High-level view

The coverage gate has two implementations that can drift: a well-typed `CoverageGate` class in TypeScript and a separate inline `node -e` script in the workflow YAML. Both enforce 80% statement coverage on auth/payments/security, but they share no code. If thresholds or package lists change, someone must update both places.

Mutation testing is config-only. Stryker is installed at CI time (`pnpm add -D`), targeting auth/payments/security with a 60% break threshold. The `MutationGate` TypeScript class generates equivalent config and validates results, but the CI job doesn't use it either — it reads `reports/mutation/mutation.json` directly with another inline Node script.

The e2e layer defines 30 journey skeletons as typed data (not executable Playwright tests). These are `E2EScenario` objects with steps like `navigateTo`, `clickElement`, `assertVisible`. There is no Playwright test runner glue that consumes them — `playwright.config.ts` points `testDir` at `packages/testing/src/e2e` but those files export data, not `test()` calls. The journeys will not run as-is.

The k6 load tests are standalone JavaScript scripts that exercise chat, feed, and search APIs against a configurable base URL. They define meaningful thresholds (p95 latency, error rate) but run only on pushes to `main` and use `continue-on-error`, so they function as observability, not gates.

Security scanning adds comprehensive ZAP and Snyk TypeScript configs covering all 13 apps. The ZAP config hardcodes test credentials. The workflow `security-scan.yml` properly separates SAST (CodeQL), SCA (Trivy + Snyk), and DAST (ZAP baseline) with issue-creation on failure.

A large `packages/observability` addition (instrument decorator, OTel setup, SLO definitions, burn-rate alerting, chaos experiments, runbook generator, synthetic monitoring, dashboard config, PagerDuty integration, 9 YAML SLO files, 5 runbook docs, 9 test files) is included on this branch. It is Phase 28 material unrelated to quality gates.

## Coverage gate: two sources of truth

`packages/testing/src/coverage-gate.ts` provides a `CoverageGate` class that parses vitest coverage JSON, extracts per-package statement coverage, and returns a typed result with pass/fail per package. The workflow's `coverage-gate` job duplicates this logic as an inline Node one-liner:

```javascript
const criticalPackages = ['packages/auth', 'packages/payments', 'packages/security'];
const threshold = 80;
// ... filters files, counts statements, checks threshold
```

The TypeScript module uses short package names (`auth`) and `extractPackageName` via regex; the YAML uses full path prefixes (`packages/auth`). They will diverge the first time someone adds a critical package to one and forgets the other. The TypeScript module should be the single source invoked via `npx tsx packages/testing/src/coverage-gate.ts` or a small CLI wrapper.

## E2E framework: data without a runner

The 30 journey definitions in `journeys.ts` are `E2EScenario[]` — arrays of typed step objects (`navigateTo`, `clickElement`, `assertVisible`). Playwright doesn't know how to execute them. The `playwright.config.ts` points to `packages/testing/src/e2e` as `testDir`, but Playwright expects files containing `import { test } from '@playwright/test'` with `test('...', async ({ page }) => { ... })` calls.

Without a test file that iterates `journeys`, maps each step to Playwright page actions, and wraps them in `test()` blocks, `npx playwright test` will report 0 tests. The CI step has `continue-on-error: true`, masking this silently.

## Load tests: thresholds without enforcement

The k6 scripts define sensible thresholds (chat fanout p95 < 500ms / < 1% failure; search p95 < 300ms, p99 < 800ms / < 0.1% failure; feed ranking p95 < 1000ms), but the `load-tests` job uses `continue-on-error: true` on every `k6 run` step and only triggers on pushes to main. PRs never see load test results, and pushes to main never actually fail the workflow regardless of threshold violations. The load tests are fire-and-forget observability, not gates.

## Security config: hardcoded credentials

In `packages/testing/src/security/owasp-zap.ts`:

```typescript
authentication: {
  method: 'form',
  loginUrl: '/api/auth/login',
  credentials: { username: 'test@quant.app', password: 'test-password' },
  tokenExtraction: 'Set-Cookie: session=(.+)',
},
```

Even for test-only credentials, embedding them as string literals in source means they appear in git history, code search, and any tooling that indexes source. The config factory (`createZapConfig`) accepts overrides, so the default should reference `process.env.ZAP_TEST_USER` / `process.env.ZAP_TEST_PASS` with the literals relegated to local `.env` files or CI secrets.

## Mutation testing: runtime installation and path mismatch

The `mutation-testing` job installs Stryker packages on every run via `pnpm add -D @stryker-mutator/core @stryker-mutator/vitest-runner @stryker-mutator/typescript-checker`. The version is not pinned in `package.json` and will resolve to "latest" each time — the next run could get a breaking change. These should be declared in the root `package.json` devDependencies with pinned versions.

More critically, after `npx stryker run` completes, the threshold check looks for `reports/mutation/mutation.json`. The `stryker.conf.mjs` specifies `reporters: ['html', 'json', 'clear-text', 'progress']` but no output path override. Stryker's JSON reporter writes to a different default location. The threshold check will always hit the "No mutation report found, skipping threshold check" branch and pass silently.

## Test coverage gaps

Not tested: the actual workflow execution path (coverage gate inline JS vs. the TypeScript module), whether Playwright can discover and run the journey definitions, or whether the Stryker config produces a report at the expected path. The unit tests validate the TypeScript module logic in isolation but cannot catch the integration failures described above.

## Bundled Phase 28 observability package

The diff includes the complete `packages/observability` package with 11 source modules, 9 test files, 9 SLO YAML definitions, 5 runbook documents, and the agent task metadata for Phase 28. This is ~2500 lines unrelated to quality gates. It should ship on its own branch/PR for independent review.
