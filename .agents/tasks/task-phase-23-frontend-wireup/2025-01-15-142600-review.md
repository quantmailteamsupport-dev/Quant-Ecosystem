# React Query wire-up across 7 frontend apps

This change replaces hardcoded mock data in the Quant Ecosystem's frontend apps with React Query hooks that call typed API clients. QueryClientProvider is added at the root of each app, new hooks wrap API calls with `useQuery`/`useMutation`/`useInfiniteQuery`, and pages are rewritten to show loading, error, and empty states via shared-ui components. Error boundaries (class-based for pages-dir apps, `error.tsx` for app-router apps) catch unhandled rendering failures.

Watch for:

- **QuantMax `useFeed` and `useMatching` are not wired to the API** (confirmed) â€” the index page and matching page import hooks that still rely on local state and `setTimeout` simulation.
- **API surface mismatch in matching page** (confirmed) â€” `matching.swipe('like')` is called but the hook exposes `like(profileId)`, `pass(profileId)`, `superLike(profileId)`.
- **QuantTube `shows.tsx` retains hardcoded data** (confirmed) â€” renamed `MOCK_SHOWS` to `SAMPLE_SHOWS` but still loads from constants, not the API.
- **Dead hook: `useForYouFeed`** (confirmed) â€” created with proper React Query wiring but never imported by any page.

## High-level view

The QueryClientProvider setup is consistent: app-router apps (quantai, quantchat, quantmail) use a `'use client'` provider wrapper imported into `layout.tsx`, while pages-dir apps (quantedits, quantmax, quantneon, quantube) add `_app.tsx` with ErrorBoundary wrapping QueryProvider. The QueryClient configuration is identical across all 7 apps: 30s stale time, 3 retries, no refetch on window focus.

The new hooks follow a uniform pattern but treat the API response shape as opaque in several places, relying on `(response.data as any)?.posts` casts rather than typed generics. This defers shape-mismatch errors from compile time to runtime.

QuantMax is the most incomplete app in this phase. Five of its eight pages import hooks (`useFeed`, `useMatching`, `useLive`, `useCreator`, `useSpeedDating`) that contain no React Query or API client usage â€” they remain pure local state with simulated delays. Meanwhile, proper React Query hooks like `useForYouFeed`, `useSwipe`, `useRecommendations`, `useMatches` exist but are unused by any page.

The wire-up for quantai, quantchat, quantmail, quantedits, quantneon, and quantube's primary pages is substantially complete. All `FIXME(phase-23)` markers and `generateMock*` functions have been removed from these apps.

## QuantMax incomplete wire-up

The pattern in quantmax is inconsistent: some hooks were rewritten with React Query (e.g., `useChallenge`, `useGroupRooms`, `useNearby`) while others were left untouched (`useFeed`, `useMatching`, `useLive`, `useCreator`, `useSpeedDating`, `useVideoChat`). The pages importing the untouched hooks display `LoadingState`/`ErrorState`/`EmptyState` from shared-ui, which makes them look correct at first glance, but the state driving those conditions is never populated by a real network call.

The `useFeed` hook's `loadMore` function contains:

```ts
setTimeout(() => {
  setState((prev) => ({
    ...prev,
    isLoading: false,
    // In production, this would fetch from API
  }));
}, 1000);
```

The feed starts empty, shows a loading state briefly, then renders "No videos yet" because no data ever arrives.

The matching page has a more direct bug. It calls `matching.swipe('like')` expecting a unified `swipe(action)` method, but `useMatching` returns separate `like(profileId)`, `pass(profileId)`, `superLike(profileId)` methods â€” each requiring a profile ID argument. This will throw `matching.swipe is not a function` at runtime.

## Query key design note

`useNearby` in quantmax includes the entire `filters` object in the query key, which means any filter change creates a new cache entry. Switching back to a previous filter set re-fetches rather than using cache. This could cause unnecessary network requests if the filter UI allows rapid toggling between presets.

## QuantTube `shows.tsx` retained mock data

The diff shows `MOCK_SHOWS` renamed to `SAMPLE_SHOWS` and reformatted, but the data is still hardcoded constants loaded via `setShows(SAMPLE_SHOWS)` in a `useEffect`. No React Query hook is used. The `useTrending` hook exists and uses the API client, but `shows.tsx` doesn't import it or any other hook.

## Defensive response shape handling

Pages like `quantmail/page.tsx` handle multiple possible response shapes defensively:

```tsx
{
  typeof email.from === 'string' ? email.from : email.from?.name || email.from?.email;
}
```

And `quantchat/chat/[id]/page.tsx`:

```tsx
message={msg.message ?? msg.content ?? ''}
sender={(msg.sender ?? msg.role ?? 'other') as 'self' | 'other'}
```

These fallback chains handle backend schema uncertainty but hide breakage at runtime rather than surfacing it at compile time. Adding proper response type definitions in each app's `types/` directory would let TypeScript catch shape mismatches before deployment.
