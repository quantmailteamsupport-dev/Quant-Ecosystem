# Streaming TTS with ElevenLabs, Kokoro, prefetch buffer, and adaptive VAD (v2)

This change adds a streaming TTS subsystem to the quant-live voice pipeline: two providers (ElevenLabs WebSocket, Kokoro HTTP streaming), a prefetch buffer for sub-300ms first-byte latency, an adaptive VAD with noise floor calibration via EMA smoothing, and pipeline integration with latency tracking and interruption handling. All 90 tests pass across 11 files. The v1 issues â€” `firstChunk = true` bug, missing EMA smoothing, and missing `synthesizeResponse` test â€” have been fixed.

**Watch for:** The prefetch buffer does not propagate cancellation to the underlying source iterator, relying on the pipeline's `for await` loop to trigger cleanup â€” this is safe in the current interruption path but fragile if `cancel()` is called outside active iteration (likely). The ElevenLabs provider sends `xi_api_key` in the WebSocket message body, which matches their API docs but exposes the key in frame-level logging (confirmed). The ElevenLabs polling loop adds up to 10ms artificial latency per chunk via `setTimeout(r, 10)` busy-wait (confirmed).

## High-level view

The three v1 bugs are resolved cleanly. `synthesizeResponse` now sets `firstChunk = false` after recording the measurement, and the new pipeline test verifies that TTS latency is tracked with exactly one sample and all chunks reach the `audioOut` callbacks. The adaptive VAD applies exponential moving average using `noiseFloorSmoothing` as the alpha coefficient, weighting recent calibration frames more heavily than early ones.

The prefetch buffer lifecycle concern from v1 persists as a design-level issue rather than a bug. In the current pipeline, `handleInterruption` cancels the buffer and calls `provider.stop()`, and the `for await` loop's subsequent iteration triggers `return()` on the generator, which propagates to the source iterator. The gap is narrow: if `cancel()` is invoked when the pipeline is not actively iterating (e.g., between `synthesizeResponse` calls or after it has already returned), the source iterator is never closed.

## Prefetch buffer iterator lifecycle

The buffer's async generator obtains an iterator from the source and drives it with manual `next()` calls:

```typescript
const iterator = this.source[Symbol.asyncIterator]();

for (let i = 0; i < this.bufferSize; i++) {
  if (this.cancelled) return;
  const { done, value } = await iterator.next();
  if (done) break;
  buffer.push(value);
}
```

When the generator function returns (via `return` statement from `if (this.cancelled) return`), the runtime does not call `iterator.return()` on the manually-obtained source iterator. The `for await` loop in `synthesizeResponse` calls `return()` on the _outer_ generator (the PrefetchBuffer), but that outer generator's implicit `return()` handler does not propagate to the inner iterator.

In the current pipeline, `handleInterruption` always calls `provider.stop()` alongside `buffer.cancel()`, so network resources get cleaned up through the provider's own abort mechanism. The risk is that a future caller uses `PrefetchBuffer` standalone without also calling `provider.stop()`, leaving the source dangling. Fix: hoist `iterator` to a class field and call `this.iterator?.return?.()` inside `cancel()`.

## ElevenLabs polling loop

The current bridge from push-based WebSocket to pull-based async generator:

```typescript
while (!done && !this.aborted) {
  if (chunks.length > 0) {
    const chunk = chunks.shift()!;
    yield chunk;
  } else {
    await new Promise((r) => setTimeout(r, 10));
  }
  if (error) throw error;
}
```

Every 10ms the event loop wakes, checks the array, and goes back to sleep. A zero-overhead alternative:

```typescript
let resolve: (() => void) | null = null;
const signal = () =>
  new Promise<void>((r) => {
    resolve = r;
  });

ws.addEventListener('message', (event) => {
  // ... parse and push to chunks ...
  resolve?.();
});

while (!done && !this.aborted) {
  if (chunks.length > 0) {
    yield chunks.shift()!;
  } else {
    await signal();
  }
}
```
