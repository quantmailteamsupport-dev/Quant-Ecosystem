# Foundation polish: reduced-motion guards, auth centralization, real AI streaming, QuantSync App Router

Four independent features across the Quant monorepo: (1) adding `prefers-reduced-motion` guards to 11 shared-ui components that relied on CSS animation classes unconditionally, (2) extracting QuantChat's scattered `localStorage.getItem('token')` into a centralized `lib/auth.ts`, (3) replacing QuantAI's fake `setInterval`-based token streaming with real SSE/ReadableStream + AbortController cancel, and (4) bootstrapping Next.js 15 App Router into QuantSync (previously bare React + Fastify). The legacy pages in QuantSync were renamed to `legacy-pages/` to avoid Next.js Pages Router conflicts.

**Watch for:** Hardcoded `ws://localhost:3001` in the new auth utility leaks into production builds (confirmed). Auth token passed as a query parameter in WebSocket URLs is logged in server access logs (confirmed). The AI streaming hook in QuantAI still uses raw `localStorage.getItem('token')` directly, bypassing the centralization pattern established in FEAT-002 (confirmed). The `[DONE]` sentinel in SSE parsing breaks out of the inner loop but not the outer read loop, causing the reader to continue reading after the stream is logically complete (confirmed).

## High-level view

The reduced-motion fix applies a consistent pattern across shared-ui: import `useReducedMotion` from framer-motion, conditionally strip animation class names. Every animated component becomes a client component, though they all already were.

The auth centralization in QuantChat consolidates 15+ raw `localStorage.getItem('token')` call sites into four exported utilities (`getAuthToken`, `getAuthHeaders`, `getAuthHeadersWithContent`, `getWsAuthUrl`). The consolidation succeeds at the stated goal but introduces a hardcoded `ws://localhost:3001` WebSocket URL and passes the auth token as a URL query parameter rather than via a protocol header or first-message auth handshake.

The AI streaming rewrite replaces the simulation with real `fetch` + `ReadableStream` reader + SSE line parsing. It handles both `text/event-stream` and regular JSON fallback, and wires AbortController for user cancel. The SSE parser has a control-flow gap where `[DONE]` only breaks the inner `for` loop, and the hook still uses inline `localStorage` access for its token.

The QuantSync App Router migration adds Next.js 15, a root layout with `QueryProvider` + `AppProviders` (theme, command palette), a feed page with loading/error/empty states using shared-ui components, and renames the old `src/pages/` to `src/legacy-pages/`. The legacy files also received substantial code expansions bundled into this PR, inflating the diff beyond the App Router scaffolding.

<summary>Issues (6)</summary>

1. **Hardcoded localhost in WebSocket URL** — `getWsAuthUrl` returns `ws://localhost:3001/...` which will fail in any non-local deployment. Use an environment variable or `window.location.host` derivation.
2. **Token in URL query param** — `getWsAuthUrl` passes the auth token as `?token=...`, which appears in server access logs, browser history, and referrer headers. Use a protocol-level auth message or subprotocol header instead.
3. **SSE `[DONE]` break is scoped to inner loop** — In `processSSEStream`, when `data === '[DONE]'` the `break` exits the `for (const line of lines)` loop but not the outer `while (true)` reader loop. The stream will keep reading until the server closes, potentially swallowing a final buffered line or causing a wasted read cycle.
4. **QuantAI auth not centralized** — `useAIChat.ts` line 240 still does `localStorage.getItem('token')` directly, inconsistent with the centralization done in FEAT-002 for QuantChat. If these apps should share a pattern, this is a gap.
5. **QuantSync legacy-pages code churn bundled** — The diff includes ~1,800 lines of modifications to legacy page files (analytics, communities, search, spaces, messages, etc.) that are unrelated to the Next.js migration. These should ideally be separate changes for reviewability.
6. **React version bump from 18 to 19** — QuantSync jumped from React 18.3.1 to React 19.2.6 as part of the Next.js 15 addition. This is a major version bump that could have runtime behavior differences with existing code in `legacy-pages/`.

## Hardcoded WebSocket URL and token-in-query-param

The newly created `apps/quantchat/src/lib/auth.ts` exports `getWsAuthUrl` which constructs:

```ts
export function getWsAuthUrl(conversationId: string): string {
  const token = getAuthToken() || '';
  return `ws://localhost:3001/ws/chat?conversationId=${conversationId}&token=${token}`;
}
```

Two separate problems. First, `localhost:3001` is hardcoded with no environment variable override. The other auth utilities (`getAuthHeaders`, `getAuthHeadersWithContent`) correctly produce transport-agnostic headers, but the WebSocket helper bakes in the host. In production, this will attempt to connect to `ws://localhost:3001` on the user's machine.

Second, the auth token appears as a URL query parameter. WebSocket doesn't support custom headers in the browser's `new WebSocket()` call, which is the underlying constraint. The common mitigation is to send the token as the first message after connection (`ws.send(JSON.stringify({ type: 'auth', token }))`), or use the `Sec-WebSocket-Protocol` header as a token carrier. The query-param approach means the token will appear in HTTP access logs if there's a reverse proxy in front of the WebSocket server, and potentially in CDN or load balancer logs.

## SSE parser control flow gap

In `processSSEStream`:

```ts
for (const line of lines) {
  if (line.startsWith('data: ')) {
    const data = line.slice(6);
    if (data === '[DONE]') break; // <-- breaks the for loop only
    // ...parse token...
  }
}
```

When the server sends `data: [DONE]`, the `break` exits the `for` loop over buffered lines but does not signal the outer `while (true)` loop to stop. The next iteration calls `reader.read()` again. In practice, the server likely closes the stream immediately after `[DONE]` so the next read returns `{ done: true }`, making this a single wasted cycle rather than a hang. But if the server sends additional data after `[DONE]` (e.g., a trailing newline or metadata), that data would be silently processed.

Fix: set a `streamDone` flag when `[DONE]` is encountered and check it in the `while` condition.

## Auth inconsistency between QuantChat and QuantAI

FEAT-002 invested effort in removing all direct `localStorage.getItem('token')` calls from QuantChat in favor of `lib/auth.ts`. FEAT-003, in the same PR, introduces a new direct `localStorage.getItem('token')` call in `useAIChat.ts` (line 240). The inconsistency isn't a bug — QuantAI doesn't import from QuantChat — but it establishes divergent patterns in the same PR. If the goal is centralized token access, QuantAI should either import from a shared package or create its own `lib/auth.ts` following the same pattern.

## Unrelated code churn in legacy-pages

The QuantSync migration diff shows ~1,800 lines of modifications in files that were renamed from `pages/` to `legacy-pages/`. Files like `analytics.tsx`, `communities.tsx`, `search.tsx`, `spaces.tsx`, `messages.tsx`, and `settings.tsx` received substantial feature expansions (new interfaces, new fetch calls, new UI sections) as part of the rename. These modifications are unrelated to the Next.js App Router bootstrap and make the diff harder to review by conflating "moved files" with "added new functionality to moved files."

## React 18 to 19 version jump in QuantSync

The `package.json` changes show QuantSync's `react` dependency moved from `^18.3.0` to `^19.0.0` (resolved to 19.2.6). React 19 includes breaking changes: `forwardRef` deprecation path, new `use()` hook, and changes to context behavior. The existing code in `legacy-pages/` was written for React 18. This major version bump warrants verification of the existing pages under the new runtime, especially components using refs or context in non-standard ways.
