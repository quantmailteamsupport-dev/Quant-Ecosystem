# Runtime integration layer and unified product shell

This change splits into two distinct efforts shipped together. Phase 4 establishes the local developer experience: root-level orchestration scripts (`doctor`, `env:check`, `db:reset`, `smoke`), a reusable Fastify health-server package, a typed service-discovery registry, and health-check stubs for 11 services that previously had no runnable entry point. Phase 5 adds the product shell to `shared-ui`: a GlobalNav, CommandMenu (Cmd+K), AppSwitcher, NotificationCenter, WorkspaceSwitcher, UserMenu, AIDock, AISidePanel, SharingModal, plus state components, route/auth/onboarding guards, onboarding flow steps, and expanded design tokens (density, elevation, breakpoints, motion, accessibility).

Watch for:

- Guards are purely presentational and do not enforce access server-side; a consumer that renders `AuthGuard` without corresponding server checks has no security boundary. (confirmed)
- Seed data writes a plaintext password-hash placeholder (`demo-password-hash-not-for-production`) that could reach a deployed environment if the seed script runs outside local dev. (confirmed)
- Stub services all listen on arbitrary `HEALTH_PORT` defaults in the 3030-3040 range that disagree with the service-discovery registry defaults (3001-3025). (confirmed)

## High-level view

The developer-experience scripts (`doctor`, `smoke`, `env-check`, `db-reset`) give a new contributor a single command to validate their setup and reset state. They shell out to pnpm/docker via `tsx`, and the smoke script swallows all stderr, making failures hard to diagnose.

The health-server package is a thin Fastify wrapper exposing `/healthz` and `/readyz`. It runs readiness checks sequentially, so a single slow external dependency blocks the entire probe.

Service discovery maps service names to default ports with env-var overrides. The registry and the stub services disagree on port assignments: the registry maps `quantube` to 3005 but `tube-api/src/main.ts` defaults to 3039. This discrepancy exists for all stub services and will produce routing failures once real orchestration is wired up.

The shell components are presentation-only React with good accessibility markup (aria labels, roles, keyboard navigation in CommandMenu). The guards render fallback UI rather than redirecting or throwing, so without server-side enforcement there is no actual security boundary.

The onboarding flow collects an "AI memory consent" flag and workspace invite emails but has no validation logic and no backend integration.

Design tokens add density, elevation, breakpoints, accessibility, and motion constants. The accessibility tokens store Tailwind class strings rather than CSS values, making them unusable outside a Tailwind context.

## Health-port conflict between service-discovery and stub services

The service-discovery registry defines each service's default port:

```ts
quantube: { defaultPort: 3005, envVar: 'QUANTUBE_PORT' },
quantads: { defaultPort: 3006, envVar: 'QUANTADS_PORT' },
```

But the stub services pick their own HEALTH_PORT defaults independently:

```ts
// tube-api/src/main.ts
const port = Number(process.env['HEALTH_PORT'] ?? '3039');
// ads-api/src/main.ts
const port = Number(process.env['HEALTH_PORT'] ?? '3030');
```

When a consumer calls `getServiceUrl('quantube')` it gets `http://localhost:3005`, but `tube-api` listens on 3039. Either the stubs should import from `@quant/service-discovery` and use the canonical port, or the registry should be updated to match.

## Sequential health-check execution

`startHealthServer` runs readiness checks in a `for..of` loop:

```ts
for (const [name, check] of Object.entries(checks)) {
  try {
    const result = await check();
    // ...
  }
}
```

If `db` takes 3 seconds and `redis` takes 2 seconds, the probe takes 5 seconds total. For Kubernetes liveness/readiness probes with tight timeouts this can cause cascading restarts. Running independent checks concurrently with `Promise.allSettled` would halve the window.

## Seed data security posture

The seed script writes `passwordHash: 'demo-password-hash-not-for-production'`. If this seed accidentally runs against a deployed database (e.g., someone sets `DATABASE_URL` to a staging instance), it creates accounts with a known non-hashed string. The string isn't an actual bcrypt output, so login would presumably fail, but it still pollutes real data. An environment guard (`if (process.env.NODE_ENV === 'production') throw`) or actually hashing a known dev password would be safer.

## Guards as presentation boundaries

All three guards check a boolean prop, render children or fallback UI. None redirect, none interact with a token store, none call into a server:

```tsx
export const AuthGuard: React.FC<AuthGuardProps> = ({ isAuthenticated, loginPath, children }) => {
  if (!isAuthenticated) {
    return; /* fallback UI */
  }
  return <>{children}</>;
};
```

The protected `children` are still in the bundle and render tree — a user can bypass the guard by manipulating React state or inspecting the DOM. This should be documented as "UI hint, not security enforcement" or augmented with a redirect/throw mechanism.

## CommandMenu activeIndex drift

`activeIndex` can desync from the filtered list if the user backspaces (expanding results after the index pointed past the new list boundary). The `setActiveIndex(0)` reset in `onChange` covers normal typing but not list expansion via deletion, which could produce an out-of-bounds highlight.

## Missing docker-compose.dev.yml

`package.json` adds:

```json
"dev:infra": "docker compose -f docker-compose.dev.yml up -d"
```

No `docker-compose.dev.yml` file exists in the diff. The `doctor` script doesn't check for this file either. Running `pnpm dev:infra` will fail immediately for any new developer.

## Test coverage gaps

The 32 shell-component tests cover rendering, basic interactions, filtering (CommandMenu), and conditional rendering (guards). Not tested: AISidePanel message sending and Enter-key handling, SharingModal clipboard interaction, WorkspaceSwitcher creation flow, StarredItems/RecentItems callbacks, OnboardingGuard behavior, keyboard navigation in CommandMenu (arrow keys, Enter-to-execute), and the health-server/service-discovery tests don't verify port alignment between the two packages.
