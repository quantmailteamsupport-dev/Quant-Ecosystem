# Universal Search: Hybrid BM25 + Vector with Cross-App Pipeline (v2)

The v1 review flagged seven issues. Six have been addressed: Qdrant auth/TLS support is now wired through VectorClientOptions, the embedding placeholder emits a startup warning, isAdmin passes through from the request schema, over-fetching (3x limit) compensates for permission filtering, MeiliSearch `_rankingScore` is used when available for fusion scoring, and Kafka retry with exponential backoff and a dead-letter queue handles transient failures. The pre-query filtering issue remains deliberately deferred with a TODO comment explaining the trade-off.

Watch for: The DLQ implementation creates and disconnects a new Kafka producer on every failed event, which will cause connection storms under sustained failures (likely). The `_rankingScore` fix is likely a no-op because the SearchClient call doesn't pass `showRankingScore: true` to MeiliSearch (likely). The `retryWithBackoff` and `deadLetterPublish` functions have no unit tests despite being critical reliability infrastructure (confirmed).

## High-level view

The VectorClient now accepts `VectorClientOptions` with `apiKey` and `https` fields, and attaches the `api-key` header to all requests when configured. The protocol-selection logic in the constructor uses three branches based on string prefix checks; the fallback branch produces a URL without a scheme prefix if the host doesn't start with `http`, which would break `fetch` for bare hostnames.

The hybrid search engine checks for `_rankingScore` on MeiliSearch hits and uses it for min-max normalization when present. However, MeiliSearch only returns `_rankingScore` when the search request sets `showRankingScore: true`, and the engine's call to `SearchClient.search` doesn't pass that option â€” so the fallback to positional-rank normalization will fire on every query unless the SearchClient defaults it.

The Kafka consumer now wraps every event through `retryWithBackoff` (3 attempts, exponential backoff) before publishing to a DLQ topic. Every DLQ publish spins up a fresh Kafka producer, connects, sends, and disconnects. Under sustained downstream failures this creates O(failures) connection churn against the Kafka broker.

## DLQ producer lifecycle creates connection storm risk

```typescript
export async function deadLetterPublish(...): Promise<void> {
  const dlqTopic = `${originalTopic}.dlq`;
  const producer = kafka.producer();
  try {
    await producer.connect();
    await producer.send({ topic: dlqTopic, messages: [...] });
  } catch (dlqError) {
    pinoLogger.error(...);
  } finally {
    await producer.disconnect();
  }
}
```

Under a sustained downstream outage, every incoming Kafka message exhausts retries and hits the DLQ path. With default consumer settings, this can mean hundreds of producer connect/disconnect cycles per second. Kafka brokers rate-limit new connections; exceeding the limit causes `ConnectionError` which is caught and swallowed by the try/catch, silently dropping the DLQ message with only a log line. The event is then lost from both the main pipeline and the DLQ.

A shared, long-lived DLQ producer initialized at startup and reused across all `deadLetterPublish` calls would eliminate this. The `kafka` instance is already available in `main()` scope.

## MeiliSearch score availability depends on request configuration

```typescript
const hasMeiliScores = bm25Hits.length > 0 && typeof bm25Hits[0]._rankingScore === 'number';
```

MeiliSearch only includes `_rankingScore` in response payloads when the search request sets `showRankingScore: true`. The engine calls `this.searchClient.search(opts.index, query, { limit: 50, filter: opts.filter })` â€” no ranking score option. Unless `SearchClient` internally appends this option on all searches, `_rankingScore` will be `undefined` and the improved scoring path is dead code. The engine will permanently use `1 - idx / length` for BM25 normalization, which is exactly the v1 behavior.

Fix: pass `showRankingScore: true` in the search options, or have the SearchClient set it as a default.

## Retry/DLQ logic is untested

`retryWithBackoff` and `deadLetterPublish` are exported from `main.ts` and testable, but `main.test.ts` only covers `buildHandlerMap` and `routeEvent`. Key scenarios without coverage:

- Handler fails twice then succeeds on third attempt (verifies retry count and success path)
- Handler fails all 3 attempts, DLQ `send` is called with correct payload
- DLQ publish itself fails (should log error but not throw, allowing the outer catch to handle)
- Exponential backoff timing (200ms, 400ms between attempts)
