# Wire QuantTube channel/upload to real API, remove @simulated annotations

The channel page now fetches data through `useQuery` + `apiClient.getChannel` instead of rendering hardcoded arrays, and the upload page sends files via XMLHttpRequest with real progress events instead of faking it with `setInterval`. Twelve agent-runtime pilot files had their obsolete `@simulated` JSDoc blocks removed since they already delegate to `aiEngine.infer()`. The upload page's fallback is aggressive: on any failure it silently completes and advances the user to the next step, which hides real errors.

**Watch for:** The channel page casts `response.data` through `as unknown as ChannelApiResponse` to paper over a type mismatch between the API client's declared return shape (`{ channel: Channel }`) and the page's expected shape (`{ channel, videos, playlists, communityPosts }`). This cast will suppress type errors at compile time but will produce undefined fields at runtime unless the backend actually returns the wider shape (confirmed). The upload page swallows publish failures and shows "Published Successfully!" regardless of outcome (confirmed).

## High-level view

The channel page adopts `useQuery` with loading/error/retry states and derives filtered video lists from the response. The integration has one structural gap: the `ChannelApiResponse` interface expects four fields (`channel`, `videos`, `playlists`, `communityPosts`) but the API client only declares `{ channel: Channel }` as the generic type for `getChannel`. The diff bridges this with an `as unknown as` cast, which means videos and playlists will be `undefined` at runtime unless the endpoint delivers them — falling through to fallback arrays only when `error` is truthy, not when the fields are simply absent.

The upload page replaces timer-based progress with a real XHR upload to `/api/videos/upload`. On failure (HTTP error or network error) it calls `handleUploadFallback` which marks the file as "complete" and advances anyway. The publish step calls `apiClient.uploadVideo` (metadata-only, misnamed relative to the file-upload XHR), catches all errors, and moves to the success screen regardless. Users have no signal that their video didn't actually publish.

The annotation cleanup across 12 pilot files is mechanical deletion of a 4-line JSDoc block. No behavioral change.

## Channel page type contract gap

The API client declares `getChannel` as returning `ApiResponse<{ channel: Channel }>`. The channel page defines a local `ChannelApiResponse` with four fields:

```typescript
interface ChannelApiResponse {
  channel: ChannelData;
  videos?: ChannelVideo[];
  playlists?: ChannelPlaylist[];
  communityPosts?: CommunityPost[];
}
```

The diff bridges the mismatch with:

```typescript
return response.data as unknown as ChannelApiResponse;
```

If the backend's `/channels/{id}` endpoint doesn't bundle videos and playlists into the channel response (the more common REST pattern — separate endpoints per resource), the optional fields will be `undefined` at runtime. The fallback logic only engages when `error` is truthy:

```typescript
const videos: ChannelVideo[] = data?.videos ?? (error ? FALLBACK_VIDEOS : []);
```

When the API succeeds but returns only `{ channel }`, videos resolves to `undefined ?? (false ? ... : [])` = `[]`. The user sees an empty video grid with no indication anything went wrong and no fallback content. Fix: widen the API client generic to match the real endpoint shape, or treat missing/empty arrays the same as error-state arrays.

## Upload failure swallowing

The upload page has two failure points that both fail silently toward success:

**File upload (XHR):** Both the `load` handler (non-2xx status) and the `error` handler call `handleUploadFallback`, which marks the file as complete and advances to the details step. The user's file never actually reached the server, but the UI shows a green "complete" badge and moves on.

**Metadata publish:** `handlePublish` wraps the API call in try/catch and advances to the review (success) screen in both paths:

```typescript
} catch (err) {
  // Fallback: still move to review on failure (graceful degradation)
  setCurrentStep('review');
}
```

The `error` state variable is never set in this path — it's only used for a top-level error screen that's unreachable from the publish flow. Showing "Video Published Successfully!" when the publish actually failed is misleading. At minimum, distinguish between success and degraded-fallback states in the review screen.

## XHR upload vs. apiClient endpoint divergence

The file upload constructs its own XHR to:

```typescript
xhr.open('POST', `${window.location.origin}/api/videos/upload`);
```

This targets a Next.js API route (the app's own `/api/` proxy). The subsequent metadata publish calls:

```typescript
apiClient.uploadVideo({ title, description, type, tags })
```

Which resolves to `http://localhost:3005/videos/upload` — the backend service directly. The apiClient attaches a Bearer token if one is set (`this.token`), while the XHR sends no auth headers. These two requests target semantically different operations (binary file upload vs. metadata registration) but share the same endpoint path `/videos/upload`, which suggests the backend might not distinguish them or one of the paths doesn't exist yet.
