# Identity, permissions, and agentic execution pipeline

This change adds two new subsystems to the Quant monorepo: a workspace-scoped identity/permissions package (`@quant/identity-permissions`) with RBAC, resource tracking, context graph, memory management, and consent; and a tiered agent execution pipeline in `@quant/agent-runtime` with typed tools, plan generation, safety classification, cost tracking, and five end-to-end workflows. Together they form the foundation for multi-tenant AI agent governance.

Watch for: the ExecutionEngine auto-approves every approval request in the same synchronous call (confirmed), the PlanGenerator includes all available tools regardless of relevance (confirmed), the safety classifier only inspects the tool name and a flat context object rather than actual arguments passed to the tool (confirmed), and no integration exists between the two packages — the identity-permissions consent/RBAC checks are not wired into the agent execution pipeline (confirmed).

## High-level view

The identity-permissions package provides six in-memory modules (RBAC, ResourceRegistry, ContextGraph, MemoryManager, ConsentManager, WorkspaceAuditLog) that enforce workspace-scoped isolation through Map-based stores. The design is self-contained: each module owns its own state, there are no cross-module dependencies, and no persistence layer exists. The RBAC engine uses a static permission matrix with four roles and six permission levels, and workspace isolation falls out naturally from requiring a workspace ID on every query.

The agent-runtime additions introduce a five-tier action model (ReadOnly through Admin) and a pipeline that sequences permission checks, safety classification, approval, execution, audit logging, undo registration, and cost tracking. The pipeline is well-structured but the approval step is ceremonial — the engine submits and immediately approves in the same call, meaning no external actor ever gates execution. This is the most significant behavioral gap in the runtime.

The five workflows (PlanMyDay, EmailReply, MeetingToTasks, CrossAppSearch, ContentLaunch) are implemented as subclasses of BaseWorkflow. Each declares its own tools inline with hardcoded stub handlers returning canned data. The workflows demonstrate the tier model across tiers 0–3 (no workflow exercises tier 4) and serve as integration tests of the pipeline, but they don't consume real services.

The PlanGenerator claims to match tools to intent via name/category overlap but falls through to `|| true`, making it a pass-through that includes every available tool. Cost estimation is purely tier-based (fixed cost per tier level), not related to actual LLM token usage or external API calls.

The safety classifier operates on action name string matching and a flat context dict. It has five default rules (PII without consent, high-value financial, admin actions, moderation override, bulk actions). The rules check context properties like `piiAccess` and `consent` that must be explicitly passed in by the caller — there is no automatic extraction of these signals from tool arguments or execution context.

<details>
<summary>Issues (6)</summary>

1. **Auto-approval defeats the approval gate** — ExecutionEngine calls `approvalQueue.submit()` then immediately `approvalQueue.approve()` in the same synchronous block. No external human or system ever has the opportunity to reject. Either make the approval asynchronous (return a pending result and resume on callback) or document this as intentionally auto-approved for the current phase.
2. **PlanGenerator relevance matching is a no-op** — The `isRelevant` condition includes `|| true`, so every available tool is always included in every plan. Remove the dead matching logic or implement actual relevance filtering.
3. **Safety classifier requires caller-provided signals** — The classifier checks `context['piiAccess']` and `context['consent']` but the ExecutionEngine passes `step.args` as the context, which are tool arguments, not safety metadata. Unless callers manually inject these keys, the PII and financial rules will never trigger in production.
4. **No integration between identity-permissions and agent-runtime** — The consent manager, RBAC engine, and resource registry are never consulted by the ExecutionEngine. An agent can execute actions on resources without checking workspace membership or AI access toggles.
5. **Tier1_DraftOnly maps to OBSERVE permission** — In `tierToPermissionLevel`, both Tier0 (read-only) and Tier1 (draft creation) map to `PermissionLevel.OBSERVE`. An agent with OBSERVE permission can create drafts, which contradicts the semantic meaning of "observe only."
6. **Unbounded in-memory growth** — All stores (audit logs, cost records, memories, consent prompts, context graph nodes) are in-memory Maps with no eviction, TTL, or size limits. This is acceptable for a foundation phase but should be documented as a known limitation before any load testing.

## ExecutionEngine auto-approves its own requests

The ExecutionEngine's approval step (lines 88–97 of `execution-engine.ts`) submits a request to the ApprovalQueue and then immediately approves it:

```typescript
if (step.requiresApproval) {
  const approvalId = generateId('approval');
  this.approvalQueue.submit({ ... });
  this.approvalQueue.approve(approvalId);
  step.status = 'approved';
}
```

This makes the approval gate entirely ceremonial. The queue is populated and drained in the same tick — no external system or human can intercept. The tests verify this works but don't test rejection or timeout paths. In a real system this means a Tier 3 "create ad campaign" action gets auto-approved with no human in the loop despite the workflow declaring `requiresApproval: true`.

The `ApprovalQueue.approve()` method (pre-existing code) validates that the request exists and is pending, but since submit and approve happen synchronously there's no race condition — just a missing architectural seam for async human approval.

## PlanGenerator includes all tools regardless of relevance

In `plan-generator.ts`, the `isRelevant` check is:

```typescript
const isRelevant =
  intentLower.includes(toolLower) ||
  intentLower.includes(categoryLower) ||
  toolLower.split(/[-_]/).some((word) => word.length > 2 && intentLower.includes(word)) ||
  categoryLower.split(/[-_]/).some((word) => word.length > 2 && intentLower.includes(word)) ||
  true; // Include all available tools in the plan
```

The trailing `|| true` means the preceding conditions are dead code. Every tool passed in `availableTools` ends up in the plan. The workflows work around this by only declaring their own tool names in `buildPlan`, but if the PlanGenerator is used with a shared registry containing many tools, every plan would include all of them.

## Safety classification operates on opaque context

The SafetyClassifier's `classify(action, context)` signature takes the action name and a `Record<string, unknown>` context. The five default rules inspect:

- `context['piiAccess']` and `context['consent']` for the PII rule
- `context['amount']` for the financial rule
- `context['affectedCount']` for the bulk action rule

The ExecutionEngine passes `step.args` (the tool's arguments) as context. The workflows' tool arguments don't include these keys — they pass things like `{ query: "..." }` or `{ topic: "..." }`. So the PII rule would only fire if a caller explicitly constructs `{ piiAccess: true, consent: false }` in the tool arguments. There's no automatic enrichment of context from the identity-permissions consent manager or resource registry.

## Identity-permissions and agent-runtime are islands

The two packages have no code-level integration. The RBAC engine's `hasPermission` / `canAccessResource` methods are never called from the ExecutionEngine. The ConsentManager's `hasConsent` is not checked before agent tool execution. The ResourceRegistry's `isAIAccessEnabled` toggle is never consulted.

The common types (`AgentGrant`, `AppGrant`) define grant structures, but neither package instantiates or validates them at runtime. A future integration layer would need to bridge: before executing a tool that touches a resource, check (1) workspace membership via RBAC, (2) AI access toggle via ResourceRegistry, (3) user consent via ConsentManager.

## Tier-to-permission mapping conflates reading with drafting

`tierToPermissionLevel` maps both `Tier0_ReadOnly` and `Tier1_DraftOnly` to `PermissionLevel.OBSERVE`. This means an agent granted only OBSERVE can create email drafts, save documents, and perform any Tier 1 action. The semantic intent of OBSERVE ("watch but don't touch") conflicts with allowing draft creation. The `SUGGEST` permission level exists but is never used in this mapping.

A possible fix: map Tier1 to `PermissionLevel.SUGGEST` instead.

## Workflow tool handlers are static stubs

All five workflows hardcode their tool handlers as inline arrow functions returning canned responses. The handlers don't accept or use their arguments — `EmailReplyWorkflow.draftReply` ignores the `emailId` parameter, `CrossAppSearchWorkflow.searchEmails` ignores the `query` parameter.

This is expected for a foundation phase (the handlers are placeholders for real integrations), but it means the test suite validates pipeline mechanics, not workflow correctness. No test verifies that the right arguments flow from plan steps to tool handlers.

## Test coverage gaps

The tests cover the pipeline end-to-end (permission denial, safety blocking, audit logging, undo registration, cost tracking) and all five workflows execute successfully. Not tested: rejection flow through the approval queue, budget enforcement blocking execution, tool argument validation via `TypedToolRegistry.validateArgs` during plan execution (the engine never calls `validateArgs`), concurrent plan execution, and error propagation when a mid-plan step fails (the engine continues but marks the plan as failed).

## CSV export in WorkspaceAuditLog lacks escaping

The `exportEvents` method in `workspace-audit.ts` concatenates fields with commas:

```typescript
`${e.id},${e.workspaceId},${e.actorId},${e.actorType},${e.action},${e.resourceId ?? ''},${e.timestamp}`;
```

If any field contains a comma (e.g., an action string like "create,update"), the CSV output will be malformed. The `metadata` field is also omitted from the CSV export entirely.
