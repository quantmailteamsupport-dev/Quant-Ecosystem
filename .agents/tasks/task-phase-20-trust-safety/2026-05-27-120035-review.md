# Real Trust & Safety layer (Phase 20)

Phase 20 replaces stubs with functional trust & safety infrastructure across five domains: CSAM hash matching with fail-closed semantics, perceptual hashing and bloom filters for cross-app dedup, video/live-stream moderation with tiered keyframe extraction, account integrity (phone, email, device fingerprint, behavioral signals, age gating), and an appeals queue with SLA enforcement plus safety microfeatures. The approach is provider-adapter DI throughout — every external dependency (PhotoDNA, Whisper, Twilio, MSG91, ffmpeg) is behind an interface with a mock/test implementation so the pipeline can run without credentials. A DPIA document accompanies the code.

**Watch for:** Verification code generated with `Math.random()` instead of CSPRNG (confirmed — phone verification). Verification state and phone codes live purely in-memory maps that won't survive a restart or scale across instances (confirmed). The CSAM timeout rejects with an error but the error propagates to the caller unbounded — if the caller doesn't treat this rejection as "block the upload," the fail-closed contract is violated at integration level (likely). Live moderator continues processing violations past the disconnect threshold to reach the ban threshold — meaning a disconnected stream can still accumulate events (confirmed, by design per FEAT-003).

## High-level view

The CSAM detection pipeline replaces the old CSAMGuard stub with a CSAMMatchService that wraps pluggable providers (PhotoDNA, TestHashProvider, NullProvider). NullProvider throws on every call, enforcing the fail-closed posture that no media is accepted without explicit provider configuration. Timeout enforcement uses a Promise.race pattern capped at 500ms, and on timeout the error propagates up — meaning the caller must catch-and-block rather than the service itself returning `{matched: true}` on timeout.

Video moderation now uses a real KeyframeExtractor with tiered intervals (1/s for 30s, then 1/5s) via a DI backend interface instead of the old simulated frame URL generation. The moderation worker wires this up with a MockFrameExtractorBackend in production startup code, which means production keyframe extraction is currently no-op until a real ffmpeg backend is swapped in.

Account integrity is built as a set of independent services: PhoneVerificationService (geographic SMS routing with rate limiting), EmailVerificationService (disposable blocklist + MX validation + signed tokens), DeviceFingerprintService (SHA-256 signal hashing for sybil detection), BehavioralSignalAnalyzer (risk scoring from account age, typing speed, mouse entropy), and AgeGateService (DOB-based restriction with step-up verification). All use in-memory Maps for state, which is appropriate for the library layer but requires external persistence for production deployment.

The appeals queue adds SLA deadlines to AppealRecords (1h/4h/24h/72h by priority tier), a priority-ordered dequeue, overdue detection, and escalation. Safety microfeatures provide one-tap actions with escalation tracking, distress pattern detection, mass-unfollow protection, self-harm search redirection with localized crisis hotlines for 25+ countries, and a deterministic grooming pattern detector using keyword rules.

<summary>Issues (7)</summary>

1. **Phone code uses Math.random()** — `generateCode()` produces OTP codes with `Math.random()`, which is not cryptographically secure. Replace with `crypto.randomInt()` or `crypto.getRandomValues()`.
2. **In-memory verification state not production-viable** — Phone codes, rate limits, cooldowns, email verified-set, and device fingerprint maps all live in class-level Maps. Document the persistence integration point or inject a store interface.
3. **CSAM timeout error propagation** — On timeout, `withTimeout` rejects with an error rather than returning `{matched: true}`. The fail-closed contract depends entirely on the caller treating the thrown error as a block signal. The image handler in the worker does not wrap this call in a catch-and-block — an uncaught timeout error may crash the job rather than explicitly rejecting the upload.
4. **MockFrameExtractorBackend in production wiring** — `createHandlerDeps()` in `services/moderation-worker/src/main.ts` instantiates the video handler with `MockFrameExtractorBackend`, meaning production video moderation extracts no real frames. This should be gated behind a feature flag or env-var check that fails loudly.
5. **Grooming detector single-match-per-category** — `analyze()` only checks whether _any_ keyword in a category matches (`Array.some`) and adds the category weight once. Conversations with many matches across multiple keywords in the same category score the same as one match, reducing sensitivity for high-indicator conversations.
6. **SMS providers are stub implementations** — Both `TwilioSMSProvider.send()` and `MSG91SMSProvider.send()` void their parameters and return success without making network calls. The geographic routing logic is tested against provider names but never exercises real delivery, which is fine for library code but should be flagged for integration testing.
7. **TextClassifier.mapToScoreCard relies on mutable lastResponse** — The scorecard method uses a `lastResponse` field set as a side effect of `classify()`. Concurrent calls to `classify()` from different contexts will race on this field. Accept the response as a parameter (which the method already supports) and deprecate the implicit path.

## CSAM match service and fail-closed timeout semantics

CSAMMatchService wraps a CSAMHashProvider and enforces a configurable timeout (default 500ms). The `withTimeout` implementation races the provider promise against a setTimeout rejection:

```typescript
private async withTimeout<T>(promise: Promise<T>, ms: number): Promise<T> {
  return new Promise<T>((resolve, reject) => {
    const timer = setTimeout(() => {
      reject(new Error(`CSAM hash check timed out after ${ms}ms. Upload blocked (fail-closed).`));
    }, ms);
    promise.then(resolve, reject).finally(() => clearTimeout(timer));
  });
}
```

The error message says "upload blocked" but the service doesn't return `{matched: true}` on timeout — it throws. The image handler in the moderation-worker calls `this.csamMatchService.checkHash(imageHash)` without a try/catch around the timeout case. If the BullMQ job processor catches the error and retries, the upload might eventually succeed after the timeout window passes. The fail-closed contract is architecturally correct (NullProvider always throws, timeout throws, unknown errors throw) but requires the job processor to treat _any_ exception from the CSAM path as a hard block rather than a retryable error. (confirmed — the image handler has no catch around checkHash)

## Phone verification code generation with Math.random()

```typescript
private generateCode(): string {
  const digits = '0123456789';
  let code = '';
  for (let i = 0; i < this.config.codeLength; i++) {
    code += digits[Math.floor(Math.random() * digits.length)];
  }
  return code;
}
```

For a 6-digit verification code, `Math.random()` reduces the effective entropy below what an attacker needs to brute-force within the 3-attempt limit and 5-minute expiry window. In practice the attempt limit (3 tries) mitigates online brute-force, but the code is predictable if an attacker can observe the PRNG state. `crypto.randomInt(0, 10)` per digit is the standard fix. (confirmed)

## MockFrameExtractorBackend wired into production handler

In `services/moderation-worker/src/main.ts`:

```typescript
const videoHandler = new VideoModerationHandler({
  imageClassifier,
  keyframeExtractor: new KeyframeExtractor(new MockFrameExtractorBackend()),
  policyEngine,
  actionExecutor,
});
```

The mock backend returns synthetic `Buffer.from(\`frame*at*${ts}\`)` strings as frame data, which will be base64-encoded and sent to the image classifier for classification. The classifier will run against garbage data and produce meaningless scores. This is effectively dead video moderation until an ffmpeg backend is substituted. The old frame-URL simulation was also dead, so this isn't a regression — but the code moved from _obviously fake_ (URL-based) to _subtly fake_ (buffers with text content). (confirmed)

## In-memory state across verification services

PhoneVerificationService stores pending codes, rate limits, and cooldowns in `Map<string, ...>` instance fields. EmailVerificationService stores `verifiedEmails` in a `Set<string>`. DeviceFingerprintService stores `deviceToUsers` and `userToDevices` maps. None of these have constructor parameters for injecting an external store.

This means:

- A pod restart loses all pending verifications (users must re-request codes)
- Horizontal scaling means rate limits are per-instance, not global
- The sybil detection fingerprint mapping is local to one process

The DI pattern used everywhere else (providers, backends) should extend to a `StateStore` interface for these maps. (confirmed — no store interface exists)

## Grooming detector sensitivity ceiling

The grooming detector's `analyze()` method iterates each pattern category and uses `.some()` to check if _any_ keyword matches:

```typescript
for (const pattern of ALL_PATTERNS) {
  const matched = pattern.keywords.some((kw) => combinedText.includes(kw));
  if (matched) {
    matchedIndicators.push(pattern.description);
    totalRisk += pattern.weight;
  }
}
```

Maximum possible risk from keyword categories alone: 0.2 + 0.25 + 0.2 + 0.15 + 0.2 = 1.0. But a conversation where every single keyword in a category appears scores identically to one with a single match per category. A clear grooming conversation with 10+ isolation phrases scores the same (0.25) as one with a single "don't tell anyone." The structural bonus (+0.1 for 3+ categories, +0.1 for persistent contact) partially compensates, but per-category sensitivity is flat. (confirmed)

## Age gate restriction lookup — data-redundancy coupling

The `getRestrictedFeatures` method checks `restrictionIndex >= groupIndex`, where `AGE_GROUP_ORDER = ['under13', 'under16', 'under18', 'adult']`. For an `under18` user (index 2), only restrictions with `minAgeGroup` of `'under18'` or `'adult'` apply — it does _not_ inherit restrictions defined solely for `'under13'` or `'under16'`. The net behavior is currently correct only because `RESTRICTED_FEATURES` duplicates every entry across all applicable age groups (e.g., `random_chat` appears for under13, under16, and under18 separately). If someone adds a restriction only to `under13` expecting it to implicitly apply to older minors, it won't. The logic inverts the intuitive inheritance direction and relies on exhaustive data duplication. (confirmed)
