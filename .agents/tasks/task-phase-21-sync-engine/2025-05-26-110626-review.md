# Offline-First CRDT Sync Engine

New `@quant/sync-engine` package providing Yjs-backed collaborative data structures, a WebSocket-with-HTTP-fallback sync protocol, offline request queuing via a service worker abstraction, and a state-machine-driven sync status tracker. The package is cleanly decomposed into six modules with interface-driven persistence and transport layers, making each concern testable in isolation. The naming references "Automerge" but the implementation is built entirely on Yjs, which may confuse consumers of the public API.

**Watch for:** unbounded message queue growth when offline or reconnecting (confirmed); missing `await` on `save()` for `Uint8Array` data in `LocalStore` (confirmed); silent message drops with no error propagation in `SyncProtocol.onmessage` (confirmed); HTTP fallback polling is a no-op placeholder with no actual data path (confirmed).

## High-level view

The CRDT layer wraps Yjs `Doc`, `Array`, and `Map` types behind three classes (`CRDTDocument`, `CRDTList`, `AutomergeState`). Each creates its own `Y.Doc`, so collaborative merging between them requires explicit `encodeState` / `applyUpdate` calls rather than shared-doc awareness. This means the integration point — where a document's updates flow into the sync protocol and back — is left to the consumer to wire.

`SyncProtocol` manages WebSocket connectivity with exponential-backoff reconnection and falls back to HTTP after retries are exhausted. The fallback is structurally present but functionally hollow: the poll timer fires but never fetches data, and there's no mechanism for the HTTP path to deliver inbound messages to handlers. Messages sent while disconnected or reconnecting accumulate in an unbounded in-memory queue.

`LocalStore` provides key-value persistence behind an `IStorageBackend` interface (only an in-memory implementation ships). It handles binary and JSON payloads and maintains a separate offline action queue. A missing `await` on one code path means Uint8Array writes can fail silently.

`ServiceWorkerManager` is a request-queuing layer decoupled from actual browser service worker lifecycle. It queues serialized HTTP requests and replays them through a caller-supplied sender function, retaining failed requests for later retry. There is no size cap on the queue.

The sync status state machine enforces valid transitions between `synced`, `syncing`, `offline`, and `conflict`. The transition graph prevents illegal jumps but nothing in the package actually drives these transitions — the `SyncProtocol` never calls into `SyncStatusManager`, so the two must be coordinated externally.

## Unbounded queues across two layers

`SyncProtocol.send()` pushes messages onto `this.messageQueue` whenever the connection is not in a state that can immediately deliver. There is no cap, no eviction policy, and no back-pressure signal to the caller. A client that continues generating sync updates while offline will grow this queue without limit. The same issue exists in `ServiceWorkerManager.queueRequest()` — requests accumulate in an array with no maximum. Both should either cap the queue and drop the oldest entries, or expose a mechanism for the caller to detect saturation.

```typescript
// sync-protocol.ts, send():
} else {
  this.messageQueue.push(validated);  // unbounded
}
```

## Missing await in LocalStore.save

Line 58 of `local-store.ts`:

```typescript
if (data instanceof Uint8Array) {
  this.backend.set(storeKey, data); // <-- no await
} else {
  await this.backend.set(storeKey, JSON.stringify(data));
}
```

The JSON branch correctly awaits the backend write, but the Uint8Array branch fires and forgets. If the backend rejects (e.g., IndexedDB quota exceeded), the error is silently lost. This is straightforward to fix — add `await`.

## Silent error swallowing in SyncProtocol message handling

When an inbound WebSocket message fails JSON parsing or Zod validation, the handler catches the error and does nothing:

```typescript
} catch {
  // Invalid message, ignore
}
```

In a sync engine, silently dropping messages makes debugging extremely difficult. A malformed message from the server could indicate a protocol version mismatch, a serialization bug, or a truncated frame — all scenarios where the client should at least log or emit a diagnostic event. Consider adding an `onError` handler set or a counter that consumers can observe.

## HTTP fallback is structurally present but functionally empty

`startHttpPolling` sets up a recurring timer, but the timer callback does nothing except schedule the next tick. There is no fetch, no call to `messageHandlers`, and no mechanism for the HTTP path to deliver inbound messages. The outbound path (`void this.httpSender(validated)`) works only if the consumer provides a sender function, but `httpSender` is never set via the public API during fallback activation — it requires the consumer to call `setHttpSender()` before the fallback triggers. The overall result: the HTTP fallback is a skeleton that won't deliver data in either direction without external wiring that isn't documented.

## Naming: AutomergeState is Yjs underneath

The class is called `AutomergeState` and the file is `automerge-state.ts`, but the implementation imports only Yjs and uses `Y.Doc` + `Y.Map`. Automerge is a distinct CRDT library with different semantics and wire formats. Consumers reading the public API will reasonably expect Automerge-style documents or at least Automerge compatibility. This should be renamed to something like `CRDTState` or `SyncedState` to match reality.

## SyncProtocol ↔ SyncStatusManager integration gap

`SyncStatusManager` defines a clean state machine (`synced → syncing → offline → conflict`), but `SyncProtocol` — the class that actually knows about connection and disconnection events — never references it. The two live in separate modules with no coupling. Consumers must manually subscribe to WebSocket lifecycle events and call `statusManager.transition()` at the right times, but the protocol class doesn't expose lifecycle events beyond `getConnectionState()`. A `onConnectionStateChange` callback or event emitter on `SyncProtocol` would make this integration achievable without polling.

## Test coverage gaps

Tests cover each module in isolation with good breadth on the happy path and basic error cases. Notably missing:

- No integration test wiring `SyncProtocol` updates into a `CRDTDocument` and back, which is the primary use case of the package.
- No test for the reconnection/backoff path beyond the mock-based protocol test (actual timer behavior with `vi.useFakeTimers` is used, but the HTTP fallback activation after max retries is tested only for state, not for message delivery).
- No test for `LocalStore` behavior when the backend throws (testing the missing-await issue).
- `ServiceWorkerManager` max-queue behavior is untested because no max exists.
