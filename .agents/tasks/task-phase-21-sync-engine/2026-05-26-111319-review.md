# Offline-First CRDT Sync Engine (v2)

All seven issues from the v1 review have been addressed. The `await` bug in `LocalStore.save` is fixed, both `SyncProtocol` and `ServiceWorkerManager` enforce queue caps with oldest-entry eviction, the class formerly called `AutomergeState` is now `CRDTState`, error handlers and an error counter exist on `SyncProtocol`, the HTTP fallback is documented as an intentional placeholder, and `onConnectionStateChange` provides the hook consumers need to drive `SyncStatusManager` transitions.

**Watch for:** `flushQueue` silently drops messages when `httpSender` is null during fallback activation (confirmed); no tests exercise the newly-added queue eviction or error-handler plumbing (confirmed); `CRDTState.set` validation bypasses the schema when the Zod type lacks a `.shape` property (likely).

## High-level view

The queue cap fix in `SyncProtocol` applies correctly on the `send()` path, but `activateHttpFallback()` calls `flushQueue()` immediately after transitioning to `http_fallback` state. If no `httpSender` has been registered (the common case, since the consumer must call `setHttpSender()` proactively), the flush loop's `else if` condition fails for each message and the message is neither re-queued nor delivered — it is simply discarded from the queue. This is a data-loss path for any messages buffered during reconnection attempts.

The new error-handling surface (`onError`, `errorCount`) and queue-cap behaviors are structurally correct but completely untested. The test suite still matches the v1 scope — no test sends an invalid message and asserts the error callback fires, no test overflows the queue and verifies eviction. The fixes exist in production code but can regress silently.

`CRDTState.set` validates field values by reaching into `schema.shape`, but this property only exists on `z.object()` schemas. If a consumer provides a refined or transformed schema (e.g., `z.object({...}).refine(...)`), the `.shape` lookup returns undefined, validation is silently skipped, and invalid data enters the Yjs map. The current tests only use a plain `z.object()`, so this path is not exercised.

## Message loss in flushQueue during HTTP fallback

When WebSocket retries are exhausted, `activateHttpFallback()` calls `flushQueue()`:

```typescript
private activateHttpFallback(): void {
  this.setConnectionState('http_fallback');
  this.flushQueue();
  this.startHttpPolling();
}

private flushQueue(): void {
  while (this.messageQueue.length > 0) {
    const message = this.messageQueue.shift()!;
    if (this.connectionState === 'connected' && this.ws) {
      this.ws.send(JSON.stringify(message));
    } else if (this.connectionState === 'http_fallback' && this.httpSender) {
      void this.httpSender(message);
    }
    // If neither branch matches, message is dropped
  }
}
```

The state is `http_fallback` but `this.httpSender` is null unless the consumer called `setHttpSender()` before connecting. The message is shifted off the queue but neither sent nor put back. Every queued message accumulated during the reconnection phase is lost. The fix is straightforward: if neither branch can deliver, push the message back or don't flush at all when the delivery path is unavailable.

## Missing test coverage for v1 fix behaviors

The production code now has three new behaviors that lack corresponding tests:

- `SyncProtocol.send()` evicts the oldest message when queue exceeds `maxQueueSize` — no test sends `maxQueueSize + 1` messages and asserts the first was dropped.
- `ServiceWorkerManager.queueRequest()` evicts oldest when exceeding `maxQueueSize` — same gap.
- `SyncProtocol` error handler fires on invalid inbound messages — no test calls `triggerMessage` with malformed JSON and asserts the error handler callback.
- `onConnectionStateChange` is never tested despite being the fix for v1 issue #7.

These are all single-assertion tests that would take minutes to write and would prevent regressions on the exact issues v1 identified.

## CRDTState schema validation conditionality

```typescript
set<K extends keyof T>(key: K, value: T[K]): void {
  const schema = this.schema as unknown as { shape?: Record<string, z.ZodType<unknown>> };
  if (schema.shape && key in schema.shape) {
    const fieldSchema = schema.shape[key as string];
    if (fieldSchema) {
      fieldSchema.parse(value);
    }
  }
  this.map.set(key as string, value);
}
```

The `as unknown as { shape? }` cast works for `z.object()` but not for `z.object().refine()`, `z.object().transform()`, `z.intersection()`, or `z.lazy()`. In those cases `schema.shape` is undefined, the `if` block is skipped entirely, and the value is written to the Yjs map without validation. The class signature (`schema: z.ZodType<T>`) accepts any of these forms, so the type system doesn't prevent it. A consumer could pass a refined schema expecting validation and get silent acceptance of invalid data.
