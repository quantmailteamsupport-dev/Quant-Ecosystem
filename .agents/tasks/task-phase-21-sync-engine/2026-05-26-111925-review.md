# Offline-First CRDT Sync Engine (v3)

The three issues from v2 are resolved. `flushQueue` retains messages when no delivery path is available (tested). Queue eviction in both `SyncProtocol` and `ServiceWorkerManager` is tested with explicit oldest-entry assertions. Error callbacks and `errorCount` are tested with malformed inbound messages. `CRDTState.set` falls back to full-object `safeParse` for schemas without `.shape` (refined schemas), and a test exercises this path with a `.refine()` constraint.

**Watch for:** `applySnapshot` bypasses Zod validation entirely, writing directly to the Yjs map without checking field types or refinements (confirmed). The `CRDTState` full-object fallback validation path is lenient during incremental population — it silently accepts invalid values when other fields are unset (possible).

**Verdict**: APPROVED

## High-level view

`CRDTState` now handles refined schemas via a full-object validation fallback, but `applySnapshot` still writes raw values to the Yjs map without any schema check. This is a data corruption vector if called with untrusted or deserialized input, though it parallels `applyUpdate` semantics where the data is assumed pre-validated.

The incremental-build leniency in the `set()` fallback path — filtering `safeParse` errors to only those matching the current key or custom refinements — works for simple single-field constraints but could misattribute errors in schemas with complex cross-field refinements.

## applySnapshot validation gap in CRDTState

```typescript
applySnapshot(data: Partial<T>): void {
  this.doc.transact(() => {
    for (const [key, value] of Object.entries(data)) {
      this.map.set(key, value);
    }
  });
}
```

This method writes each key-value pair directly into the Yjs map without invoking `set()` (which runs validation) or any schema check. A caller passing an object with incorrectly-typed fields — from deserialization errors, API response parsing, or manual construction — would introduce invalid state that subscription callbacks then propagate to consumers.

The fix is either to route each field through `this.set(key, value)` or to validate the full partial against the schema before the transaction.

## CRDTState fallback validation during incremental population

The fallback path in `set()` builds a candidate object from the current snapshot plus the new value, then runs `safeParse`. If the schema has cross-field refinements (e.g., "endDate must be after startDate") and `startDate` hasn't been set yet, the refinement might fail on the missing field rather than on the value being written. The code filters errors to those whose path matches the current key or whose code is `'custom'`. A cross-field refinement with path `[]` and code `'custom'` would incorrectly block a valid write during state initialization — while a refinement with path `['otherField']` would be ignored even if the new value causes it.

This edge case only manifests with cross-field constraints during state-building. Acceptable for v1 with the caveat documented.
