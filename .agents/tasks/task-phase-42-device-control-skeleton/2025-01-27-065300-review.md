# Device control package skeleton with capability interfaces

The `@quant/device-control` package establishes the contract layer for 11 device capabilities (phone, SMS, contacts, camera, location, sensors, bluetooth, wifi, files, notifications, accessibility). It introduces a `CapabilityRegistry` for runtime provider discovery, a `PermissionManager` mapping capabilities to numeric tiers 0-4, and an append-only `DeviceActionAudit`. This is interfaces-only scaffolding at 492 LOC — no platform implementations ship here. The package structure, tsconfig, and barrel exports match the conventions set by `@quant/identity-permissions`.

**Watch for:** PermissionManager.request() auto-grants every prompt-state request without delegation to platform APIs (confirmed) — this means the skeleton can never be safely used as-is in a non-test context without replacing that logic. The audit log diverges from the established `AuditTrail` in `@quant/agent-runtime` by skipping zod schema validation on `log()` (confirmed). `@capacitor/core` is declared as a dependency but never imported anywhere in the source (confirmed).

**Verdict**: APPROVED

## High-level view

The capability type system uses a union type enumerating 11 device domains, a generic `CapabilityProvider<T>` interface for the lifecycle contract (isAvailable/initialize/dispose), and per-domain interfaces extending it. This will compose with the existing `AgentActionTier` enum when platform implementations arrive.

The permission model maps each capability to a tier integer (0-4) and maintains per-capability permission state. The auto-grant behavior in `request()` is the most significant design gap: it hardcodes the happy path, making the stub usable for tests but potentially dangerous if imported as a real permission gate before platform-specific request logic is wired in.

The audit trail follows the append-only pattern from `@quant/agent-runtime`'s `AuditTrail` but omits the zod parse step on ingest, relying on TypeScript's structural typing at compile time instead. Runtime callers constructing entries from untrusted input will get no validation.

`CAPABILITY_TIER_MAP` uses raw numbers rather than referencing the `AgentActionTier` enum. The values align today (sensors=1, contacts=2, phone=3, accessibility=4), but there's no type-level coupling ensuring they stay in sync once `@quant/agent-runtime` is wired as a workspace dependency.

## PermissionManager auto-grant semantics

`PermissionManager.request()` checks current state and, if it's `'prompt'`, immediately writes `'granted'` and returns. There's no hook for platform-level permission dialogs, no delegate interface, no way to deny from within the flow. The test confirms this:

```typescript
it('request grants permission from prompt state', async () => {
  const result = await pm.request('location');
  expect(result).toBe('granted');
});
```

The method signature returns a Promise (suggesting it anticipates an async platform call), while the body is fully synchronous. A future implementor might forget to replace this with real logic. Adding a `// STUB: replace with platform delegate` comment or accepting an optional `RequestDelegate` in the constructor would make the intent explicit.

Additionally, the guard only checks for `'granted'` and `'denied'` before auto-granting:

```typescript
if (current === 'granted' || current === 'denied') return current;
this.states.set(capability, 'granted');
```

This means `'unavailable'` state also falls through to auto-grant, which is semantically wrong — an unavailable capability shouldn't become granted via request.

## Audit log validation gap

The existing `AuditTrail` in `@quant/agent-runtime` runs `AuditEntrySchema.parse(entry)` on every `log()` call, catching malformed entries at ingest time. `DeviceActionAudit` skips this — `log()` just pushes the entry onto the array. Since zod is already a dependency of this package, adding a matching schema would be trivial and would keep the audit pattern consistent across packages.

## Tier mapping drift risk

`CAPABILITY_TIER_MAP` uses raw numbers (0-4) rather than referencing the `AgentActionTier` enum from `@quant/agent-runtime`. The numeric values happen to align (sensors=1 maps to Tier1_DraftOnly, contacts=2 to Tier2_LowRisk, phone=3 to Tier3_HighRisk, accessibility=4 to Tier4_Admin), but there's no type-level or import-level coupling ensuring they stay in sync. This is acceptable for a standalone skeleton that avoids workspace cross-deps, but the mapping should reference the enum once `@quant/agent-runtime` is wired as a dependency.

## Unused @capacitor/core

The package.json declares `@capacitor/core: ^6.0.0` but no source file imports from it. This adds install weight without utility. If it's here as a forward declaration for platform implementations that will live in this package, a comment would clarify intent and prevent a future contributor from removing it during cleanup.
