# QuantCalendar: standalone calendar app with AI scheduling (v2)

QuantCalendar adds a calendar backend to the monorepo with event CRUD, hand-rolled RRULE recurrence, multi-user availability, Calendly-style booking links, and six AI features. The five blocking issues from v1 have been resolved: attendee status updates now require authorization, booking confirmation checks for conflicts before creating, focus-blocks uses the AI response to select gaps, exception tests exercise the actual EXDATE code path, and the COUNT bug is fixed. Two new concerns emerged during this pass related to query semantics and an unauthenticated AI endpoint.

**Watch for:** The overlap-detection query in `confirmBooking` (and all availability queries) only finds events fully contained within the time window, missing partially-overlapping events — the 409 check can be bypassed by existing events that straddle slot boundaries (confirmed). The `POST /detect-cancellation` endpoint performs AI inference without any authentication check (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The booking confirmation overlap check added in v2 fixes uses a Prisma query with `startTime: { gte: slotStart }, endTime: { lte: slotEnd }`. This finds events whose start and end both fall within the slot, but misses events that start before the slot and end within it, or start within the slot and end after it. The JavaScript-level overlap check (`eStart < endTime && eEnd > startTime`) is correct, but operates only on the pre-filtered result set — it can't catch what the query didn't return. The same pattern affects `getAvailableSlots`, `getAvailability`, `addBufferTime`, and `reserveFocusBlocks`.

The `POST /ai/detect-cancellation` route is the only non-GET endpoint that doesn't extract or validate `userId` from the request. It calls `ai.infer` with a hardcoded `userId: 'system'`, meaning any unauthenticated caller can trigger AI inference and consume compute budget.

<summary>Issues (2)</summary>

1. **Overlap query misses straddling events** — `confirmBooking` queries `startTime >= slotStart AND endTime <= slotEnd`, which only returns events fully inside the slot. Events starting before or ending after the slot won't be found, so the overlap check passes even when a conflict exists. Fix: query for `startTime < slotEnd AND endTime > slotStart` (standard interval overlap). The same query pattern in `getAvailableSlots`, `getAvailability`, `addBufferTime`, and `reserveFocusBlocks` has the same gap.
2. **Unauthenticated AI inference endpoint** — `POST /detect-cancellation` doesn't check `request.auth.userId`. Any caller can invoke AI inference at the server's cost. Add the same auth extraction pattern used by the other AI endpoints.

## Overlap query finds only fully-contained events

The `confirmBooking` availability check queries Prisma like this:

```typescript
const existingEvents = await this.prisma.event.findMany({
  where: {
    userId: link.userId,
    startTime: { gte: startTime },
    endTime: { lte: endTime },
  },
});
```

For a 30-minute slot from 10:00 to 10:30, this query returns events where `event.startTime >= 10:00 AND event.endTime <= 10:30`. An existing event from 9:45 to 10:15 has `startTime < 10:00`, so it fails the `gte` condition and is excluded from results. The subsequent JavaScript overlap check (`eStart < endTime && eEnd > startTime`) correctly identifies overlaps, but only within the result set the query returned — which already excluded the conflict.

The standard interval-overlap predicate for the database query should be `startTime < slotEnd AND endTime > slotStart`. This catches any event that intersects the slot regardless of which boundary it crosses.

The same query pattern appears in:

- `BookingLinkService.getAvailableSlots` — events straddling the working-hours boundary won't appear as busy
- `AvailabilityService.getAvailability` — same gap for multi-user availability computation
- `AIBufferService.addBufferTime` — events spanning midnight or starting before query range are missed
- `AIFocusBlocksService.reserveFocusBlocks` — early-morning events extending into the 9-5 window aren't detected

In practice the booking scenario is the most exploitable: a user with a 9:30-10:30 meeting would show the 10:00 slot as conflicting in `getAvailableSlots` (because the 9:30 start < dayStart so it's excluded), but then `confirmBooking` would also fail to detect it — resulting in a double-booking that `getAvailableSlots` would have caught if the query were correct.

## Unauthenticated detect-cancellation endpoint

Every other mutating AI endpoint in `routes/ai.ts` follows this pattern:

```typescript
const userId = (request as unknown as { auth: { userId: string } }).auth?.userId;
if (!userId) {
  throw createAppError('Authentication required', 401, 'UNAUTHORIZED');
}
```

The `POST /detect-cancellation` handler skips this entirely:

```typescript
fastify.post('/detect-cancellation', async (request, reply) => {
  const parseResult = detectCancellationSchema.safeParse(request.body);
  if (!parseResult.success) {
    throw parseResult.error;
  }

  const ai = (fastify as unknown as { ai: unknown }).ai;
  const service = new AICancelDetectorService(ai as never);
  const result = await service.detectCancellation(parseResult.data.messageText);
  return reply.send({ success: true, data: result });
});
```

This endpoint accepts arbitrary text and calls `ai.infer()`. Without authentication, it's an open proxy to the AI backend. Even if this is intentionally unauthenticated for integration use cases, it should at minimum be rate-limited separately or require an API key.
