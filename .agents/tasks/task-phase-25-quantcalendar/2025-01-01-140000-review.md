# QuantCalendar: standalone calendar app with AI scheduling

QuantCalendar adds a new workspace package (`apps/quantcalendar/`) implementing a calendar backend with CRUD events, RRULE-based recurrence expansion, multi-user availability intersection, Calendly-style booking links, and six AI-powered features (smart scheduling, buffer insertion, focus blocks, cancellation detection, rescheduling, weekly digest). The implementation follows the established monorepo pattern: Fastify routes with zod validation, service classes accepting injected Prisma/AI dependencies, and vitest-based unit tests using mocks. The ecosystem type registry is extended with `quantcalendar`.

**Watch for:** The `updateAttendeeStatus` endpoint lacks authorization — any caller can update any attendee's status on any event (confirmed). Booking link `confirmBooking` creates events without verifying slot availability, enabling double-booking (confirmed). The AI focus-blocks service discards the AI response entirely and always fills all eligible gaps (confirmed). The recurring event exception test doesn't exercise the exception code path it claims to test (confirmed).

## High-level view

The event CRUD layer stores attendees and reminders as JSON strings in a single `event` table. Authorization is per-userId on the event record, but the attendee-status update path skips this gate entirely — it doesn't verify the caller is the attendee being updated or the event owner, creating an open-access gap.

Recurrence expansion is a hand-rolled RRULE engine covering daily/weekly/monthly/yearly frequencies with BYDAY, BYMONTH, COUNT, and UNTIL support. Exceptions are not persisted or serializable — they exist only as an in-memory property on the parsed rule, and the test for exceptions never actually invokes the code path.

The six AI services follow a consistent pattern: gather context from Prisma, construct a prompt, call `ai.infer()`, parse JSON from the response, and fall back gracefully on parse failure. The focus-blocks service is the exception — it calls the AI but ignores the response, always creating blocks in every eligible gap.

Booking links provide a public-facing scheduling API. Slot generation checks owner events for conflicts, but the booking confirmation step skips this check, so concurrent bookings for the same slot will both succeed.

The route layer instantiates services per-request from `fastify.prisma` and `fastify.ai` accessed via unsafe casts. All 44 tests target service-layer logic with no route-level coverage.

## Attendee status update authorization gap

The `PATCH /:id/attendees/:attendeeId/status` route in `events.ts` is the only event-mutating endpoint that doesn't extract `userId` from the auth context:

```typescript
// routes/events.ts - PATCH handler
fastify.patch<{ Params: { id: string; attendeeId: string } }>(
  '/:id/attendees/:attendeeId/status',
  async (request, reply) => {
    const parseResult = updateAttendeeStatusSchema.safeParse(request.body);
    // ... no userId extraction, no auth check ...
    const event = await service.updateAttendeeStatus(
      request.params.id,
      request.params.attendeeId,
      parseResult.data.status,
    );
```

At the service layer, `updateAttendeeStatus` also skips the `getEvent(id, userId)` authorization gate that all other mutations use. It finds the event by ID alone and updates any attendee's status. Any caller who can reach this endpoint can change any attendee's RSVP on any event.

## Booking confirmBooking race condition

`BookingLinkService.confirmBooking` fetches the link metadata, computes the end time, and creates the event — but never checks whether the requested slot is still free. Contrast with `getAvailableSlots`, which queries existing events and marks overlapping slots as unavailable. Two requests arriving simultaneously for the same slot will both succeed, creating duplicate events.

The fix: call the same overlap-detection logic inside `confirmBooking` before creating the event, or use a database-level constraint.

## Focus-blocks AI is decorative

```typescript
// ai-focus-blocks.service.ts
try {
  await this.ai.infer({ ... });
} catch {
  // Continue with all eligible gaps even if AI fails
}

// Then unconditionally iterates over ALL eligible gaps:
for (const gap of eligibleGaps) {
  const created = await this.prisma.event.create({ ... });
}
```

The AI call's response is never assigned or used. Whether it succeeds or throws, the behavior is identical: every gap meeting the minimum duration gets a focus block.

## Recurring exception test gap

The exceptions test in `recurring.service.test.ts` acknowledges its own limitation:

```typescript
// Since serializeRRule doesn't serialize exceptions, let's test via parseRRule + expandRecurrence
const occurrencesBase = service.expandRecurrence(event, startRange, endRange);
expect(occurrencesBase).toHaveLength(5); // without exceptions in rrule string
```

The test verifies that 5 occurrences are generated _without_ exceptions, then sets `parsedRule.exceptions` on a detached object but never re-expands. The actual exception-skipping logic in `expandRecurrence` (which checks `exceptionSet.has(dateKey)`) is never executed in any test.

Additionally, `serializeRRule` doesn't handle exceptions — there's no way to persist exception dates through the RRULE string format. If they're meant to be stored separately (as EXDATE records), there's no persistence path for them.

## Prompt injection in AI services

Both `AICancelDetectorService` and `AIRescheduleService` embed user-controlled text directly into prompts:

```typescript
// ai-cancel-detector.service.ts
prompt: `Analyze the following message...
"${messageText}"
...`;
```

```typescript
// ai-reschedule.service.ts
prompt: `Parse this scheduling instruction:
"${instruction}"
...`;
```

A user who submits text containing prompt override instructions can manipulate the AI response. Currently the routes just return results to the caller (impact limited to self-deception), but if automated actions are wired up later (e.g., auto-cancelling events based on `suggestedAction`), this becomes a functional bypass.

## Exceptions consume RRULE COUNT

In `expandRecurrence`, when an exception date is encountered within the range:

```typescript
if (!exceptionSet.has(dateKey)) {
  if (this.matchesRule(current, rule)) {
    occurrences.push({ ... });
    count++;
  }
} else {
  count++;  // exception still eats a count slot
}
```

RFC 5545 specifies that COUNT is the number of _generated_ instances. An exception should not decrement the remaining count. With this implementation, a rule like `FREQ=DAILY;COUNT=5` with one exception will produce 4 visible occurrences instead of the expected 5.

## No route-level test coverage

All 44 tests operate at the service layer with fully mocked Prisma clients. Specific gaps:

- Zod validation rejection (malformed bodies) is not tested
- The `request.auth.userId` extraction pattern (via unsafe cast) is not tested
- Error response format (the shape returned when services throw `createAppError`) is not tested
- The booking routes' public-facing endpoints (no auth required for GET /links/:slug) are not tested for their auth-free behavior
